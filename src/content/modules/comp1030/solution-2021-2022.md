## Question 1: Networks (25 marks)

### (a) CSMA/CA Protocol and Why It Is Needed for Wireless LAN [3 marks]

**CSMA/CA (Carrier Sense Multiple Access with Collision Avoidance) Operation:**

1. **Carrier Sensing:** A station wishing to transmit first listens to the medium. If the medium is busy, it defers.
2. **Inter-Frame Spacing (IFS):** When the medium becomes idle, the station waits for a mandatory IFS period (e.g., DIFS for data frames).
3. **Random Backoff:** After the IFS, the station selects a random backoff interval (countdown timer). The timer decrements only when the medium is idle. If the medium becomes busy during countdown, the timer freezes and resumes when idle again.
4. **Transmission:** When the timer reaches zero, the station transmits.
5. **RTS/CTS (optional):** The sender may first send a short Request-to-Send (RTS) frame. If the receiver replies with Clear-to-Send (CTS), the sender proceeds. All other stations hearing RTS or CTS defer for the indicated duration (via NAV timer).
6. **ACK:** The receiver sends an acknowledgment. If no ACK is received, the sender assumes the frame was lost and retransmits with a longer backoff.

**Why CSMA/CA is needed for wireless LANs (not CSMA/CD):**

- **Collision detection is impractical in wireless:** A station cannot transmit and listen at the same time on the same radio frequency — the transmitted signal overwhelms the received signal (the transmitted power is millions of times stronger than any received signal).
- **Hidden terminal problem:** Station A and Station C may both be in range of Station B but not of each other. A cannot detect C's transmission, so CSMA/CD would fail. CSMA/CA with RTS/CTS solves this because B's CTS is heard by both A and C.
- **Signal fading and attenuation:** Wireless signals weaken over distance and are affected by obstacles, making reliable detection of collisions by sensing impossible.

### (b) Overhead Factors in Ethernet and Token Ring [4 marks]

**Ethernet Overhead Factors:**

| Factor | Description | Effect on Performance |
|--------|-------------|----------------------|
| **Collisions** | When two stations transmit simultaneously, frames collide and must be retransmitted | Wasted bandwidth and increased delay; gets worse as load increases |
| **Backoff time** | After a collision, stations must wait a random exponential backoff period before retrying | Adds unpredictable delay; repeated collisions cause exponentially longer waits |
| **Inter-frame gap** | Mandatory 96-bit time gap between frames for signal recovery | Reduces maximum usable throughput (no data sent during gap) |
| **Minimum frame size** | Frames must be at least 64 bytes to ensure collision detection works | Small payloads waste bandwidth due to padding |
| **Propagation delay** | Time for signal to travel the full network length | Longer networks have larger collision windows, reducing efficiency |

**Token Ring Overhead Factors:**

| Factor | Description | Effect on Performance |
|--------|-------------|----------------------|
| **Token rotation time** | A station must wait for the token to arrive before transmitting | Adds latency; under light loads, stations wait even when the ring is otherwise idle |
| **Token maintenance** | Requires monitor station to detect lost/duplicate tokens and regenerate them | Uses bandwidth for monitoring; if monitor fails, ring is disrupted until new monitor elected |
| **Latency per station** | Each station adds a one-bit delay as the signal passes through its interface | More stations increase total ring latency and token rotation time |
| **Token holding time** | Maximum time a station may hold the token limits how much data it can send per turn | Under heavy load with many stations, each gets limited bandwidth per rotation |
| **Frame removal** | The sending station must remove its frame from the ring after it circulates | The frame occupies the ring for a full rotation even after being read by the recipient |

**Performance comparison:**
- Ethernet performs well under light loads (no waiting for token) but degrades under heavy loads (more collisions).
- Token Ring provides guaranteed access and deterministic delays, performing better under heavy loads, but has higher overhead under light loads (waiting for token when ring is mostly idle).

### (d) Connecting Two Buildings Across a Public Road [2 marks]

**(i) When to use a Leased Line Link:**
- When the buildings are relatively close (within a few kilometers) and the public road prevents laying private cable.
- A leased line from a telecommunications provider provides a dedicated, permanent, private connection.
- Appropriate when: reliable, high-bandwidth, low-latency communication is needed; the distance is moderate; a continuous guaranteed connection is required; and the organisation can justify the recurring monthly cost.
- Best for: organisations needing constant connectivity with guaranteed bandwidth and SLA (Service Level Agreement).

**(ii) When to use a Satellite Link:**
- When the buildings are in a very remote location where no terrestrial infrastructure (leased lines, fiber) is available.
- When the distance is extremely large or terrain makes wired connections impractical.
- Appropriate when: no ground-based alternatives exist; mobility is needed; or temporary connectivity is required.
- Drawback: high latency (especially geostationary satellites with ~500ms round-trip), weather sensitivity, and high cost.
- For two buildings across a road in an urban area, a satellite link would typically NOT be the preferred choice due to cost and latency. It would only be used if no terrestrial alternative is available.

### (f) Two Criteria for "Best" Path in Routing [2 marks]

The two primary criteria used to determine the "best" path in routing are:

1. **Minimum hop count (fewest intermediate nodes):** The path that passes through the fewest number of routers/switches is preferred. This minimises processing delay and reduces the chance of failure along the path. Used by protocols like RIP.

2. **Minimum cost/distance (link metrics):** The path with the lowest total cost is preferred, where cost can represent bandwidth (higher bandwidth = lower cost), delay (latency), monetary cost, or a combination. Used by protocols like OSPF and Dijkstra's algorithm.

Other possible criteria include: reliability (fewest link failures), current load/congestion, and maximum throughput.

### (g) Supporting Multiple Network Protocols via Ethernet Frame [2 marks]

The Ethernet frame contains a **Type field** (in Ethernet II / DIX format) or **LLC/SNAP header** (in IEEE 802.3 format) that identifies which upper-layer network protocol the frame's payload belongs to.

- The **EtherType** field (2 bytes) specifies the protocol of the encapsulated payload. For example:
  - 0x0800 = IPv4
  - 0x86DD = IPv6
  - 0x0806 = ARP

- When a host receives a frame, it examines the Type field and passes the payload to the appropriate network-layer protocol handler.

- This allows a single physical network interface card (NIC) and a single Ethernet attachment to simultaneously support multiple network protocols (e.g., IPv4, IPv6, ARP, IPX) by demultiplexing incoming frames based on the Type field.

### (h) Two Ways to Increase LAN Capacity [2 marks]

1. **Segmentation using bridges or switches:** Dividing the LAN into multiple smaller collision domains using bridges or layer-2 switches. Each segment has its own dedicated bandwidth, so fewer stations compete for the medium on each segment. A switch provides a dedicated collision-free path between communicating stations (microsegmentation), effectively giving each port full bandwidth.

2. **Upgrading to higher-speed technology:** Replacing the existing network infrastructure with faster hardware — for example, upgrading from 100 Mbps Fast Ethernet to 1 Gbps Gigabit Ethernet or 10 Gbps. This directly increases the raw bandwidth available for all users.

### (b) Variable-Length Instructions (CISC) [4 marks]

#### (i) Advantages of Variable-Length Instructions:

1. **Efficient use of memory:** Simple, common instructions can use shorter formats (fewer bytes), while complex instructions use longer formats. This reduces overall program size compared to fixed-length instructions.

2. **Rich instruction set:** Variable-length allows a wide variety of addressing modes and operand combinations. Complex operations can be encoded in a single instruction, reducing instruction count.

3. **Powerful addressing modes:** Longer instructions can include large immediate values, absolute addresses, or complex operand specifiers that would be impossible with a fixed short format.

4. **Backward compatibility:** New, longer instructions can be added without changing the encoding of existing shorter instructions, making it easier to extend the architecture.

#### (ii) Disadvantages of Variable-Length Instructions:

1. **Complex decoding:** The processor cannot determine instruction boundaries without decoding sequentially. The instruction decoder needs more complex hardware to handle different lengths, increasing chip complexity and potentially slowing clock speed.

2. **Difficult pipelining:** Since instruction length is not known until decoding begins, fetching and aligning the next instruction is more complex. This complicates pipeline design and can cause pipeline stalls.

3. **Slower instruction fetch:** The processor may need multiple memory accesses to fetch a single long instruction, or must use complex alignment logic for instruction buffers.

4. **Difficult parallel execution:** Superscalar designs (issuing multiple instructions per cycle) are much harder because instruction boundaries are not known in advance without sequential decoding.

## Question 3: Interrupts, I/O and Instruction Pipelining (25 marks)

### (a) Two Drawbacks of Interrupt-Driven I/O [4 marks]

Although interrupt-driven I/O is more efficient than programmed I/O (because the processor can do other work while waiting), it still has two significant drawbacks:

**Drawback 1: Processor is involved in every data transfer**
- Every word/byte of data transferred between the I/O device and memory must pass through the processor. The processor must execute instructions to move data from the I/O module's data register to memory (or vice versa).
- This consumes processor cycles for every unit of data transferred. For high-speed devices or bulk transfers (e.g., disk I/O), this becomes a significant overhead, slowing down both the I/O transfer and the processor's ability to execute other programs.
- The data transfer rate is limited by the speed at which the processor can service interrupts and execute transfer instructions.

**Drawback 2: Context switching overhead**
- Every interrupt requires the processor to save its current state (registers, program counter, status flags), execute the interrupt service routine (transfer data), and then restore the previous state to resume the interrupted program.
- This context switching consumes time and processor resources. If interrupts are frequent (e.g., one interrupt per byte transferred on a fast device), the overhead of repeated context switches can degrade system performance significantly — potentially making it worse than programmed I/O.
- The processor spends a disproportionate amount of time on overhead (saving/restoring state) rather than useful computation.

*Note: DMA (Direct Memory Access) solves both drawbacks by allowing the I/O module to transfer data directly to/from memory without processor intervention for each word.*

---

### (b) Data Hazards [8 marks]

#### (i) What is a data hazard? Hardware technique to overcome it [4 marks]

**Data Hazard:**
A data hazard occurs in a pipelined processor when an instruction depends on the result of a previous instruction that has not yet completed writing its result. Because pipeline stages overlap, a later instruction may try to read an operand before the earlier instruction has produced it. This forces the pipeline to stall (insert bubbles), reducing the throughput benefit of pipelining.

The three types of data hazards:
- **RAW (Read After Write):** An instruction tries to read a register before a preceding instruction has written its result to that register. Most common and problematic.
- **WAR (Write After Read):** An instruction writes to a register before a preceding instruction has read from it (possible in out-of-order execution).
- **WAW (Write After Write):** Two instructions write to the same register; the writes must occur in program order (possible in out-of-order execution).

**Hardware Technique: Data Forwarding (also called Bypassing or Short-circuiting)**

Data forwarding overcomes most RAW data hazards by:
- Adding dedicated hardware paths (forwarding/bypass paths) that route a computed result directly from the output of one pipeline stage (typically the ALU output at the end of the Execute stage) to the input of the same or earlier stage where it is needed.
- A **forwarding unit** (hardware comparator) continuously monitors the pipeline and detects when a source register of the instruction currently in the decode/execute stage matches the destination register of an instruction in a later pipeline stage (EX/MEM or MEM/WB).
- When a match is detected, the forwarding unit selects the forwarded value via a multiplexer instead of the stale value from the register file.
- This eliminates most stall cycles because the data is available one or more cycles earlier than it would be if the pipeline waited for the write-back stage.
- Limitation: forwarding cannot eliminate all stalls (e.g., a load-use hazard where data is loaded from memory still requires one stall cycle because the data is not available until after the Memory stage).

### (c) Multiple Device Interrupts: Priority Handling [8 marks]

When multiple device interrupts have occurred simultaneously or while another is being serviced, the processor must determine which one to process. Several mechanisms exist:

**1. Software Polling:**
- When an interrupt signal is received, the processor branches to a general interrupt handler.
- This handler sequentially tests (polls) the status register of each I/O device in a predetermined order.
- The first device found with a pending interrupt request is serviced.
- Priority is implicitly determined by the polling order — devices tested first have higher priority.
- **Advantage:** Simple, no special hardware required.
- **Disadvantage:** Slow, especially with many devices; wastes time checking devices that did not interrupt.

**2. Daisy-Chain (Hardware Priority):**

```
        INT                    INTA
CPU ◄─────────── All devices assert INT line (wired-OR)
CPU ──INTA──▶ Dev1 ──INTA──▶ Dev2 ──INTA──▶ Dev3 ──INTA──▶ Dev4
             (highest)                              (lowest)
```

- All devices share a common interrupt request line (wired-OR).
- The interrupt acknowledge (INTA) signal is passed serially through devices in a daisy-chain.
- The device closest to the processor receives INTA first. If it has a pending interrupt, it captures the acknowledge, places its vector on the data bus, and blocks INTA from reaching subsequent devices.
- If it has no pending interrupt, it passes INTA to the next device.
- Priority is determined by physical position on the chain.
- **Advantage:** Simple hardware, easily expandable.
- **Disadvantage:** Fixed priorities; far devices have slow response.

**3. Parallel Priority Interrupt Controller (PIC):**

```
Device 0 (IRQ0) ──┐
Device 1 (IRQ1) ──┤
Device 2 (IRQ2) ──┼──▶ [Priority Interrupt Controller] ──INT──▶ CPU
Device 3 (IRQ3) ──┤       • Priority Resolver
    ...           ┤       • Mask Register
Device n (IRQn) ──┘       • In-Service Register
```

- A dedicated hardware controller (e.g., Intel 8259 PIC) accepts interrupt requests from multiple devices on separate input lines.
- The **priority resolver** compares all pending requests and selects the highest-priority one.
- The **mask register** allows software to enable/disable individual interrupt lines.
- The **in-service register** tracks which interrupts are currently being serviced.
- The controller signals the CPU with the vector of the highest-priority pending interrupt.
- Supports programmable priority schemes: fixed priority, rotating priority, or specific priority.

**4. Interrupt Priority Levels and Nesting:**
- The processor maintains a current priority level in its status register.
- Only interrupts with priority higher than the current level can interrupt the processor.
- When servicing an interrupt at priority level N, only interrupts at level N+1 or higher can pre-empt it (nested interrupts).
- Lower-priority interrupts remain pending and are serviced when the processor returns to a lower priority level.

**5. Priority decision process:**
1. Multiple devices assert their interrupt request lines.
2. The interrupt controller's priority resolver determines which has the highest priority.
3. The controller checks if this priority exceeds the processor's current interrupt level (mask check).
4. If yes, the controller asserts INT to the CPU and provides the interrupt vector when acknowledged.
5. The CPU saves its state and jumps to the corresponding interrupt service routine.
6. Other pending interrupts are serviced in priority order as each ISR completes (or via nesting if higher priority arrives).

### (d) What does the processor do when DMA controls the bus? [3 marks]

When the DMA (Direct Memory Access) module takes control of the system bus, the following occurs:

**1. Cycle Stealing:** The DMA module "steals" bus cycles from the processor. During these cycles, the processor cannot access the system bus (cannot fetch from memory or write to memory via the bus).

**2. Processor continues internal operations:** The processor does not halt completely. It can continue performing operations that do not require the system bus:
- Executing instructions already fetched into the pipeline or instruction cache.
- Performing ALU operations using data already in registers.
- Accessing data in L1/L2 cache (if the cache system uses a separate internal bus).

**3. Bus request/grant protocol:**
- The DMA module asserts a Bus Request (BR) signal to the CPU.
- The CPU finishes its current bus cycle, then asserts Bus Grant (BG) and releases the bus.
- The DMA module performs one or more data transfers directly between the I/O device and memory.
- After the transfer (one word or a burst), the DMA releases the bus and de-asserts BR.
- The CPU regains bus access and continues normally.

**4. No interrupt or context switch:** Unlike interrupt-driven I/O, the processor does NOT save state or branch to a handler. The processor simply pauses bus activity for a brief period. This makes DMA much more efficient — only when the entire block transfer is complete does the DMA generate an interrupt to notify the processor.

**5. Slight performance reduction:** The processor runs slightly slower because it occasionally has to wait for bus access, but this is far less overhead than having the processor handle every individual data transfer.

### (e) Multicore Processor vs Coprocessor [2 marks]

**Multicore Processor:**
- A single integrated circuit (chip) containing two or more independent, general-purpose processing cores.
- Each core has its own execution unit, registers, and often its own L1 cache.
- Each core can independently fetch, decode, and execute its own instruction stream.
- Cores typically share L2/L3 cache and main memory.
- All cores are equal in capability — any core can run any program or thread.
- Used for true parallel processing of multiple threads or processes simultaneously.
- Example: A quad-core CPU where each core runs a different application thread.

**Coprocessor:**
- A separate, specialised processing unit designed to perform a specific class of operations more efficiently than the main CPU.
- It operates under the control and direction of the main CPU — it cannot run programs independently.
- It has hardware optimised for specific computations (e.g., floating-point arithmetic, graphics rendering, signal processing, encryption).
- When the main CPU encounters an instruction meant for the coprocessor, it delegates execution to the coprocessor and may continue other work or wait for the result.
- Example: Floating-point unit (FPU), GPU used as compute coprocessor, Intel 8087.

**Key Differences:**

| Feature | Multicore | Coprocessor |
|---------|-----------|-------------|
| Independence | Each core operates independently | Depends on main CPU for control |
| Purpose | General-purpose computing | Specialised tasks only |
| Instruction set | Full general-purpose ISA | Limited, specialised instruction set |
| Can run OS/programs | Yes | No (works under CPU direction) |
| Number on chip | Multiple homogeneous cores | Usually one per function type |
