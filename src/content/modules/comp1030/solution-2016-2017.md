## Question 1: Networks (25 marks)

### (a) Advantages and Disadvantages of Replacing Coaxial with Optical Fibre [4 marks]

**Advantages:**

1. **Much higher bandwidth:** Optical fibre supports significantly greater data rates (up to terabits per second) compared to coaxial cable. This allows the cable TV company to deliver more channels, higher resolution content (4K/8K), and faster internet simultaneously.

2. **Immunity to electromagnetic interference (EMI):** Fibre optic cables transmit light rather than electrical signals, making them immune to interference from power lines, radio frequencies, and other sources of electromagnetic noise. This results in a cleaner, more reliable signal.

**Disadvantages:**

1. **High installation cost:** Replacing an entire existing coaxial infrastructure with fibre optics requires enormous capital expenditure. Fibre cables, connectors, splicing equipment, and trained installers are all more expensive than their coaxial equivalents.

2. **Fragility and difficulty of repair:** Optical fibre is more fragile than coaxial cable. It is made of glass or plastic and can be damaged by sharp bends or physical stress. Splicing and repairing broken fibre requires specialised equipment and expertise, leading to longer repair times and higher maintenance costs.

### (b) Tokens and Network Access Control [3 marks]

A **token** is a special control frame (a short, predefined bit pattern) that circulates continuously around a network, typically a Token Ring or Token Bus topology.

**How tokens control network access:**

- The token circulates around the ring from station to station.
- A station may only transmit data when it possesses (captures) a free token.
- When a station wishes to send data, it waits until it receives the free token, removes it from the ring, and then transmits its data frame.
- Once the station finishes transmission (or a timer expires), it releases a new free token back onto the ring.
- This mechanism prevents collisions because only one station can hold the token (and thus transmit) at any given time, providing deterministic access to the medium.

### (d) Switch Built with Repeaters Instead of Bridges [3 marks]

**No, I would not buy one.** Here is why:

- A **bridge** operates at the Data Link layer (Layer 2). It reads MAC addresses, filters traffic, and creates separate collision domains for each port. This is what makes a switch function correctly -- each port is its own collision domain and frames are forwarded intelligently based on destination addresses.

- A **repeater** operates at the Physical layer (Layer 1). It simply regenerates and retransmits electrical signals. It does not examine frame addresses and does not filter traffic. All ports connected via repeaters remain in the same collision domain.

- A "switch" built from repeaters would not segment collision domains at all. It would behave like a hub, flooding all traffic to all ports, defeating the entire purpose of a switch. Performance would degrade significantly under load due to collisions, and there would be no intelligent forwarding. The product is functionally a hub marketed as a switch.

### (e) Circuit Switching Network [5 marks]

**Operation:**

Circuit switching establishes a dedicated communication path (circuit) between two endpoints before any data transfer occurs. The circuit remains reserved for the entire duration of the communication session.

**Three Phases:**

```
Phase 1: Circuit Establishment
[Sender] --request--> [Switch A] --request--> [Switch B] --request--> [Receiver]
                                                                       |
[Sender] <--acknowledge-- [Switch A] <--acknowledge-- [Switch B] <--ACK--+

Phase 2: Data Transfer
[Sender] ====dedicated path====> [Switch A] ====> [Switch B] ====> [Receiver]
         (data flows continuously on the reserved circuit)

Phase 3: Circuit Teardown
[Sender] --disconnect--> [Switch A] --disconnect--> [Switch B] --disconnect--> [Receiver]
         (resources released at each switch)
```

1. **Circuit Establishment:** The sender sends a connection request through intermediate switches to the receiver. Each switch allocates resources (bandwidth, time slot, or frequency) along the path. The receiver acknowledges and the circuit is set up end-to-end.

2. **Data Transfer:** Data flows along the dedicated path. The full bandwidth of the circuit is available exclusively to this connection.

3. **Circuit Teardown:** Once communication is complete, a disconnect signal releases all resources at each switch.

**Advantage:**

- Guaranteed constant bandwidth and low, predictable latency once the circuit is established. Ideal for real-time communication (e.g., voice calls).

**Disadvantage:**

- Inefficient use of resources. The circuit is reserved even during silence or idle periods. If no data is being sent, the allocated bandwidth is wasted and cannot be used by other connections.

### (f) Adaptive vs Non-Adaptive Routing Algorithms [3 marks]

**Non-adaptive routing (static routing):**
Routing decisions are computed in advance and loaded into routers before the network begins operation. Routes do not change in response to current network conditions (traffic, congestion, link failures). Examples include fixed shortest-path routing and flooding.

**Adaptive routing (dynamic routing):**
Routing decisions change dynamically based on current network conditions. Routers continuously monitor traffic, topology changes, and link states, and update their routing tables accordingly. Examples include distance vector (RIP) and link-state (OSPF) algorithms.

**Why adaptive routing is superior:**

Adaptive routing is superior because it can respond to network changes such as link failures, congestion, and varying traffic loads. If a link goes down, adaptive routing reroutes traffic around the failure. If a path becomes congested, traffic can be redistributed to less loaded paths. Non-adaptive routing cannot respond to such changes, leading to potential data loss, increased latency, or complete communication failure when the network state differs from what was assumed at configuration time.

### (g) Hidden and Exposed Station Problems [4 marks]

**Hidden Station Problem:**

```
        Range of A          Range of C
    |<------------>|    |<------------>|
    A       B              B       C
    |_______|______________|_______|
            ^
         A and C cannot hear each other
         but both can reach B
```

- Station A is transmitting to station B.
- Station C cannot hear A's transmission (C is outside A's range).
- C senses the medium as idle and begins transmitting to B.
- A collision occurs at B because it receives signals from both A and C simultaneously.
- The "hidden" station (C) is hidden from A, causing undetectable collisions at the receiver.

**Exposed Station Problem:**

```
        Range of B          Range of C
    |<------------>|    |<------------>|
    A       B       C       D
    |_______|_______|_______|
```

- Station B is transmitting to station A.
- Station C wants to transmit to station D.
- C senses B's transmission and (incorrectly) defers, believing its own transmission would cause a collision.
- However, C's transmission to D would NOT interfere with B's transmission to A, since A is outside C's range and D is outside B's range.
- The "exposed" station (C) unnecessarily refrains from transmitting, reducing network throughput.

**Solution:** The MACA/MACAW protocol (or RTS/CTS mechanism used in IEEE 802.11) addresses both problems by using short Request-to-Send and Clear-to-Send control frames to reserve the channel.

### (e) Cache Coherency [4 marks]

**Cache coherency** refers to the consistency of data stored in local caches of a shared resource (typically main memory) in a multiprocessor or multicore system.

In a system with multiple processors, each processor typically has its own local cache. When multiple caches hold copies of the same memory location, a **coherency problem** arises if one processor modifies its cached copy -- the other caches now hold stale (outdated) data that is inconsistent with the updated value.

**The cache coherency problem occurs when:**

1. Processor A reads a memory location X into its cache.
2. Processor B also reads the same location X into its cache.
3. Processor A modifies X in its local cache.
4. Processor B's cache still holds the old value of X -- the caches are now incoherent.

**Solutions to maintain coherency:**

- **Write-through with bus snooping:** All writes are immediately propagated to main memory, and other caches monitor (snoop) the bus to invalidate or update their copies.
- **MESI protocol (Modified, Exclusive, Shared, Invalid):** Each cache line is tagged with a state, and state transitions ensure that modifications are propagated correctly across all caches.
- **Directory-based protocols:** A central directory tracks which caches hold copies of each memory block and coordinates updates/invalidations.

Cache coherency is essential for correct program execution in multiprocessor systems, ensuring that all processors observe a consistent view of memory.

### (f) CISC vs RISC Processors [4 marks]

| Feature | CISC | RISC |
|---------|------|------|
| **Instruction set** | Large number of complex instructions, many taking multiple clock cycles | Small number of simple instructions, most executing in a single clock cycle |
| **Instruction format** | Variable-length instructions with many addressing modes | Fixed-length instructions with few, uniform formats |
| **Memory access** | Many instructions can operate directly on memory (memory-to-memory operations allowed) | Only LOAD and STORE instructions access memory; all other operations work on registers (load-store architecture) |
| **Registers** | Fewer general-purpose registers (operations use memory more) | Large number of general-purpose registers to reduce memory accesses |
| **Hardware complexity** | Complex hardware with microcode to implement instructions; harder to pipeline | Simple hardwired control unit; designed for efficient pipelining |
| **Code density** | Higher code density (fewer instructions needed for a task) | Lower code density (more instructions, but each is faster) |

## Question 3: Interrupts, I/O and Instruction Pipelining (25 marks)

### (a) Three I/O Techniques [6 marks]

**1. Programmed I/O (Polling):**

The processor is fully responsible for the I/O operation. The CPU issues an I/O command and then repeatedly checks (polls) the status of the I/O device in a busy-wait loop until the operation is complete. During this time, the processor cannot do any other useful work. The CPU directly transfers data between memory and the I/O device word by word.

- Advantage: Simple to implement.
- Disadvantage: Extremely wasteful of CPU time due to busy waiting.

**2. Interrupt-Driven I/O:**

The processor issues an I/O command to the device and then continues executing other instructions. When the I/O device is ready (data available or transfer complete), it sends an interrupt signal to the processor. The processor suspends its current task, services the interrupt by transferring data, and then resumes its original task. Data is still transferred through the CPU word by word.

- Advantage: CPU is free to do other work while waiting for I/O.
- Disadvantage: CPU is still involved in every data transfer (word by word), creating overhead for large transfers.

**3. Direct Memory Access (DMA):**

A dedicated DMA controller takes over the data transfer between the I/O device and main memory without continuous CPU involvement. The CPU initiates the transfer by programming the DMA controller (source address, destination address, byte count, direction), then the DMA controller handles the entire block transfer independently. The CPU is interrupted only once when the entire transfer is complete.

- Advantage: CPU is almost entirely free during the data transfer; efficient for large block transfers.
- Disadvantage: Requires additional hardware (DMA controller); DMA may steal bus cycles from the CPU.

---

### (b) Steps in Interrupt-Driven I/O [5 marks]

The following steps describe the interrupt I/O process:

1. **Device initiates interrupt:** One or more I/O devices complete their operation (or need attention) and assert an interrupt request signal on the interrupt line to the CPU.

2. **CPU completes current instruction:** The processor finishes executing the current instruction before responding to the interrupt. It checks the interrupt line at the end of each instruction cycle.

3. **CPU acknowledges interrupt:** The processor sends an interrupt acknowledge signal. If multiple devices are requesting simultaneously, a priority mechanism (hardware priority encoder, daisy chain, or vectored interrupt controller) determines which device is serviced first.

4. **Save processor state:** The CPU saves the current context -- at minimum the Program Counter (PC) and Processor Status Register (PSR/flags) -- typically onto the system stack. This allows the CPU to return to the interrupted program later.

5. **Identify interrupt source:** The CPU determines which device caused the interrupt. This can be done via:
   - Polling each device's status register
   - Vectored interrupts (device provides vector number pointing to its ISR)
   - Hardware priority interrupt controller (e.g., PIC/APIC)

6. **Load ISR address:** The processor loads the address of the appropriate Interrupt Service Routine (ISR) into the Program Counter, typically from an interrupt vector table.

7. **Execute ISR:** The ISR runs, performing the necessary I/O data transfer and any required processing. It reads/writes data from/to the device and clears the interrupt flag.

8. **Restore processor state:** Upon completion of the ISR, the saved context (PC, PSR, registers) is restored from the stack.

9. **Resume original program:** The CPU resumes execution from the point where it was interrupted, continuing normal processing until the next interrupt occurs.

### (c) Pipeline Timing Calculations [4 marks]

**Stage times:** IF=300ps, OF=250ps, E=350ps, M=700ps, OS=200ps

**(i) Non-pipelined execution time for one instruction:**

In a non-pipelined processor, the total time is the sum of all stages:

Time = IF + OF + E + M + OS
Time = 300 + 250 + 350 + 700 + 200 = **1800 ps**

**(ii) Pipelined execution time per stage (with latches):**

In a pipelined processor, each stage must take the same amount of time, equal to the slowest stage plus the latch overhead.

- Slowest stage = M = 700 ps
- Latch overhead = 20 ps per stage

Clock cycle time = 700 + 20 = **720 ps**

Time to fully execute one instruction through the pipeline:

Since there are 5 stages, and each stage takes 720 ps:

Time for one instruction to pass through all 5 stages = 5 x 720 = **3600 ps**

Note: While the latency for a single instruction increases with pipelining, the throughput improves because a new instruction completes every 720 ps (versus 1800 ps without pipelining), giving a throughput improvement factor of 1800/720 = 2.5x.

### (d) Data Hazards [3 + 3 marks]

**(i) Definition and Hardware Solution:**

A **data hazard** occurs when an instruction depends on the result of a previous instruction that has not yet completed its execution in the pipeline. The dependent instruction needs data that is not yet available, potentially leading to incorrect results if the pipeline does not handle it.

**Types of data hazards:**
- **RAW (Read After Write):** An instruction tries to read a register before a previous instruction has written its result to that register. This is the most common type.

**Hardware technique -- Forwarding (Data Bypassing):**

Forwarding (also called bypassing or short-circuiting) is a hardware technique that overcomes most data hazards without stalling the pipeline. Instead of waiting for a result to be written back to the register file (at the OS/Write-Back stage), the hardware adds special data paths that "forward" the result directly from the output of the stage where it is produced (e.g., the E or M stage) to the input of the stage where it is needed (e.g., the E stage of the next instruction).

This is implemented by adding multiplexers at the inputs of the ALU that can select between:
- The normal register file output
- The result forwarded from the EX/MEM pipeline register
- The result forwarded from the MEM/WB pipeline register

Forwarding eliminates most RAW hazards without requiring pipeline stalls.

**(ii) Identifying Data Hazards in the Instruction Sequence:**

```
Pipeline stages:      IF   OF   E    M    OS
1. ADD r0, r1, r2     IF   OF   E    M    OS
2. ADD r3, r0, r5          IF   OF   E    M    OS
3. STR r6, [r7]                 IF   OF   E    M    OS
4. LDR r8, [r7]                      IF   OF   E    M    OS
```

**Data Hazards Identified:**

1. **RAW hazard between instructions 1 and 2 on r0:**
   - Instruction 1 writes to r0 (result available after E stage, written at OS stage).
   - Instruction 2 reads r0 in its OF stage, but instruction 1 has not yet written r0 back to the register file.
   - This is a RAW hazard. Can be resolved by forwarding the result from instruction 1's E/M pipeline register to instruction 2's E stage input.

2. **RAW hazard between instructions 3 and 4 on r7 (address):**
   - Instruction 3 reads r7 for the memory address.
   - Instruction 4 also reads r7 for the memory address.
   - However, neither instruction writes to r7, so this is actually NOT a hazard -- both instructions only read r7.

3. **RAW hazard between instructions 3 and 4 on memory location [r7]:**
   - Instruction 3 (STR) writes data to memory address [r7] in its M stage.
   - Instruction 4 (LDR) reads from memory address [r7] in its M stage.
   - Since instruction 3's M stage occurs one cycle before instruction 4's M stage, instruction 4 may read the old value if the memory write has not completed. This is a **memory data hazard** (RAW through memory).
   - This requires forwarding from the memory stage or a pipeline stall.

**Summary of hazards:**
- **Hazard 1:** RAW on register r0 between instructions 1 and 2 (most significant).
- **Hazard 2:** RAW on memory location [r7] between instructions 3 (STR) and 4 (LDR) -- store followed by load to the same address.

### (e) Multicore Processor vs Coprocessor [4 marks]

**Multicore Processor:**

A multicore processor integrates two or more independent processing cores on a single chip (die). Each core is a fully functional, general-purpose processor capable of independently fetching, decoding, and executing instructions. All cores typically share the same instruction set architecture (ISA) and can run separate threads or processes simultaneously. They share resources such as the last-level cache and memory controller. Examples: dual-core, quad-core, octa-core CPUs.

**Coprocessor:**

A coprocessor is a supplementary processor designed to handle a specific type of task, offloading specialised work from the main CPU. It is not a general-purpose processor and cannot operate independently -- it works under the direction of the main CPU. It typically has a different, specialised instruction set optimised for its particular function. Examples: floating-point coprocessors (x87), graphics processing units (GPUs), cryptographic accelerators, digital signal processors (DSPs).

**Key Differences:**

| Aspect | Multicore Processor | Coprocessor |
|--------|-------------------|-------------|
| **Independence** | Each core can operate independently and run its own program | Cannot operate independently; works under CPU direction |
| **Instruction Set** | All cores share the same general-purpose ISA | Has a specialised instruction set for specific tasks |
| **Purpose** | General-purpose parallel processing | Offloads specific computational tasks (e.g., floating-point, graphics) |
| **Symmetry** | Cores are symmetric (identical capabilities) | Asymmetric -- different capabilities from the main CPU |
| **Scheduling** | OS can schedule any process/thread on any core | Main CPU explicitly delegates specific operations to the coprocessor |
