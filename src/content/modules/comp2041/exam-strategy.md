# Exam Strategy

## Time Allocation

| Section | Approx. Weight | Suggested Time (2hr exam) |
|---------|---------------|--------------------------|
| Vision/scope & requirements types | 15% | 18 min |
| Elicitation techniques | 20% | 24 min |
| Analysis & modelling | 20% | 24 min |
| SRS & validation | 20% | 24 min |
| Management & testing | 15% | 18 min |
| Short answer/definitions | 10% | 12 min |

## Topic Weighting

1. **Elicitation Techniques** -- Compare techniques, when to use which, pros/cons
2. **Analysis & Modelling** -- Use cases, UML diagrams, structuring requirements
3. **SRS Document** -- Structure, quality attributes (IEEE 830), writing good requirements
4. **Validation & Verification** -- Review techniques, inspection, walkthrough
5. **Requirements Management** -- Change control, traceability, baseline

## Common Question Types

- **Scenario-based:** Given a project scenario, identify appropriate elicitation techniques and justify
- **Write requirements:** Given a description, write functional/non-functional requirements that are SMART
- **Critique requirements:** Identify problems (ambiguity, incompleteness, inconsistency) in given requirements
- **Draw use case diagrams:** Identify actors, use cases, relationships
- **Compare and contrast:** Different elicitation methods, validation techniques, prioritisation schemes
- **Traceability:** Explain how to trace requirements through the lifecycle

## Key Frameworks

| Framework | Description |
|-----------|-------------|
| MoSCoW | Must/Should/Could/Won't -- prioritisation |
| IEEE 830 | Standard for SRS quality attributes |
| Volere Template | Requirements specification template with shells |
| SMART | Specific, Measurable, Achievable, Relevant, Time-bound |
| Kano Model | Classifies requirements as basic/performance/excitement |

## Exam Approach Tips

1. **Use the vocabulary precisely** -- distinguish elicitation from analysis, verification from validation
2. **For scenario questions** -- identify stakeholders first, then select techniques appropriate to the context
3. **When writing requirements** -- make them testable, unambiguous, and atomic (one requirement per statement)
4. **For critique questions** -- systematically check against IEEE 830 attributes
5. **Include examples** in your answers to demonstrate understanding

## Night Before Top 10 Checklist

1. Know the difference between functional and non-functional requirements with 3 examples of each
2. List 5 elicitation techniques with one pro and one con for each
3. Write the structure of a use case (actor, precondition, main flow, alternatives, postcondition)
4. Memorise IEEE 830 quality attributes: correct, unambiguous, complete, consistent, ranked, verifiable, modifiable, traceable
5. Know MoSCoW and when to use each category
6. Understand the difference between verification and validation
7. Be able to identify ambiguous requirements and rewrite them
8. Know the contents of a vision and scope document (problem statement, stakeholders, scope, constraints)
9. Understand change control process: request, impact analysis, approval, implementation, verification
10. Review traceability matrix structure: requirement ID -> design -> code -> test case
