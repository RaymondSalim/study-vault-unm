## A

| Term | Definition |
|------|-----------|
| **Acceptance Criteria** | Conditions that must be met for a requirement or story to be considered complete |
| **Acceptance Testing** | Testing performed by users/stakeholders to confirm the system meets their needs |
| **Activity Diagram** | UML diagram showing the flow of activities in a process, including branching and concurrency |
| **Actor** | An external entity (person, system, device) that interacts with the system |
| **Alternative Flow** | A variation from the main success scenario in a use case |

## B

| Term | Definition |
|------|-----------|
| **Baseline** | A formally approved snapshot of requirements at a point in time; changes require formal control |
| **Boundary Class** | In analysis modelling, a class representing a system interface (UI, API, hardware) |
| **Brainstorming** | Elicitation technique focused on generating creative ideas without initial criticism |
| **Business Requirement** | A high-level objective of the organisation that the system must support |

## C

| Term | Definition |
|------|-----------|
| **CCB (Change Control Board)** | Group responsible for evaluating and approving/rejecting change requests |
| **Change Request** | A formal proposal to modify a baselined requirement |
| **Constraint** | A restriction on the solution design imposed by technology, regulation, or business policy |
| **Context Diagram** | A Level 0 DFD showing the system as a single process with external entities |
| **Control Class** | In analysis modelling, a class that coordinates behaviour between boundary and entity classes |

## D

| Term | Definition |
|------|-----------|
| **Data Flow Diagram (DFD)** | Diagram showing how data moves through a system between processes, stores, and external entities |
| **Domain Model** | A conceptual model of the problem domain showing key entities and their relationships |

## E

| Term | Definition |
|------|-----------|
| **Elicitation** | The process of discovering and extracting requirements from stakeholders and other sources |
| **Entity Class** | In analysis modelling, a class representing a persistent domain object (stores data) |
| **Exception Flow** | A path in a use case that handles error conditions or failures |
| **<<extend>>** | A UML relationship where one use case optionally adds behaviour to another |
| **External Entity** | In a DFD, a person, system, or organisation outside the system boundary |

## F

| Term | Definition |
|------|-----------|
| **Feasibility** | Whether a requirement can realistically be implemented within constraints |
| **Fit Criterion** | A measurable condition that defines when a requirement is satisfied |
| **Functional Requirement (FR)** | A statement of what the system must do — a specific behaviour or function |
| **FURPS+** | Classification scheme: Functionality, Usability, Reliability, Performance, Supportability + constraints |

## G-H

| Term | Definition |
|------|-----------|
| **Given-When-Then** | Format for writing acceptance criteria (precondition, action, outcome) |
| **Gold-Plating** | Adding features or capabilities not requested by any stakeholder or requirement |

## I

| Term | Definition |
|------|-----------|
| **IEEE 830** | Superseded standard for SRS structure (replaced by IEEE 29148:2018) |
| **IEEE 29148** | Current standard for requirements engineering processes and documentation |
| **Impact Analysis** | Assessing the effect of a proposed change on other requirements, design, code, and tests |
| **<<include>>** | A UML relationship where one use case always incorporates another's behaviour |
| **Inspection (Fagan)** | Formal review with defined roles, phases, and metrics for finding defects |

## J-K

| Term | Definition |
|------|-----------|
| **JAD (Joint Application Development)** | A workshop-based elicitation technique involving cross-functional stakeholders |
| **Kano Model** | Classification of features by customer satisfaction impact: Basic, Performance, Excitement |

## L-M

| Term | Definition |
|------|-----------|
| **MoSCoW** | Prioritisation method: Must Have, Should Have, Could Have, Won't Have |
| **Main Success Scenario** | The primary (happy path) flow of steps in a use case |

## N

| Term | Definition |
|------|-----------|
| **Negotiation** | Process of resolving conflicting requirements between stakeholders |
| **Non-Functional Requirement (NFR)** | A quality attribute specifying how well the system performs (performance, security, usability, etc.) |

## P

| Term | Definition |
|------|-----------|
| **Planguage** | Tom Gilb's structured language for specifying measurable quality requirements (Scale, Meter, Must, Plan, Wish) |
| **Postcondition** | A condition that must be true after a use case completes successfully |
| **Precondition** | A condition that must be true before a use case can begin |
| **Problem Statement** | A concise description of the problem the system aims to solve |
| **Prototyping** | Building a preliminary version to clarify and validate requirements with stakeholders |

## R

| Term | Definition |
|------|-----------|
| **Requirements Churn** | The rate at which requirements change over time (measure of instability) |
| **Requirements Coverage** | The degree to which every requirement is addressed by at least one test case |
| **Requirements Management** | Ongoing control, tracking, and maintenance of requirements throughout the project |
| **Requirements Traceability** | The ability to follow a requirement from source through design, code, and test |

## S

| Term | Definition |
|------|-----------|
| **Scope** | The boundary defining what is and is not included in the system/project |
| **Scope Creep** | Uncontrolled expansion of scope without corresponding adjustment to time/resources |
| **SRS (Software Requirements Specification)** | Document describing the complete requirements for a software system |
| **Stakeholder** | Any person or organisation with an interest in or influence over the system |
| **Swimlane** | A visual division in an activity diagram showing which actor/role performs each activity |

## T

| Term | Definition |
|------|-----------|
| **Testability** | The degree to which a requirement allows a finite, cost-effective pass/fail test |
| **Traceability Matrix** | A table mapping requirements to their sources, design elements, code, and tests |
| **Trigger** | The event that initiates a use case |

## U-V

| Term | Definition |
|------|-----------|
| **UAT (User Acceptance Testing)** | Testing where end users verify the system meets their needs before go-live |
| **Use Case** | A structured description of a user's interaction with the system to achieve a goal |
| **Use Case Diagram** | UML diagram showing actors, use cases, and their relationships (include, extend, generalise) |
| **Validation** | Confirming the system meets actual stakeholder needs ("right product?") |
| **Verification** | Confirming the implementation matches the specification ("product built right?") |
| **Vision & Scope Document** | High-level document defining project objectives, stakeholders, and boundaries |

## W

| Term | Definition |
|------|-----------|
| **Walkthrough** | An author-led review where the document is presented and feedback gathered informally |
| **Wiegers Priority Matrix** | Prioritisation technique scoring benefit, penalty, cost, and risk for each requirement |
