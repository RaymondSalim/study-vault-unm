# Flashcards

| # | Question | Answer |
|---|----------|--------|
| 1 | What is a vision and scope document? | A high-level document defining the business context, stakeholders, project vision, scope boundaries, and success criteria for a software project. |
| 2 | What is the difference between functional and non-functional requirements? | Functional requirements describe what the system should do (features/behaviour); non-functional requirements describe quality attributes (performance, security, usability). |
| 3 | Name three elicitation techniques. | Interviews, questionnaires, workshops, observation, prototyping, document analysis, brainstorming. |
| 4 | What is a use case? | A description of how an actor interacts with the system to achieve a specific goal, including the main success scenario and alternative flows. |
| 5 | What does SMART stand for in requirements? | Specific, Measurable, Achievable, Relevant, Time-bound. |
| 6 | What is requirements traceability? | The ability to trace each requirement forward (to design/code/tests) and backward (to its source/stakeholder need). |
| 7 | What is the purpose of an SRS document? | The Software Requirements Specification formally documents all functional and non-functional requirements, serving as a contract between stakeholders and developers. |
| 8 | Define requirements validation. | The process of checking that documented requirements accurately reflect stakeholder needs and are complete, consistent, and feasible. |
| 9 | What is scope creep? | The uncontrolled expansion of project scope through addition of new requirements without corresponding adjustments to time, budget, or resources. |
| 10 | What is the difference between verification and validation? | Verification: "Are we building the product right?" (meets spec). Validation: "Are we building the right product?" (meets user needs). |
| 11 | What is a stakeholder? | Any person or organisation that has an interest in or is affected by the system being developed. |
| 12 | What is requirements prioritisation? | The process of ranking requirements by importance (e.g., using MoSCoW: Must have, Should have, Could have, Won't have). |
| 13 | Define a user story format. | "As a [role], I want [feature], so that [benefit]." |
| 14 | What is requirements negotiation? | The process of resolving conflicts between competing stakeholder requirements through discussion, trade-offs, and compromise. |
| 15 | What is a prototype in elicitation? | A preliminary model of the system used to gather feedback from stakeholders and clarify requirements before full development. |
| 16 | What are the IEEE 830 quality attributes for requirements? | Correct, unambiguous, complete, consistent, ranked for importance, verifiable, modifiable, traceable. |
| 17 | What is an acceptance criterion? | A condition that a system must satisfy to be accepted by a stakeholder; defines "done" for a requirement. |
| 18 | Explain the difference between elicitation and analysis. | Elicitation is gathering raw requirements from stakeholders; analysis is organising, modelling, and refining them into a structured, consistent set. |
| 19 | What is a context diagram? | A high-level diagram showing the system boundary, external entities (actors), and data flows between them and the system. |
| 20 | What is requirements management? | The ongoing process of tracking, controlling changes to, and maintaining requirements throughout the project lifecycle. |
| 21 | What is a use case diagram? | A UML diagram showing actors, use cases (system functions), and their relationships (include, extend, generalisation). |
| 22 | Define feasibility in requirements. | Assessing whether a requirement can be implemented within the given technical, economic, and time constraints. |
| 23 | What is the purpose of requirements testing? | To verify that the implemented system satisfies all specified requirements through systematic test cases traced to requirements. |
| 24 | What is gold plating? | Adding features or functionality beyond what was specified in the requirements, often without stakeholder approval. |
| 25 | What is a requirements baseline? | A formally approved set of requirements at a point in time, serving as the basis for development; changes after this need formal change control. |
