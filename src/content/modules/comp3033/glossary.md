## A

| Term | Definition |
|------|-----------|
| **Acceptance testing** | Testing by or with the customer to verify the system meets their requirements |
| **Agile** | Iterative, incremental development philosophy valuing working software and customer collaboration |

## B

| Term | Definition |
|------|-----------|
| **Basis path testing** | White-box technique that derives test cases from linearly independent paths through the CFG |
| **Black-box testing** | Testing based on specifications without knowledge of internal code structure |
| **Boundary Value Analysis (BVA)** | Testing at the edges of equivalence partitions where defects cluster |

## C

| Term | Definition |
|------|-----------|
| **CBO (Coupling Between Objects)** | C&K metric: count of classes a class is coupled to |
| **CFG (Control Flow Graph)** | Directed graph representing all paths through a program's code |
| **CI (Continuous Integration)** | Practice of frequently merging code and running automated builds/tests |
| **Cyclomatic complexity** | Quantitative measure of linearly independent paths: M = E - N + 2 |
| **CVE** | Common Vulnerabilities and Exposures — standardised identifier for known security flaws |

## D

| Term | Definition |
|------|-----------|
| **DAST (Dynamic Application Security Testing)** | Black-box security testing against a running application |
| **Defect detection** | QA strategy focused on finding and removing defects in the product |
| **Defect prevention** | QA strategy focused on processes that prevent defects from being introduced |
| **DevSecOps** | Integration of security practices into the DevOps pipeline |
| **DIT (Depth of Inheritance Tree)** | C&K metric: levels of inheritance from root to class |

## E

| Term | Definition |
|------|-----------|
| **Equivalence partitioning** | Dividing input domain into classes with equivalent expected behaviour |
| **Error** | Incorrect internal program state caused by executing a fault |
| **Espresso** | Android UI testing framework for writing automated UI tests |

## F

| Term | Definition |
|------|-----------|
| **Failure** | Externally observable incorrect behaviour of the system |
| **Fault** | Static defect in the source code (a bug) |
| **Fixture** | Known test environment setup/teardown (`@BeforeEach` / `@AfterEach` in JUnit) |

## H

| Term | Definition |
|------|-----------|
| **Hockey stick curve** | Latency behaviour under increasing load: flat until saturation, then exponential rise |

## I

| Term | Definition |
|------|-----------|
| **ISO/IEC 25010** | International standard defining 8 product quality characteristics |
| **ISO/IEC 25019** | International standard defining 5 quality-in-use characteristics |
| **Integration testing** | Testing interactions between combined modules/components |

## J

| Term | Definition |
|------|-----------|
| **JUnit** | Java unit testing framework (current version: JUnit 5 / Jupiter) |

## L

| Term | Definition |
|------|-----------|
| **LCOM (Lack of Cohesion of Methods)** | C&K metric: measures how unrelated a class's methods are |
| **Load testing** | Verifying system meets NFRs under expected load conditions |
| **LOC (Lines of Code)** | Simple size metric counting source code lines |

## M

| Term | Definition |
|------|-----------|
| **McCall's model** | Quality model (1977) with 11 factors in 3 categories: operation, revision, transition |
| **MDTD (Model-Driven Test Design)** | Systematic approach to test design using models of the software |

## N

| Term | Definition |
|------|-----------|
| **NFR (Non-Functional Requirement)** | Quality requirement (performance, security, usability) that must be specific and measurable |
| **NOC (Number of Children)** | C&K metric: count of direct subclasses |

## O

| Term | Definition |
|------|-----------|
| **OWASP Top 10** | Standard awareness document listing the 10 most critical web application security risks |

## P

| Term | Definition |
|------|-----------|
| **Parameterized test** | JUnit test that runs the same logic with multiple different data inputs |
| **Prepared statement** | SQL mechanism separating code from data to prevent injection attacks |

## R

| Term | Definition |
|------|-----------|
| **RFC (Response for Class)** | C&K metric: count of methods that can be invoked from a class |
| **RIPR model** | Test effectiveness model: Reachability, Infection, Propagation, Reveal |

## S

| Term | Definition |
|------|-----------|
| **SAST (Static Application Security Testing)** | White-box security analysis of source code without execution |
| **SCA (Software Composition Analysis)** | Scanning third-party dependencies for known vulnerabilities |
| **Shift left** | Moving testing/security activities earlier in the SDLC |
| **Soak testing** | Running normal load for extended periods to find time-dependent defects |
| **SQA plan** | Document defining quality activities, standards, and processes for a specific project |
| **SQL injection** | Attack that manipulates database queries through unsanitised user input |
| **Stress testing** | Testing beyond expected capacity to find the system's breaking point |
| **System testing** | Testing the complete integrated system against specifications |

## T

| Term | Definition |
|------|-----------|
| **TDD (Test-Driven Development)** | Development practice: write failing test → minimal code to pass → refactor |
| **TQM (Total Quality Management)** | Management philosophy where quality is everyone's responsibility |

## U

| Term | Definition |
|------|-----------|
| **UAT (User Acceptance Testing)** | Final testing phase where users verify the system meets business needs |
| **Unit testing** | Testing individual methods/classes in isolation |

## V

| Term | Definition |
|------|-----------|
| **V-Model** | Development model where each development phase has a corresponding testing phase |
| **V&V (Verification & Validation)** | Verification: building it right (process). Validation: building the right thing (product). |

## W

| Term | Definition |
|------|-----------|
| **White-box testing** | Testing based on knowledge of internal code structure |
| **WMC (Weighted Methods per Class)** | C&K metric: sum of cyclomatic complexity of all methods in a class |
