## A-D

| Term | Definition |
|------|-----------|
| **Abstraction** | Hiding complexity by exposing only essential features through a simplified interface |
| **Acceptance Testing** | Testing by end-users/customers to verify the system meets business requirements |
| **Actor** | An external entity (person or system) that interacts with the system in a use case |
| **Aggregation** | A "has-a" relationship where the part can exist independently of the whole (hollow diamond ◇) |
| **Agile** | A family of iterative development methodologies that embrace change and deliver incrementally |
| **Association** | A structural relationship between classes indicating they are linked |
| **Black-Box Testing** | Testing based on requirements/specifications without knowledge of internal code |
| **Boundary Value Analysis (BVA)** | Testing at the edges of equivalence partitions (min, min+1, max-1, max) |
| **Branch Coverage** | Every decision point (true/false) is taken at least once |
| **Class Diagram** | UML diagram showing classes, attributes, methods, and relationships |
| **Cohesion** | Degree to which elements within a module belong together (HIGH is good) |
| **Composition** | A strong "owns" relationship where the part's lifecycle depends on the whole (filled diamond ◆) |
| **Coupling** | Degree of interdependence between modules (LOW is good) |
| **Decision Table** | A table mapping all combinations of conditions to their resulting actions |
| **Dependency** | The weakest UML relationship; one element temporarily uses another |
| **Dependency Inversion** | SOLID principle: depend on abstractions, not concrete implementations |

## E-I

| Term | Definition |
|------|-----------|
| **Encapsulation** | Bundling data and methods together while hiding internal state from external access |
| **Equivalence Partitioning** | Dividing input domain into classes where all values in a class behave the same |
| **Extend (<<extend>>)** | Use case relationship where an optional behaviour may add to a base use case |
| **Functional Requirement** | A statement of what the system SHALL DO (behaviour, features, functions) |
| **Generalisation** | An inheritance relationship in UML (hollow triangle points to parent) |
| **Include (<<include>>)** | Use case relationship where a base case always incorporates another use case |
| **Inheritance** | OOP mechanism where a subclass acquires properties/methods of a superclass ("is-a") |
| **Integration Testing** | Testing the interaction between combined modules/components |
| **Interface** | A contract specifying methods a class must implement (no implementation provided) |
| **Interface Segregation** | SOLID principle: clients should not depend on methods they don't use |
| **INVEST** | Criteria for good user stories: Independent, Negotiable, Valuable, Estimable, Small, Testable |
| **Iteration** | A single development cycle producing a working increment |

## L-P

| Term | Definition |
|------|-----------|
| **Lifeline** | Dashed vertical line in a sequence diagram representing an object's existence over time |
| **Liskov Substitution** | SOLID principle: subtypes must be substitutable for their base types |
| **MoSCoW** | Prioritisation technique: Must have, Should have, Could have, Won't have |
| **Multiplicity** | Notation on associations indicating how many instances participate (e.g., 1..*, 0..1) |
| **Non-Functional Requirement (NFR)** | A quality constraint on HOW the system performs (performance, security, usability) |
| **Observer Pattern** | Design pattern where objects subscribe to events and are notified of changes |
| **Open/Closed Principle** | SOLID principle: open for extension, closed for modification |
| **OOP** | Object-Oriented Programming: paradigm based on objects containing data and behaviour |
| **Polymorphism** | Ability of different classes to respond to the same message/method in different ways |
| **Precondition** | A condition that must be true before a use case or method can execute |
| **Postcondition** | A condition that must be true after a use case or method completes |
| **Product Backlog** | Scrum: prioritised list of all desired product features and improvements |
| **Product Owner** | Scrum role: represents stakeholders, manages product backlog |
| **Prototype** | An early model of a system built to explore requirements or test concepts |

## R-S

| Term | Definition |
|------|-----------|
| **Realisation** | UML relationship where a class implements an interface (dashed line + hollow triangle) |
| **Refactoring** | Restructuring code without changing external behaviour to improve quality |
| **Regression Testing** | Re-running tests after changes to ensure nothing previously working is broken |
| **Requirements Elicitation** | The process of gathering requirements from stakeholders |
| **Scenario** | A specific path through a use case (one instance of execution) |
| **Scrum** | An Agile framework using sprints, backlogs, and defined roles |
| **Scrum Master** | Scrum role: facilitates the process, removes impediments |
| **Sequence Diagram** | UML diagram showing time-ordered message exchanges between objects |
| **Single Responsibility** | SOLID principle: a class should have only one reason to change |
| **SOLID** | Five OOP design principles: SRP, OCP, LSP, ISP, DIP |
| **Spiral Model** | SDLC model combining iterative development with explicit risk analysis |
| **Sprint** | Scrum: a fixed timebox (usually 2 weeks) producing a potentially shippable increment |
| **SRS** | Software Requirements Specification: formal document of all requirements |
| **Stakeholder** | Any person or group with an interest in the system |
| **State Transition Testing** | Black-box technique testing valid and invalid state changes |
| **Statement Coverage** | Every line of code is executed at least once (weakest coverage metric) |
| **Strategy Pattern** | Design pattern allowing interchangeable algorithms at runtime |
| **System Testing** | Testing the complete, integrated system against requirements |

## T-Z

| Term | Definition |
|------|-----------|
| **TDD** | Test-Driven Development: write test → fail → implement → pass → refactor |
| **Traceability** | Ability to link requirements to design, code, and test cases |
| **Unit Testing** | Testing individual methods/classes in isolation |
| **Use Case** | A description of how an actor achieves a goal using the system |
| **User Story** | Agile requirement format: "As a [role], I want [goal], so that [benefit]" |
| **UML** | Unified Modelling Language: standardised notation for software modelling |
| **Validation** | "Are we building the RIGHT product?" - meets user needs |
| **Verification** | "Are we building the product RIGHT?" - meets specification |
| **V-Model** | SDLC model pairing each development phase with a corresponding test phase |
| **V&V** | Verification and Validation: complementary quality assurance activities |
| **Waterfall** | Sequential SDLC model where each phase must complete before the next begins |
| **White-Box Testing** | Testing based on knowledge of internal code structure |
