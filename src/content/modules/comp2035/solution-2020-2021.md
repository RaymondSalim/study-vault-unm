### 1b) Preventing Starvation in Multi-Level Queues (2 marks)

Starvation in a multi-level queue occurs when lower-priority queues never get CPU time because higher-priority queues always have ready processes. Two main prevention strategies:

1. **Time-slice between queues (aging via queue promotion):** Allocate a percentage of CPU time to each queue (e.g., 80% to foreground/interactive, 20% to background/batch). This guarantees every queue gets some CPU time regardless of activity in higher-priority queues.

2. **Aging:** If a process has waited in a lower-priority queue beyond a threshold time, promote it to a higher-priority queue. This ensures that no process waits indefinitely. Eventually any starved process will reach the highest-priority queue.

### 1c) Scheduling Policies (2 marks)

**(i) Time-sharing system:** **Round Robin (RR)** scheduling is appropriate. In a time-sharing system, multiple interactive users share the system and each expects responsive service. Round Robin gives each process a fair, fixed time quantum in cyclic order. This ensures no process monopolises the CPU and response times remain short and predictable for all users.

**(ii) Batch system:** **Shortest Job First (SJF)** or **First Come First Served (FCFS)** scheduling is appropriate. In a batch system, there are no interactive users waiting; the goal is to maximise throughput and minimise average turnaround time. SJF is optimal for minimising average waiting time. FCFS is simple and fair in order of submission. Neither requires preemption, which suits the non-interactive nature of batch processing.

### 1d) LRU Implementation in Hardware (4 marks)

**Hardware implementation of LRU:**

The most common hardware approach uses a **counter-based** or **matrix-based** scheme:

1. **Counter-based:** Each page table entry has an associated time-of-use field (counter). The CPU has a logical clock or counter that increments on every memory reference. Whenever a page is referenced, the current clock value is copied into that page's counter field. When a page must be replaced, the OS scans all counters and selects the page with the smallest value (oldest reference). This requires the hardware to update the counter on every memory access.

2. **Stack/Matrix-based:** Maintain an n x n bit matrix (for n frames). When page i is referenced, set all bits in row i to 1, then set all bits in column i to 0. The page with the lowest binary row value is the LRU page. This gives exact LRU ordering but requires O(n^2) hardware bits.

**Is LRU best in all cases?**

No. LRU is not optimal in all cases:
- **LRU performs poorly with sequential/cyclic access patterns.** For example, if a process repeatedly scans through pages 1,2,3,4,5 in sequence with only 4 frames, LRU will fault on every access (Belady's anomaly does not apply to LRU, but the cyclic pattern still causes constant eviction of the next-needed page).
- The **optimal algorithm (OPT/MIN)** always outperforms or matches LRU because it has future knowledge. LRU is merely a heuristic approximation of OPT based on the assumption that recent past predicts future use.
- **Hardware overhead:** The counter or matrix approach adds significant hardware complexity and cost, making it impractical for large page tables. In practice, approximations like the clock algorithm (second-chance) or reference bits are used instead.

## Question 2: Concurrency and Deadlock (25 Marks)

### 2a) Mutexes vs Semaphores (3 marks)

**Mutex (Mutual Exclusion Lock):**
A mutex is a synchronisation primitive that provides exclusive access to a shared resource. It has two states: locked and unlocked. Only the thread that locked the mutex can unlock it (ownership semantics). It is used to protect critical sections where only one thread may execute at a time.

**Semaphore:**
A semaphore is a synchronisation primitive based on an integer counter with two atomic operations: wait (P/down) and signal (V/up). A semaphore can be initialised to any non-negative value. A **binary semaphore** (initialised to 1) behaves similarly to a mutex. A **counting semaphore** (initialised to n > 1) allows up to n threads to access a resource concurrently.

**Key differences:**

| Aspect | Mutex | Semaphore |
|--------|-------|-----------|
| Ownership | Yes — only the locking thread can unlock | No ownership — any thread can signal |
| Purpose | Mutual exclusion only | Mutual exclusion AND signalling/coordination |
| Value range | Binary (locked/unlocked) | Integer (0 to n) |
| Use case | Protecting critical sections | Resource counting, producer-consumer, ordering |
| Priority inheritance | Often supported | Typically not supported |

### 2b) Atomicity of Wait/Signal Operations (3 marks)

**Yes, wait (P) and signal (V) operations on semaphores must be atomic.** They are defined as indivisible operations — no other process can interleave during their execution.

**Why atomicity is essential:**

If wait/signal were not atomic, a race condition could occur:

```
// Non-atomic wait (INCORRECT - race condition possible)
wait(S):
    while (S <= 0)       // Step 1: Test
        ; // busy wait
    S = S - 1;           // Step 2: Decrement

// Non-atomic signal (INCORRECT - race condition possible)
signal(S):
    S = S + 1;           // Increment
```

**Problem scenario without atomicity:**
- Suppose S = 1. Two threads T1 and T2 both call wait(S).
- T1 checks S > 0 (true, S=1) — context switch before decrement.
- T2 checks S > 0 (true, S=1) — T2 decrements S to 0 and enters critical section.
- T1 resumes and decrements S to -1 and also enters critical section.
- **Mutual exclusion violated!**

**Correct atomic pseudocode:**

```
wait(S):
    disable_interrupts()   // or acquire hardware lock
    while (S <= 0):
        enable_interrupts()
        block(current_process, S.queue)
        disable_interrupts()
    S = S - 1
    enable_interrupts()

signal(S):
    disable_interrupts()
    S = S + 1
    if process_waiting_on(S.queue):
        wakeup(next_process, S.queue)
    enable_interrupts()
```

In practice, atomicity is achieved through disabling interrupts (uniprocessor) or using hardware atomic instructions like test-and-set or compare-and-swap (multiprocessor).

### 2c) Strict Alternation Code Analysis (4 marks)

**Given code:**

```c
int turn = 0;
int sum = 0;

// Thread P0:                     // Thread P1:
while (turn != 0) ;              while (turn != 1) ;
sum++;                            sum++;
turn = 1;                         turn = 0;
```

**(i) Is the code synchronised correctly?**

**No, the code is NOT synchronised correctly.** There are two problems:

1. **The turn variable is initialised to 0**, meaning P0 goes first. But after P0 sets turn=1, P1 runs and sets turn=0, and then neither thread loops again — this appears to be a one-shot execution, not a loop. If this is meant to be inside a loop, then:

2. **The `sum++` operation is not properly protected.** While strict alternation does ensure mutual exclusion (only one thread enters at a time), the bigger issue is that `turn` is set incorrectly. P0 sets `turn = 1` (should hand off to P1), and P1 sets `turn = 0` (hands back to P0). This is correct for alternation.

Actually, re-reading: the turn starts at 0. P0 sees turn==0, so it does NOT spin — it proceeds to `sum++` then sets `turn=1`. P1 was spinning on `while(turn!=1)`, now it exits the spin, does `sum++`, sets `turn=0`.

**The problem:** If this is a single execution (not in a loop), the synchronisation is correct — P0 runs first, then P1. However, `sum++` is not atomic. If intended to protect `sum++` as a critical section, the code does achieve mutual exclusion for a single pass.

**Fix (if repeated execution is intended):**

```c
int turn = 0;
int sum = 0;

// Thread P0:                     // Thread P1:
while (true) {                    while (true) {
    while (turn != 0) ;              while (turn != 1) ;
    sum++;                            sum++;
    turn = 1;                         turn = 0;
}                                 }
```

**(ii) Does it meet mutual exclusion requirements?**

**Yes**, the strict alternation mechanism does satisfy mutual exclusion. At any given time, only one thread can be past its while-loop (because turn can only be 0 or 1). When P0 is in its critical section (sum++), turn is 0, so P1 is blocked in its spin loop. When P1 is in its critical section, turn is 1, so P0 is blocked. Therefore, both threads cannot be in their critical sections simultaneously.

However, it violates other desirable properties (see part d).

### 2d) Critical Evaluation of Strict Alternation (6 marks)

Strict alternation is evaluated against the four requirements for a correct critical section solution:

**1. Mutual Exclusion: SATISFIED**
Only one process can be in the critical section at a time. The `turn` variable ensures that when one process is executing its critical section, the other is spinning. Since `turn` is a single variable that is either 0 or 1, both processes cannot simultaneously pass their entry conditions.

**2. Progress (No deadlock): PARTIALLY VIOLATED**
If one process does not wish to enter the critical section, or has terminated, it cannot set `turn` to allow the other process to proceed. For example, if P0 finishes and never executes again, `turn` remains at 1 forever after P1's first entry — but then P1 sets turn=0 and can never re-enter because turn will never become 1 again. A process outside its critical section is blocking another process — this violates the progress requirement.

**3. Bounded Waiting: SATISFIED (conditionally)**
If both processes are continuously requesting entry, each will wait at most one turn. After P0 enters, P1 is guaranteed to be next, and vice versa. There is a strict bound of one alternation.

**4. No assumptions about speed/number of CPUs: SATISFIED**
The algorithm does not depend on hardware speed or the number of processors.

**Key weaknesses of strict alternation:**

- **Enforced strict ordering:** Processes must alternate exactly. P0 cannot enter twice in succession even if P1 has no interest in the critical section. This leads to unnecessary blocking.
- **Busy waiting (spinlock):** The spinning process wastes CPU cycles. On a single-core system, this is pure waste. On multicore, it may be acceptable for short waits.
- **Violation of progress:** The scheme couples the processes unnecessarily. A slow process or one that crashes will prevent the other from making progress indefinitely.
- **Not scalable:** Extending to n>2 processes is complex and maintains the same limitations.

**Conclusion:** Strict alternation is a simple but inadequate solution. It satisfies mutual exclusion but violates progress. Superior solutions include Peterson's algorithm, hardware atomic instructions (test-and-set, compare-and-swap), or OS-provided primitives (mutexes, semaphores).

### 2e) Safeguards to Prevent Indefinite Waiting (4 marks)

Indefinite waiting (starvation) occurs when a process is perpetually denied access to a resource. The following safeguards prevent this:

1. **Aging:** Gradually increase the priority of waiting processes over time. Eventually, any waiting process will reach the highest priority and be served. This is the most common anti-starvation mechanism in scheduling and resource allocation.

2. **FIFO (First-In-First-Out) queuing:** Service requests in the order they arrive. A FIFO queue guarantees bounded waiting because each request has at most n-1 other requests ahead of it (where n is the total number of competing processes).

3. **Bounded waiting guarantee in critical section solutions:** The algorithm must ensure that there is a limit on the number of times other processes can enter the critical section while a given process is waiting. Peterson's algorithm and the bakery algorithm provide this guarantee.

4. **Fair scheduling policies:** Use scheduling algorithms that explicitly prevent starvation, such as Round Robin (every process gets a time quantum), fair-share scheduling, or lottery scheduling (every process has at least one ticket, guaranteeing non-zero probability of selection).

5. **Resource allocation ordering:** In resource allocation, ensure requests are granted in a fair order (e.g., the banker's algorithm considers all processes, not just high-priority ones).

### 2f) Unsafe State Does Not Equal Deadlock (2 marks)

An **unsafe state** means that no safe sequence exists — there is no guaranteed order in which all processes can complete without deadlock, given current allocations and maximum future claims. However, this does NOT mean deadlock has occurred or will necessarily occur.

**Why unsafe does not imply deadlock:**

1. **Deadlock requires all four conditions simultaneously** (mutual exclusion, hold and wait, no preemption, circular wait). An unsafe state merely means these conditions *could* arise, not that they have.

2. **Processes may not actually request their maximum.** The unsafe state analysis assumes worst-case behaviour (every process requests its maximum). In practice, processes may release resources, request fewer than maximum, or terminate early — avoiding deadlock entirely.

3. **Example:** Consider a system in an unsafe state. If process P1 finishes without requesting additional resources and releases its held resources, the system may return to a safe state. The unsafe state was merely a *potential* for deadlock, not a certainty.

**In summary:** Safe state guarantees no deadlock. Unsafe state means deadlock is *possible* but not *inevitable*. Deadlock is a subset of unsafe states.

### 3b) I-node File Systems (6 marks)

**(i) Description and implementation principles:**

An **i-node (index node)** is a fixed-size data structure stored on disk that describes a file. Each file has exactly one i-node containing:

- **File metadata:** owner, permissions, timestamps (created, modified, accessed), file size, reference count (number of hard links).
- **Data block pointers:** A set of pointers to the disk blocks that hold the file's data:
  - **Direct pointers** (typically 12): Point directly to the first 12 data blocks of the file. For small files, this is all that is needed.
  - **Single indirect pointer:** Points to a block containing pointers to data blocks. If a pointer is 4 bytes and a block is 4KB, this adds 1024 more block pointers.
  - **Double indirect pointer:** Points to a block of pointers, each of which points to a block of pointers to data blocks. Adds 1024 x 1024 = 1,048,576 more blocks.
  - **Triple indirect pointer:** Three levels of indirection. Adds 1024^3 more blocks.

The **directory** maps file names to i-node numbers. To access a file, the OS looks up the name in the directory, retrieves the i-node number, reads the i-node from disk, and follows the block pointers to access file data.

I-nodes are stored in a dedicated region of the disk (the i-node table). Each i-node has a unique number within the file system.

**(ii) How i-nodes resolve shortcomings of contiguous and FAT allocation:**

**Shortcomings of contiguous allocation that i-nodes solve:**
- **External fragmentation:** Contiguous allocation requires files to occupy consecutive blocks. Over time, free space becomes fragmented and large files cannot be stored even if total free space is sufficient. I-nodes allow blocks to be scattered anywhere on disk — no contiguous requirement.
- **File growth difficulty:** With contiguous allocation, growing a file may require relocating it entirely. I-nodes allow files to grow by simply allocating new blocks anywhere and adding pointers.
- **Pre-declaration of file size:** Contiguous allocation often requires knowing the file size in advance. I-nodes allow dynamic growth.

**Shortcomings of FAT (linked list) allocation that i-nodes solve:**
- **Poor random access:** In FAT, accessing block n requires traversing the chain from the beginning (O(n)). I-nodes provide direct pointers for quick access to any block. Even indirect blocks require at most 3-4 disk reads regardless of file size (O(1) for small files, O(log n) for large files).
- **FAT table size:** The FAT must have one entry per disk block and must reside in memory for performance. For large disks, this table becomes enormous. I-nodes only need to be loaded for open files — the i-node for a 1TB file is the same small fixed size.
- **Reliability:** If the FAT is corrupted, all files are lost. I-nodes distribute metadata across the disk; corruption of one i-node affects only one file.

### 3c) I/O Buffering (6 marks)

**Types of I/O buffering:**

1. **No buffering:** Data is transferred directly between the device and user process memory. The process must wait for each I/O operation to complete before proceeding.

2. **Single buffering:** The OS allocates one system buffer in kernel space. Data is read into this buffer from the device, then copied to user space. While the user processes the current block, the next block can be read into the buffer (overlap of I/O and computation).

3. **Double buffering:** Two system buffers are used. While the user processes data from one buffer, the OS fills the other. When the user finishes with buffer A, it switches to buffer B while buffer A is refilled. This eliminates the wait time between consecutive reads.

4. **Circular buffering:** Multiple buffers (more than two) arranged in a circular queue. The producer (device) fills buffers and the consumer (process) empties them. This handles bursty I/O where production and consumption rates vary.

**Why buffering is required:**

- **Speed mismatch:** Devices are much slower than CPUs. Buffering allows the CPU to continue processing while I/O occurs in the background.
- **Transfer size mismatch:** Devices may produce/consume data in fixed-size units (e.g., disk sectors of 512 bytes) that differ from what the application needs.
- **Supports copy semantics:** The OS can snapshot the data to be written, allowing the user process to modify its buffer immediately.

**Advantage case — double buffering for streaming input:**
When reading a large file sequentially, double buffering allows continuous overlap of disk I/O and processing. While the application processes block N from buffer A, the disk controller fills buffer B with block N+1. The application never waits for I/O, achieving near-maximum throughput.

**Disadvantage case — excessive buffering for real-time/interactive systems:**
Buffering introduces latency. In a real-time audio system, buffering adds delay between input capture and output. Large buffers increase latency to unacceptable levels for interactive applications (e.g., live music performance, video conferencing). Additionally, buffering consumes kernel memory that could be used for other purposes, and the extra copy from kernel buffer to user space adds CPU overhead.

### 3d) Why Concurrent Programs Are Harder to Test/Debug (4 marks)

Concurrent programs are significantly harder to test and debug than sequential programs for the following reasons:

1. **Non-deterministic execution order:** Thread scheduling depends on OS decisions, system load, and timing. The same program with the same inputs can produce different execution interleavings each run. A bug may appear in one run and disappear in the next, making it extremely difficult to reproduce.

2. **Heisenbugs (probe effect):** Adding debugging tools (print statements, breakpoints, debuggers) alters the timing of thread execution. This can make bugs disappear during debugging ("Heisenbug") because the instrumentation changes the interleaving. The act of observing the program changes its behaviour.

3. **Exponential state space:** For n threads each with k steps, there are potentially (nk)! / (k!)^n possible interleavings. Exhaustive testing of all interleavings is computationally infeasible. Bugs may only manifest in rare interleavings that are statistically unlikely during testing but can occur in production.

4. **Emergent behaviour:** Race conditions, deadlocks, livelocks, and starvation arise from interactions between threads, not from individual thread behaviour. Each thread may be correct in isolation, but their combination produces errors. Traditional unit testing of individual components cannot catch these interaction bugs.

5. **Difficulty in reproducing failures:** When a concurrent bug occurs in production, reproducing it in a test environment requires recreating the exact timing conditions. This is often impossible because the bug depends on specific scheduling decisions, cache states, memory access patterns, and system load.

### 3e) Conditions/Restrictions for Multitasking with Real-Time Processes (3 marks)

Real-time processes have strict timing constraints — they must complete by a deadline. Multitasking with real-time processes requires the following conditions and restrictions:

1. **Priority-based preemptive scheduling:** Real-time processes must have the highest priority and be able to preempt any non-real-time process immediately. The scheduler must guarantee that when a real-time process becomes ready, it gets the CPU with minimal latency (bounded preemption delay).

2. **Guaranteed CPU time / admission control:** Before admitting a new real-time process, the system must verify that all existing real-time deadlines can still be met. This requires known worst-case execution times (WCET) for all real-time tasks. Schedulability analysis (e.g., Rate Monotonic Analysis for periodic tasks) must confirm feasibility: sum of (execution time / period) for all tasks must be within bounds.

3. **Bounded interrupt latency and kernel preemption:** The OS kernel must be preemptible (or have bounded non-preemptible sections). Interrupt handling latency must be bounded and small. Priority inversion must be prevented using protocols like **Priority Inheritance** or **Priority Ceiling** — if a low-priority process holds a resource needed by a real-time process, the low-priority process temporarily inherits the real-time priority to release the resource quickly.

4. **Deterministic memory management:** No page faults allowed for real-time processes (pages must be locked in memory / pinned). Virtual memory paging introduces unbounded delays that violate real-time guarantees.

5. **Resource reservation:** CPU, memory, and I/O bandwidth may need to be reserved for real-time processes, ensuring non-real-time tasks cannot consume resources needed to meet deadlines.
