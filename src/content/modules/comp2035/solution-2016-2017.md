## Question 1: Memory Management and File Management (25 Marks)

### 1a) Page Replacement Policies (6 marks)

Given table:

| Page Frame | Load Time | Last Reference | R bit | M bit |
|-----------|-----------|---------------|-------|-------|
| 0 | 167 | 374 | 1 | 1 |
| 1 | 321 | 321 | 0 | 0 |
| 2 | 254 | 306 | 0 | 1 |
| 3 | 154 | 331 | 1 | 0 |

#### (i) FIFO — Page Frame 3

FIFO replaces the page that was loaded **earliest**. Looking at Load Times:
- Frame 3: 154 (earliest)
- Frame 0: 167
- Frame 2: 254
- Frame 1: 321

**Frame 3** is replaced because it has the earliest load time (154).

#### (ii) LRU — Page Frame 2

LRU replaces the page that was **least recently used** (oldest Last Reference time):
- Frame 2: 306 (least recent)
- Frame 1: 321
- Frame 3: 331
- Frame 0: 374

**Frame 2** is replaced because it has the oldest last reference time (306).

#### (iii) NRU — Page Frame 1

NRU (Not Recently Used) classifies pages into 4 classes based on R and M bits:
- Class 0: R=0, M=0 → Frame 1
- Class 1: R=0, M=1 → Frame 2
- Class 2: R=1, M=0 → Frame 3
- Class 3: R=1, M=1 → Frame 0

NRU replaces a page from the lowest non-empty class. **Frame 1** (Class 0) is replaced.

### 1b) Demand Paging and Thrashing

#### (i) Demand Paging (4 marks)

Demand paging is a memory management scheme where pages are only loaded into physical memory when they are actually needed (referenced), rather than loading the entire process into memory at startup.

When a process accesses a page not currently in memory, a **page fault** occurs. The OS then:
1. Traps to the kernel.
2. Locates the required page on disk (in swap space).
3. Finds a free frame (or evicts a page using a replacement algorithm).
4. Loads the page from disk into the free frame.
5. Updates the page table to map the virtual page to the physical frame.
6. Restarts the instruction that caused the fault.

This approach means only the pages actually used by the process are in memory, allowing more processes to fit in memory simultaneously and reducing startup time.

#### (ii) Thrashing (5 marks)

**Thrashing** occurs when a system spends more time paging (moving pages between disk and memory) than executing useful work. It happens when processes collectively require more physical memory than is available, causing continuous page faults.

**Detection:** The system can detect thrashing by monitoring:
- The page fault rate — if it rises above an acceptable threshold.
- CPU utilisation — thrashing causes CPU utilisation to drop dramatically while disk I/O spikes.
- The working set sizes of all processes — if their sum exceeds available physical memory.

**Elimination:**
- **Reduce the degree of multiprogramming** — suspend or swap out one or more processes to free frames for remaining processes.
- **Use working set model** — allocate each process enough frames to hold its working set; if total working set size exceeds memory, swap a process out.
- **Page fault frequency (PFF) control** — set upper and lower bounds on acceptable page fault rates; if a process exceeds the upper bound, give it more frames; if below the lower bound, reclaim frames.

### 1c) Virtual Memory in Real-Time Systems (4 marks)

Using virtual memory in a real-time system has serious implications:

**Problem:** Virtual memory relies on demand paging, which introduces **unpredictable delays** when page faults occur. A page fault requires disk I/O, which can take milliseconds — unacceptable for real-time systems that require deterministic, bounded response times.

**Implications:**
- Real-time processes may miss deadlines if a critical page fault occurs during time-sensitive operations.
- The system cannot guarantee worst-case execution times because page fault latency is variable.
- **Solutions/mitigations:** Pages for real-time processes should be **pinned (locked) in memory** so they are never swapped out. Alternatively, all pages needed by real-time tasks should be pre-loaded. This defeats much of the purpose of virtual memory but is necessary for timing guarantees.

### 1d) File Growth in Different File Systems (6 marks)

#### (i) Contiguous File (2 marks)

The entire file must be relocated. The system must find a new contiguous block of disk space large enough to hold the file plus the new data, copy the entire file to the new location, and update the directory entry with the new starting address and length. If no large enough contiguous space exists, compaction may be needed.

#### (ii) Linked Non-Contiguous File (2 marks)

A new disk block is allocated from anywhere on the disk. The pointer in the last block of the file is updated to point to the new block, and the new block's pointer is set to null (end of file). No existing data needs to be moved — growth is straightforward.

#### (iii) Indexed File (2 marks)

A new disk block is allocated from anywhere on the disk. The file's index block is updated by adding the address of the new block as the next entry. No existing data needs to be moved. If the index block itself is full, a new index block or multi-level index may be needed.

### 2b) Round Robin — Why move to end of queue (4 marks)

The main reason is **fairness and prevention of starvation**. In Round Robin, each process receives an equal share of CPU time (one time quantum at a time). If a process consumed the full time slice, it means it still has work to do but has already used its fair share for this round. Moving it to the end of the ready queue ensures that:

1. **All other processes get a turn** before this process runs again.
2. **No single process can monopolise the CPU** — even long-running processes must yield.
3. **Interactive responsiveness** is maintained — all processes make progress regularly.

If the process is moved to the blocked queue instead, it has performed an I/O request during its time slice and must wait for I/O completion.

### 2c) User-Level Threads (3 marks)

A user-level threads package supports multi-threaded applications entirely in user space without kernel involvement:

- A **thread library** (e.g., POSIX pthreads at user level) manages thread creation, scheduling, and context switching.
- The library maintains its own thread table with thread states, saved registers, and stacks.
- Thread switching is done by the library via saving/restoring registers — no system call or kernel trap is needed, making it very fast.
- The kernel sees only a single process; it is unaware of the multiple threads within it.
- The library uses its own scheduler (e.g., round-robin among threads) to decide which thread runs.

### 2d) Three I/O Techniques (6 marks)

**1. Programmed I/O (Polling)**
The CPU directly controls I/O by repeatedly checking (polling) the device's status register in a tight loop until the device is ready. The CPU transfers each unit of data. Simple but wastes CPU cycles as the processor is busy-waiting and cannot do other work.

**2. Interrupt-Driven I/O**
The CPU initiates an I/O operation and continues executing other instructions. When the device completes the transfer, it sends an interrupt signal to the CPU. The CPU then handles the interrupt via an ISR (Interrupt Service Routine) and processes the data. More efficient than polling as CPU can do useful work while waiting.

**3. Direct Memory Access (DMA)**
A DMA controller handles data transfer between the I/O device and main memory independently of the CPU. The CPU only initiates the transfer (provides source, destination, byte count), then the DMA controller takes over. The CPU is interrupted only once when the entire transfer is complete. Most efficient for large data transfers.

## Question 3: Critical Section, Deadlock and Process Synchronisation (25 Marks)

### 3a) Critical Section Problem (5 marks)

The **critical section problem** is the challenge of designing a protocol that allows multiple concurrent processes/threads to access shared resources without creating race conditions. Each process has a critical section where it accesses shared data, and the solution must ensure that no two processes are in their critical sections simultaneously.

**Three requirements:**

1. **Mutual Exclusion:** If a process is executing in its critical section, no other process may be executing in its critical section. Only one process at a time can access the shared resource.

2. **Progress:** If no process is in its critical section and some processes wish to enter, only those processes not in their remainder section can participate in the decision of which enters next, and this decision cannot be postponed indefinitely.

3. **Bounded Waiting:** There must be a limit on the number of times other processes are allowed to enter their critical sections after a process has made a request to enter and before that request is granted. No process should wait forever (no starvation).

---

### 3b) Indefinite Postponement vs Deadlock (4 marks)

**Indefinite postponement (starvation):** A process is ready to run and repeatedly passed over by the scheduler or resource allocator, never getting the resource/CPU time it needs. The process could theoretically proceed — resources become available — but it is never selected. Other processes continue to run.

**Deadlock:** Two or more processes are permanently blocked, each waiting for a resource held by another process in the set. No process can proceed because the dependencies form a cycle. No process in the deadlock set is running.

**Key difference:** In deadlock, NO process in the set can ever proceed (permanent blockage). In indefinite postponement, the starved process COULD proceed if selected — it is just never chosen. The system is still making progress (other processes are running), but one process is unfairly denied service.

---

### 3c) Consequences of Indefinite Postponement (6 marks)

#### (i) Batch Processing

In batch systems, indefinite postponement means a job stays in the queue indefinitely while other jobs are selected. Since batch jobs don't have interactive users waiting, the consequence is less severe — the job simply remains queued. However, jobs may have deadlines (e.g., month-end reports), so indefinite delay can still cause business problems. Aging-based priority can solve this.

#### (ii) Timesharing

In timesharing systems, indefinite postponement is more serious. Users expect interactive response times. If a user's process is starved, they experience an unresponsive system — commands get no response, applications freeze. This degrades the user experience severely and defeats the purpose of timesharing (fair sharing among multiple users).

#### (iii) Real-Time

In real-time systems, indefinite postponement can be **catastrophic**. Real-time processes have hard deadlines — missing them can cause system failure (e.g., missing a sensor reading in an aircraft control system). If a critical process is indefinitely postponed, it misses its deadline, which could lead to physical damage, safety hazards, or system failure.

---

### 3d) Banker's Algorithm Weaknesses (4 marks)

1. **Processes must declare maximum resource needs in advance** — In practice, processes rarely know their maximum resource requirements before execution. This makes the algorithm impractical for general-purpose systems.

2. **Fixed number of processes and resources** — The algorithm assumes a fixed number of processes and resource types. In real systems, processes arrive and leave dynamically, and resources can fail.

3. **Conservative/inefficient** — The algorithm may refuse requests even when deadlock won't actually occur (it only guarantees a safe state exists, not that deadlock will happen). This reduces resource utilisation.

4. **Computational overhead** — Running the safety algorithm on every resource request is expensive (O(n²m) where n=processes, m=resources), which adds latency to every allocation.

### 3e) Semaphore Deadlock and Postponement (6 marks)

#### (i) Can deadlock occur? (3 marks)

**Yes.** Consider this execution sequence:
1. P1 executes `semWait(S)` → S becomes 0. P1 holds S.
2. P2 executes `semWait(R)` → R becomes 0. P2 holds R.
3. P1 executes `semWait(R)` → R is 0, P1 blocks (waiting for R held by P2).
4. P2 executes `semWait(S)` → S is 0, P2 blocks (waiting for S held by P1).

**Both processes are deadlocked.** P1 holds S and waits for R; P2 holds R and waits for S. This is a classic circular wait.

#### (ii) Can indefinite postponement occur? (3 marks)

**No** (assuming fair semaphore scheduling, e.g., FIFO queues). If deadlock does not occur, both processes will eventually complete their operations and release both semaphores. With fair scheduling, neither process can be indefinitely postponed because:
- If one process holds both semaphores and completes, it signals both, allowing the other to proceed.
- There are only two processes competing, so if one finishes its critical section, the other will be next.

However, if the semaphore implementation does not guarantee FIFO ordering, one process could theoretically be starved, but this is an implementation issue rather than an inherent property of the code.

## Question 4: Process Synchronisation, Mutual Exclusion and Semaphore (25 Marks)

### 4a) Busy Waiting (5 marks)

**Busy waiting** (spin-waiting) is when a process repeatedly checks a condition in a tight loop while waiting for it to become true, consuming CPU cycles without doing useful work.

```
while (condition == false) { } // spin — wastes CPU
```

**Is it always wasteful?** No. Busy waiting is appropriate and efficient when:
- The wait is expected to be **very short** (e.g., waiting for a spinlock on a multiprocessor where the lock holder is on another core and will release within a few cycles).
- The overhead of context switching (sleeping/waking) would exceed the expected wait time.

**Advantage:** No overhead of context switching — if the lock is released quickly, the process resumes immediately without scheduler involvement. Useful on multiprocessor systems for short critical sections.

**Disadvantage:** Wastes CPU cycles that could be used by other processes. On a single-processor system, it is always wasteful because the process holding the lock cannot run while the spinner occupies the CPU.

---

### 4b) Test and Set Lock (TSL) (3 marks)

**TSL** is a hardware instruction that atomically reads a memory location and sets it to a non-zero value in a single, indivisible operation.

```
TSL REGISTER, LOCK
// Copies LOCK to REGISTER and sets LOCK = 1 atomically
```

**Importance for mutual exclusion:**
- It is **atomic** — cannot be interrupted between the read and write, preventing race conditions.
- It provides a hardware-level primitive that can be used to build correct mutual exclusion without disabling interrupts.
- Works correctly on **multiprocessor systems** where disabling interrupts only affects one CPU.

A process enters the critical section only if TSL reads 0 (lock was free); all other processes see 1 and spin-wait.

---

### 4c) Java Threads (4 marks)

**Two ways to implement Java threads:**

1. **Extending the Thread class:** Create a subclass of `Thread` and override the `run()` method.
   ```java
   class MyThread extends Thread {
       public void run() { /* thread code */ }
   }
   ```

2. **Implementing the Runnable interface:** Create a class that implements `Runnable`, then pass an instance to a `Thread` constructor.
   ```java
   class MyRunnable implements Runnable {
       public void run() { /* thread code */ }
   }
   new Thread(new MyRunnable()).start();
   ```

**Advantage of Runnable:** Since Java doesn't support multiple inheritance, implementing Runnable allows the class to extend another class. This is generally preferred.

**Why start() instead of run():** Calling `start()` creates a **new OS thread** and invokes `run()` on that new thread. Calling `run()` directly simply executes the method on the **current thread** — no new thread is created, so there is no concurrent execution.

### 4d) Producer/Consumer with Semaphores (6 marks)

**Problem:** Producers generate items and place them in a bounded buffer; consumers remove items. Synchronisation is needed to ensure: producers don't write to a full buffer, consumers don't read from an empty buffer, and mutual exclusion on buffer access.

```java
semaphore mutex = 1;     // mutual exclusion for buffer access
semaphore empty = N;     // counts empty slots (initially N = buffer size)
semaphore full = 0;      // counts full slots (initially 0)

void producer() {
    while (true) {
        item = produce_item();
        semWait(empty);      // wait for empty slot
        semWait(mutex);      // enter critical section
        insert(item);        // add item to buffer
        semSignal(mutex);    // leave critical section
        semSignal(full);     // signal item available
    }
}

void consumer() {
    while (true) {
        semWait(full);       // wait for item
        semWait(mutex);      // enter critical section
        item = remove();     // remove item from buffer
        semSignal(mutex);    // leave critical section
        semSignal(empty);    // signal empty slot
        consume(item);
    }
}
```

This works for multiple producers and multiple consumers because:
- `mutex` ensures only one thread modifies the buffer at a time.
- `empty` and `full` are counting semaphores that track available slots/items.

### 4e) Two-Thread Mutual Exclusion Analysis (7 marks)

This code is a variant of **Dekker's algorithm**. Let's analyse whether it enforces mutual exclusion:

**Mutual Exclusion — YES:** The code does enforce mutual exclusion. Only the thread whose turn it is (turn == 1 for T1, turn == 2 for T2) can proceed past the outer while loop into the critical section. Since `turn` can only be 1 or 2 at any given time, only one thread can be in the critical section.

**Progress — YES:** If one thread wants to enter and the other does not want to enter (its `WantsToEnter` flag is false), the waiting thread's inner while loop (`while(t2WantsToEnter)` or `while(t1WantsToEnter)`) will exit, and it will set `turn` to its own value and proceed.

**Bounded Waiting — YES:** The `WantsToEnter` flags combined with the turn variable ensure that if both threads want to enter, the one whose turn it is proceeds. After executing the critical section, a thread sets its `WantsToEnter` to false, allowing the other thread's inner loop to terminate.

**Analysis:** This is essentially Dekker's algorithm and correctly solves the critical section problem for two threads. It satisfies all three requirements: mutual exclusion (via turn variable), progress (via the WantsToEnter flags allowing deference), and bounded waiting (at most one pass before entry).

However, it does use **busy waiting** (spin-locks), which wastes CPU on uniprocessor systems.
