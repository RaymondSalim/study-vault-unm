## Question 1: OS Structure, Memory Management and File Management (25 Marks)

### 1a) Microkernel Approach (6 marks)

**Structure:** A microkernel design moves as much functionality as possible out of the kernel and into user-space server processes. The kernel itself contains only the absolute minimum: inter-process communication (IPC), basic scheduling, and low-level address space management. All other OS services — file systems, device drivers, networking, memory management — run as separate user-space processes that communicate via message passing through the microkernel.

**Main advantages:**
1. **Reliability/Fault isolation:** If a service (e.g., file system driver) crashes, it doesn't bring down the entire OS — only that service fails and can be restarted. The kernel remains stable.
2. **Security:** Services run in user mode with restricted privileges. A compromised driver cannot corrupt kernel memory.
3. **Extensibility/Flexibility:** New services can be added or existing ones replaced without modifying or recompiling the kernel. Different services can even run different implementations.
4. **Portability:** The small kernel is easier to port to new hardware architectures — only the microkernel needs hardware-specific code.

---

### 1b) Linked List File System (4 marks)

**File Allocation Table (FAT):** Each file is stored as a linked list of disk blocks. A File Allocation Table in memory contains one entry per disk block. Each entry holds the index of the next block in the file chain, or a special end-of-file marker. The directory entry stores the first block number, and the FAT is followed to find subsequent blocks.

**Advantages:**
1. **No external fragmentation** — blocks can be scattered anywhere on disk; any free block can be used.
2. **Files can grow dynamically** — just allocate a new block and link it; no need to relocate the file.

**Disadvantage:**
- **Poor random access performance** — to read block n of a file, you must traverse the FAT chain from the beginning through n-1 entries. This is O(n) rather than O(1).

---

### 1c) Symbolic Linking (2 marks)

A **symbolic link** (soft link) is a special file that contains a path reference to another file or directory. Its purpose is to allow multiple directory entries to refer to the same file without duplicating the data, and across file system boundaries. Unlike hard links, symbolic links can cross partition boundaries and can point to directories. If the original file is deleted, the symbolic link becomes a "dangling" reference.

---

### 1d) Internal Fragmentation (3 marks)

Each process is allocated a fixed chunk of 2000 bytes. Internal fragmentation is the unused space within each chunk.

| Process | Size | Chunk | Wasted |
|---------|------|-------|--------|
| P1 | 1000 | 2000 | 1000 |
| P2 | 2000 | 2000 | 0 |
| P3 | 300 | 2000 | 1700 |
| P4 | 10 | 2000 | 1990 |

**Total internal fragmentation = 1000 + 0 + 1700 + 1990 = 4690 bytes**

### 1e) LRU in Hardware (6 marks)

**Hardware implementation of LRU:**

One approach uses a **counter (timestamp) register** in the CPU:
- The CPU has a logical clock/counter that increments on every memory reference.
- Each page table entry has a "time-of-use" field.
- When a page is referenced, the current counter value is copied to its page table entry.
- When a page must be replaced, the page with the smallest counter value (oldest reference) is chosen.

Another approach uses an **n×n matrix** (for n page frames):
- When page frame k is referenced, set all bits in row k to 1, then set all bits in column k to 0.
- The row with the lowest binary value represents the LRU page.

**Is LRU always the best?**

No. LRU is not always optimal:
- It cannot predict future access patterns. The **optimal algorithm (OPT/Bélády's)** replaces the page that won't be used for the longest time in the future — but this requires knowledge of future references.
- LRU performs poorly with certain access patterns, e.g., sequential scans through large arrays larger than memory (all pages are used once and never again — LRU keeps recently seen pages that won't be reused).
- LRU suffers from **Bélády's anomaly** in that more frames don't always reduce page faults for certain patterns. (Note: actually LRU is a stack algorithm and doesn't exhibit Bélády's anomaly, but it can still perform poorly compared to OPT.)

---

### 1f) Paging and Memory Protection (4 marks)

**Why a process cannot access memory it doesn't own:**
Each process has its own **page table** that maps virtual addresses to physical frames. The page table only contains entries for pages that belong to that process. When a process generates a virtual address, the MMU translates it using that process's page table. If the virtual page number isn't in the table (or is marked invalid), a page fault/segmentation fault occurs. A process simply has no way to generate a physical address belonging to another process — the hardware enforces isolation.

**How the OS allows access to other memory:**
- **Shared memory:** The OS can map the same physical frame into the page tables of multiple processes. Both processes have a virtual page pointing to the same physical frame, allowing them to read/write shared data.
- **System calls:** A process can request the OS to copy data between address spaces (e.g., IPC mechanisms like pipes or message passing).

## Question 2: Process Scheduling (25 Marks)

### 2a) Scheduling Algorithms (6 marks)

#### (i) Highest Response Ratio Next (HRRN) (2 marks)

HRRN is a non-preemptive scheduling algorithm that selects the process with the highest **response ratio**: `R = (W + S) / S`, where W = waiting time and S = service (burst) time. This favours short jobs (like SJF) but also considers how long a process has been waiting, preventing starvation of long jobs.

Example: Process A (burst=4, waited=8) has R=(8+4)/4=3.0. Process B (burst=2, waited=2) has R=(2+2)/2=2.0. HRRN selects A.

#### (ii) Shortest Job First (SJF) (4 marks)

SJF selects the process with the **shortest burst/service time** next. It can be non-preemptive (once started, a process runs to completion) or preemptive (SRTF — a new shorter process preempts the current one).

Example: Processes with bursts P1=6, P2=2, P3=8, P4=1. SJF order: P4(1), P2(2), P1(6), P3(8).
- Average waiting time = (0+1+3+9)/4 = 3.25ms
- SJF is provably optimal for minimising average waiting time among non-preemptive algorithms.
- Disadvantage: Can cause **starvation** of long processes; requires knowing burst times in advance (estimated via exponential averaging in practice).

---

### 2b) Round Robin Favours CPU-Bound (2 marks)

Round Robin gives each process a fixed time quantum. **CPU-bound** processes use the full quantum each time and are simply placed back in the ready queue. **I/O-bound** processes typically use only a fraction of their quantum before blocking on I/O, then must wait in the blocked queue until I/O completes, then rejoin the ready queue. Result: CPU-bound processes get more total CPU time per unit of real time because they are always ready, while I/O-bound processes spend much of their time blocked.

---

### 2c) Round Robin Degrades to FCFS (2 marks)

Round Robin degrades to FCFS when the **time quantum is set to a value larger than the longest burst time** of any process. In this case, every process completes within its first quantum — no preemption ever occurs — and processes run in arrival order until completion, exactly like FCFS.

---

### 2d) Multilevel Feedback Queue (4 marks)

The Multilevel Feedback Queue (MLFQ) uses **multiple queues with different priority levels** and allows processes to **move between queues** based on their behaviour.

**Structure:**
- Queue 0 (highest priority): time quantum = 8ms, uses Round Robin
- Queue 1: time quantum = 16ms, uses Round Robin
- Queue 2 (lowest priority): uses FCFS

**Rules:**
1. New processes enter the highest priority queue.
2. If a process uses its full time quantum, it is demoted to the next lower queue.
3. If a process blocks before using its quantum (I/O-bound), it stays at the same level (or is promoted).
4. Higher priority queues are serviced first.

**Example:** A CPU-bound process enters Queue 0 with 8ms quantum. It uses all 8ms → demoted to Queue 1 (16ms). Uses all 16ms → demoted to Queue 2 (FCFS). An I/O-bound process enters Queue 0, uses 3ms then blocks on I/O — stays in Queue 0 (high priority).

---

### 2e) Different Time Quantum Sizes at Different Levels (3 marks)

**Advantage:** Processes that have been demoted to lower priority queues are known to be CPU-bound (long-running). Giving them **larger time quanta** reduces context-switching overhead for these processes — they get more CPU time per switch, amortising the cost of the context switch. Meanwhile, the higher-priority queues with smaller quanta ensure that interactive/I/O-bound processes get fast, responsive service. This balances responsiveness (short quanta at top) with efficiency (longer quanta at bottom).

---

### 2f) Preventing Starvation in Multi-Level Queue (2 marks)

**Aging:** Gradually increase the priority of processes that have been waiting for a long time. If a process in a lower-priority queue has waited beyond a threshold, promote it to a higher-priority queue. This ensures that even long CPU-bound processes will eventually reach the highest queue and get served, preventing indefinite starvation.

---

### 2g) Scheduling Policy Choices (2 marks)

**(i) Time-sharing system:** **Round Robin** (or Multilevel Feedback Queue) — provides fair sharing of CPU among interactive users and guarantees bounded response time.

**(ii) Batch system:** **Shortest Job First** (or FCFS) — maximises throughput and minimises average turnaround/waiting time. Interactive response time is not a concern.

---

### 2h) Real-Time Systems (4 marks)

A **real-time system** is a system where the correctness of the computation depends not only on producing the correct result but on producing it within a specified **deadline**. Tasks have timing constraints that must be met.

**Impact on OS design:**
- **Scheduling:** Must use priority-based preemptive scheduling (e.g., Rate Monotonic, Earliest Deadline First) that guarantees deadlines are met. Time-sharing fairness is sacrificed for predictability.
- **Memory management:** Virtual memory/paging is problematic due to unpredictable page fault delays. Real-time OSes often lock pages in memory or avoid paging entirely.
- **Interrupt handling:** Must be fast and predictable with bounded latency.
- **Minimal kernel overhead:** The OS must have deterministic, bounded execution times for all system calls — no unbounded algorithms.

## Question 3: Deadlock, Device Management and Process Synchronisation (25 Marks)

### 3a) Banker's Algorithm Weaknesses (4 marks)

1. **Processes must declare maximum resource needs in advance** — rarely possible in practice.
2. **Fixed number of processes assumed** — in real systems, processes dynamically enter and exit.
3. **Fixed number of resources assumed** — resources can fail, be added, or be removed at runtime.
4. **Processes must be independent** — the algorithm doesn't account for synchronisation dependencies between processes that may affect execution order.

---

### 3b) Deadlock Definition (2 marks)

**Deadlock** is a situation where two or more processes are permanently blocked, each waiting for a resource currently held by another process in the set. None of the processes can proceed because they form a circular chain of dependencies. The four necessary conditions are: mutual exclusion, hold and wait, no preemption, and circular wait.

---

### 3c) Unsafe State ≠ Deadlock (2 marks)

An unsafe state means there is **no guaranteed safe sequence** to complete all processes — but deadlock is not certain. Processes might happen to release resources in a favourable order, or might not actually request their maximum resources. An unsafe state simply means the system **cannot guarantee** avoidance of deadlock; it is a necessary but not sufficient condition for deadlock. Deadlock occurs only if processes actually make the worst-case requests.

---

### 3d) DMA (6 marks)

**How DMA works:**
1. The CPU initiates a transfer by programming the DMA controller with: the source address (e.g., disk), the destination address (memory), the byte count, and the direction (read/write).
2. The DMA controller takes control of the system bus and transfers data directly between the I/O device and main memory, one block at a time.
3. The CPU is free to execute other instructions during the transfer.
4. The DMA controller uses **cycle stealing** — it briefly takes the bus from the CPU for each word transferred, causing slight CPU slowdown.
5. When the entire transfer is complete, the DMA controller sends an **interrupt** to the CPU.
6. The CPU handles the interrupt and processes the completed transfer.

**Usefulness:** DMA frees the CPU from managing data transfers byte-by-byte. For high-speed devices (disk, network), the CPU would otherwise spend most of its time copying data rather than computing. DMA allows overlap of I/O and computation, dramatically improving system throughput.

---

### 3e) Banker's Algorithm Application (7 marks)

Given:

| Process | Allocation (R1-R4) | Max Demand (R1-R4) | Available: 2,1,0,0 |
|---------|-------------------|--------------------|---------------------|
| P1 | 0,0,1,2 | 0,0,1,2 | |
| P2 | 2,0,0,0 | 2,7,5,0 | |
| P3 | 0,0,3,4 | 6,6,5,6 | |
| P4 | 2,3,5,4 | 4,3,5,6 | |
| P5 | 0,3,3,2 | 0,6,5,2 | |

#### (i) Need matrix (Max - Allocation) (2 marks)

| Process | Need (R1,R2,R3,R4) |
|---------|-------------------|
| P1 | 0,0,0,0 |
| P2 | 0,7,5,0 |
| P3 | 6,6,2,2 |
| P4 | 2,0,0,2 |
| P5 | 0,3,2,0 |

#### (ii) Is the system in a safe state? (2 marks)

Available = (2,1,0,0)

- **P1:** Need(0,0,0,0) ≤ Available(2,1,0,0) ✓ → Execute P1, Available becomes (2,1,0,0) + (0,0,1,2) = (2,1,1,2)
- **P4:** Need(2,0,0,2) ≤ Available(2,1,1,2) ✓ → Execute P4, Available becomes (2,1,1,2) + (2,3,5,4) = (4,4,6,6)
- **P5:** Need(0,3,2,0) ≤ Available(4,4,6,6) ✓ → Execute P5, Available becomes (4,4,6,6) + (0,3,3,2) = (4,7,9,8)
- **P2:** Need(0,7,5,0) ≤ Available(4,7,9,8) ✓ → Execute P2, Available becomes (4,7,9,8) + (2,0,0,0) = (6,7,9,8)
- **P3:** Need(6,6,2,2) ≤ Available(6,7,9,8) ✓ → Execute P3

**Safe sequence: P1, P4, P5, P2, P3. Yes, the system is in a safe state.**

#### (iii) Can P3's request (0,1,0,0) be granted? (3 marks)

Check: Request(0,1,0,0) ≤ Available(2,1,0,0)? → (0≤2, 1≤1, 0≤0, 0≤0) ✓

Pretend to grant:
- New Allocation for P3: (0,1,3,4)
- New Available: (2,0,0,0)
- New Need for P3: (6,5,2,2)

Run safety check with Available = (2,0,0,0):
- P1: Need(0,0,0,0) ≤ (2,0,0,0) ✓ → Available = (2,0,1,2)
- P4: Need(2,0,0,2) ≤ (2,0,1,2) ✓ → Available = (4,3,6,6)
- P5: Need(0,3,2,0) ≤ (4,3,6,6) ✓ → Available = (4,6,9,8)
- P2: Need(0,7,5,0) ≤ (4,6,9,8)? → (0≤4, 7≤6?) ✗

Try P3: Need(6,5,2,2) ≤ (4,6,9,8)? → (6≤4?) ✗

No safe sequence found after P5. Let's retry: after P5, Available = (4,6,9,8):
- P2: Need(0,7,5,0) ≤ (4,6,9,8)? → 7≤6? ✗
- P3: Need(6,5,2,2) ≤ (4,6,9,8)? → 6≤4? ✗

**No safe sequence exists. The request cannot be granted immediately.**

### 3f) Safeguards Against Indefinite Waiting (4 marks)

1. **Timeouts:** Set maximum wait times for blocked processes. If a process waits beyond the timeout, the OS takes corrective action (e.g., abort the request, signal an error, or preempt the resource).

2. **Aging:** Gradually increase the priority of waiting processes over time. Eventually, any waiting process reaches maximum priority and gets served.

3. **Deadlock detection and recovery:** Periodically run a deadlock detection algorithm. If deadlock is found, terminate or roll back one or more processes to break the cycle.

4. **Resource ordering:** Impose a global ordering on resource acquisition to prevent circular wait, eliminating one condition for deadlock.

## Question 4: Process Synchronisation, Mutual Exclusion and Semaphore (25 Marks)

### 4a) Problems with Disabling Interrupts (3 marks)

1. **CPU cannot respond to external events** — hardware interrupts (disk completion, timer ticks, network packets) are missed or delayed, potentially causing data loss or device timeouts.

2. **System clock stops ticking** — the timer interrupt is suppressed, so the OS loses track of time, affecting scheduling and time-dependent operations.

3. **Only works on single-processor systems** — on multiprocessor systems, disabling interrupts on one CPU doesn't prevent other CPUs from accessing shared data, so mutual exclusion is not achieved.

---

### 4b) Difficulty of Concurrent Program Testing (5 marks)

1. **Non-determinism:** Concurrent programs have many possible interleavings of instruction execution. The same program may produce different results on different runs depending on timing, making bugs intermittent and hard to reproduce.

2. **State space explosion:** The number of possible execution orderings grows exponentially with the number of threads and instructions. Testing all interleavings is infeasible.

3. **Timing-dependent bugs (Heisenbugs):** Adding debugging instrumentation (print statements, debugger breakpoints) changes timing and may make the bug disappear, making diagnosis extremely difficult.

4. **Race conditions are subtle:** A race condition may only trigger under specific interleaving that is extremely rare, perhaps occurring once in millions of executions.

5. **Difficulty of formal proof:** Proving correctness requires reasoning about all possible interleavings — tools like model checkers help but face state explosion for realistic programs.

---

### 4c) Busy Waiting (4 marks)

**Busy waiting** (spinning) is when a process continuously tests a condition in a loop, consuming CPU cycles while waiting for the condition to change.

**Not always wasteful:** On multiprocessor systems, if the lock is expected to be released very quickly (e.g., another CPU is in a short critical section), spinning avoids the overhead of context-switching (saving state, switching to another process, then switching back). Spinlocks are efficient when the expected wait time is less than the context-switch overhead.

**Wasteful when:** On single-processor systems or when the critical section is long — the spinning process prevents other processes from running (including the one holding the lock), wasting CPU time entirely.

---

### 4d) Synchronisation Without Busy Waiting — Semaphores (5 marks)

**Semaphore** is a synchronisation tool that doesn't require busy waiting (when implemented with blocking):

**Structure:** A semaphore S consists of an integer value and a queue of blocked processes.

**Operations:**
```
semWait(S):
    S.value--;
    if (S.value < 0) {
        add this process to S.queue;
        block();  // process is put to sleep — no busy waiting
    }

semSignal(S):
    S.value++;
    if (S.value <= 0) {
        remove a process P from S.queue;
        wakeup(P);  // move P to ready queue
    }
```

**How it avoids busy waiting:** When a process cannot enter the critical section (semaphore value < 0), it is placed in a waiting queue and moved to the **blocked state**. The CPU can execute other processes. When the semaphore is signalled, a blocked process is woken up and moved to the ready queue. No CPU cycles are wasted spinning.

---

### 4e) Two-Phase Locking Not Generally Applicable (3 marks)

Two-phase locking (acquire all locks in phase 1, release all in phase 2) is not generally applicable because:

1. **Processes don't always know all needed resources in advance** — in general computing, a process discovers what resources it needs as it executes. Databases can determine this from query plans, but general programs cannot.

2. **Resource utilisation is poor** — holding all resources for the entire duration even when some are only needed briefly reduces concurrency and system throughput.

3. **May cause unnecessary deadlock/starvation** — if a process cannot acquire all resources at once, it must release them and retry, potentially leading to livelock.

---

### 4f) Strict Alternation Analysis (5 marks)

```c
int threadNumber = 0;
// ThreadZero enters when threadNumber == 0
// ThreadOne enters when threadNumber == 1
```

**Mutual Exclusion — YES:** At any instant, `threadNumber` is either 0 or 1. Only one thread can pass its while loop. So only one thread can be in the critical section at a time.

**Progress — NO:** This algorithm enforces **strict alternation**. If ThreadZero finishes its critical section and sets threadNumber=1, but ThreadOne is in `OtherStuffOne` (not wanting to enter), ThreadZero is blocked even though no one is in the critical section. A process outside its critical section is blocking another from entering — violating the progress requirement.

**Bounded Waiting — YES (given alternation):** Since the threads strictly alternate, each thread waits at most one turn.

**Conclusion:** The code **does NOT fully satisfy** mutual exclusion requirements because it fails the **progress condition**. It forces strict turn-taking even when only one thread wants the critical section. A slower thread holds back a faster thread unnecessarily.
