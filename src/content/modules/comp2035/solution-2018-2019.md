## Question 1: Memory Management (25 Marks)

### 1a) Address Translation in a Paged Memory Allocation Scheme (7 marks)

**Paging** divides the logical address space into fixed-size units called **pages**, and physical memory into same-sized units called **frames**. The page table maps each page number to a frame number.

**Address Translation Process:**

```
Logical Address: [ Page Number (p) | Offset (d) ]
                         |
                         v
                  +-------------+
                  | Page Table  |
                  |-------------|
                  | p=0 -> f=5  |
                  | p=1 -> f=2  |
                  | p=2 -> f=8  |
                  | p=3 -> f=1  |
                  +-------------+
                         |
                         v
Physical Address: [ Frame Number (f) | Offset (d) ]
```

**Steps:**

1. The CPU generates a logical (virtual) address, which is divided into two parts: the **page number (p)** (high-order bits) and the **page offset (d)** (low-order bits).
2. The page number is used as an index into the **page table**, which is maintained per-process by the OS.
3. The page table entry at index p contains the corresponding **frame number (f)** in physical memory.
4. The physical address is formed by concatenating the frame number with the offset: `physical address = f * page_size + d`.
5. The **offset remains unchanged** because page size equals frame size.

**Hardware support:** The Translation Lookaside Buffer (TLB) caches recent page-to-frame mappings to speed up address translation. On a TLB miss, the page table in memory is consulted.

**Key features:**
- Eliminates external fragmentation (any free frame can hold any page)
- Provides memory protection via valid/invalid bits in page table entries
- Enables non-contiguous memory allocation for processes

### 1c) Thrashing: Definition, Detection, and Elimination (5 marks)

**What is thrashing?**

Thrashing occurs when a process (or the system) spends more time paging (swapping pages in and out of memory) than executing useful instructions. It happens when the working set of active pages for the running processes exceeds the available physical memory. The system enters a vicious cycle: high page fault rate leads to more I/O, which makes the CPU appear idle, causing the OS to increase the degree of multiprogramming, which further increases page faults.

**How to detect thrashing:**

1. **Monitor page fault rate:** If the page fault rate exceeds an acceptable upper threshold, the system may be thrashing.
2. **Monitor CPU utilisation vs. degree of multiprogramming:** If CPU utilisation drops sharply as multiprogramming increases, thrashing is occurring.
3. **Track working set sizes:** If the sum of all processes' working sets exceeds total physical memory, thrashing is imminent.

**How to eliminate thrashing:**

1. **Working Set Model:** Track each process's working set (the set of pages referenced in the recent time window). Only allow processes to run if their working sets fit in memory. Suspend excess processes.
2. **Page Fault Frequency (PFF):** Set upper and lower bounds on acceptable page fault rates. If a process exceeds the upper bound, allocate more frames. If below the lower bound, reduce its frame allocation. If no frames are available, suspend a process.
3. **Reduce the degree of multiprogramming:** Swap out (suspend) one or more processes to free frames for the remaining active processes.
4. **Increase physical memory** (hardware solution).

### 1f) Second-Chance (Clock) Page Replacement Policy (4 marks)

The **second-chance** algorithm is a modification of FIFO that avoids replacing a frequently-used page.

**How it works:**

1. Pages are maintained in a circular queue (FIFO order), with a pointer (the "clock hand") indicating the next candidate for replacement.
2. Each page has a **reference bit**, set to 1 by hardware whenever the page is accessed.
3. When a page replacement is needed:
   - Examine the page at the clock hand position.
   - If the reference bit is **1**: give it a "second chance" -- clear the bit to 0, advance the pointer to the next page, and repeat.
   - If the reference bit is **0**: this page has not been accessed recently -- select it for replacement.
4. The pointer advances circularly, so in the worst case (all bits are 1), it goes around the entire buffer, clearing all bits, and replaces the page where it started (degenerating to FIFO).

**Advantage over FIFO:** Pages that are being actively used (reference bit = 1) get a reprieve, approximating LRU without the overhead of maintaining full usage history.

**Relationship to Clock algorithm:** Second-chance is often implemented as the "clock algorithm" with a circular buffer and a single pointer, which is more efficient than the queue-based description.

### 2b) Multilevel Feedback Queue (4 marks)

A **Multilevel Feedback Queue (MLFQ)** is a scheduling algorithm that uses multiple ready queues at different priority levels, and allows processes to move between queues based on their observed behaviour.

**Key characteristics:**
1. Multiple queues, each with a different priority level and scheduling algorithm.
2. Processes can be **promoted** or **demoted** between queues based on CPU usage patterns.
3. Higher-priority queues have shorter time quanta; lower-priority queues have longer quanta (or use FCFS).
4. New processes enter the highest-priority queue.

**Example configuration:**
- **Queue 0 (highest priority):** Round Robin, quantum = 8ms
- **Queue 1 (medium priority):** Round Robin, quantum = 16ms
- **Queue 2 (lowest priority):** FCFS

**Rules:**
- A new job enters Queue 0. If it uses its full 8ms quantum without completing or blocking, it is demoted to Queue 1.
- In Queue 1, if it again uses the full 16ms quantum, it is demoted to Queue 2.
- If a process blocks for I/O before its quantum expires, it stays at the current level (or is promoted), since it exhibits interactive behaviour.
- Periodically, all processes may be boosted to the top queue to prevent starvation (priority boost).

**Example:** A text editor (I/O-bound) enters Queue 0, runs for 2ms then blocks for keyboard input -- stays in Queue 0 and gets fast response. A video encoder (CPU-bound) enters Queue 0, uses all 8ms, gets demoted to Queue 1, uses all 16ms, gets demoted to Queue 2 where it runs in FCFS with long uninterrupted bursts.

### 2c) User-Level vs Kernel-Level Threads (4 marks)

**Key differences:**

| Aspect | User-Level Threads | Kernel-Level Threads |
|--------|-------------------|---------------------|
| Management | Managed by thread library in user space | Managed by the OS kernel |
| Kernel awareness | Kernel sees only a single process | Kernel is aware of each individual thread |
| Context switch | Fast (no kernel mode switch needed) | Slower (requires kernel mode transition) |
| Scheduling | Done by user-level scheduler | Done by OS scheduler |
| Blocking | If one thread blocks (system call), entire process blocks | Only the blocking thread is suspended; others continue |
| Multiprocessor use | Cannot utilise multiple CPUs (kernel schedules process, not threads) | Can run on different CPUs simultaneously |

**Two advantages of user-level threads:**

1. **Fast context switching:** Switching between user-level threads does not require a trap to the kernel. The thread library saves/restores registers entirely in user space, which is significantly faster (often 10-100x) than a kernel-mode context switch.

2. **Portability and customisability:** User-level threads can be implemented on any OS, even those without native thread support. The scheduling algorithm can be customised per-application (e.g., priority-based for one application, round-robin for another) without modifying the OS.

### 2d) Three Multithreading Models (9 marks)

#### (i) Many-to-One Model

**Description:** Many user-level threads are mapped to a single kernel thread. Thread management is done in user space by the thread library, and the kernel sees only one schedulable entity.

**Characteristics:**
- Thread creation and switching is fast (no kernel involvement)
- If one thread makes a blocking system call, all threads in the process block
- Cannot exploit multiprocessor parallelism (only one kernel thread = one CPU at most)

**Example:** Green threads (early Java), GNU Portable Threads

**Advantage:** Very efficient thread management (lightweight)
**Disadvantage:** No true parallelism; one blocking call blocks all threads

#### (ii) One-to-One Model

**Description:** Each user-level thread maps to exactly one kernel thread. Creating a user thread also creates a corresponding kernel thread.

**Characteristics:**
- The kernel is aware of all threads and can schedule them independently
- If one thread blocks, others can continue
- True parallelism on multiprocessors (different threads on different CPUs)
- Thread creation overhead is higher (kernel involvement)
- Number of threads may be limited by kernel resources

**Example:** Linux (NPTL), Windows threads

**Advantage:** Full concurrency and multiprocessor utilisation; no blocking of other threads
**Disadvantage:** Higher overhead per thread; kernel resource limits on number of threads

#### (iii) Many-to-Many Model

**Description:** Many user-level threads are multiplexed onto a smaller or equal number of kernel threads. The thread library and the kernel cooperate to schedule user threads onto available kernel threads.

**Characteristics:**
- Developers can create as many user threads as needed
- Kernel threads can run in parallel on multiprocessors
- When a thread blocks, the kernel can schedule another user thread onto the available kernel thread
- Combines the advantages of both Many-to-One and One-to-One
- More complex to implement

**Example:** Solaris prior to version 9, Windows ThreadFiber package, IRIX

**Advantage:** True concurrency without limiting thread creation; flexible resource usage
**Disadvantage:** Complex implementation; scheduler coordination between user and kernel levels

## Question 3: Deadlock, Device Management, File Management (25 Marks)

### 3a) Definition of Deadlock (2 marks)

**Deadlock** is a state in which a set of two or more processes are permanently blocked because each process is holding at least one resource and waiting to acquire a resource that is held by another process in the set. No process can proceed, release its held resources, or be unblocked without external intervention.

The four necessary conditions for deadlock (Coffman conditions):
1. **Mutual exclusion** -- at least one resource must be non-shareable
2. **Hold and wait** -- processes hold resources while waiting for others
3. **No preemption** -- resources cannot be forcibly taken from a process
4. **Circular wait** -- a circular chain of processes exists where each waits for a resource held by the next

### 3b) Resource Allocation Graph and Deadlock Analysis (6 marks)

Given:
- Resources: R1 (available=1), R2 (available=0), R3 (available=1)
- p1: requests R1, R3
- p2: produces R1, R3; requests R2
- p3: requests R1, R3
- p4: produces R2; requests R3

Note: These are **consumable resources** (produced and consumed, not reusable).

#### (i) Resource Allocation Graph

```
        R1 (available=1)         R2 (available=0)         R3 (available=1)
       /    |    \                    |    \               /    |    \
      /     |     \                  |     \             /     |     \
   <--p1  <--p3   p2-->           <--p2   p4-->       <--p1  <--p3  <--p4
                   p2-->R3                                     p2-->R3
```

More precisely:
- **Request edges (process -> resource):** p1->R1, p1->R3, p2->R2, p3->R1, p3->R3, p4->R3
- **Producer edges (process -> resource, produces):** p2 produces R1, p2 produces R3, p4 produces R2

With consumable resources:
- R1 has 1 available; p2 can produce more R1
- R2 has 0 available; p4 can produce R2
- R3 has 1 available; p2 can produce more R3

#### (ii) Which processes are deadlocked?

Analysis of possible execution:
- **p2** needs R2. R2 has 0 available. R2 can only be produced by p4.
- **p4** needs R3. R3 has 1 available. So p4 CAN acquire R3.
- Once p4 executes, it produces R2 (making R2 available=1) and consumes R3.
- Now p2 can acquire R2. Once p2 executes, it produces R1 and R3.
- Now R1 and R3 are replenished. p1 needs R1 and R3 -- both can be satisfied.
- p3 needs R1 and R3 -- after p1 finishes or if p2 produces enough, p3 can proceed.

Let's trace more carefully with the initial available counts:
- R1=1, R2=0, R3=1

Can p4 proceed? p4 requests R3. R3 available=1, so yes. p4 gets R3, produces R2. Now: R1=1, R2=1, R3=0.

Can p2 proceed? p2 requests R2. R2=1, so yes. p2 gets R2, produces R1 and R3. Now: R1=2, R2=0, R3=1.

Can p1 proceed? p1 requests R1 and R3. R1=2>=1 and R3=1>=1, so yes. p1 gets them. Now: R1=1, R2=0, R3=0.

Can p3 proceed? p3 requests R1 and R3. R1=1>=1 but R3=0<1. **p3 cannot proceed.**

However, no more producers of R3 remain (p2 already executed). So **p3 is deadlocked** if we assume each process executes only once.

Actually, let's reconsider: if we allocate resources differently:
- p4 gets R3 (R3: 1->0), produces R2 (R2: 0->1)
- p2 gets R2 (R2: 1->0), produces R1 and R3 (R1: 1->2, R3: 0->1)
- p1 requests R1 and R3: gets both (R1: 2->1, R3: 1->0)
- p3 requests R1 and R3: R1=1 available but R3=0. No one left to produce R3.

Alternatively:
- p4 gets R3, produces R2. (R1=1, R2=1, R3=0)
- p2 gets R2, produces R1, R3. (R1=2, R2=0, R3=1)
- p3 gets R1 and R3. (R1=1, R2=0, R3=0)
- p1 needs R1 and R3. R1=1 but R3=0. **p1 is deadlocked.**

In either ordering, one of p1 or p3 will be deadlocked (there are only enough total producible R3 units for one of them). With a single-execution model for consumable resources, there is a potential deadlock involving **p1 and p3** -- they both compete for R1 and R3, and only one set can be satisfied.

**Answer:** If processes execute only once, **either p1 or p3 will be deadlocked** (both request R1 and R3, but there are insufficient consumable resources for both). With optimal ordering, p4 and p2 can always complete, but only one of p1/p3 can be satisfied. If no particular priority is assigned, we can say **p1 and p3 are involved in potential deadlock** due to competing for limited consumable resources.

### 3c) Three Methods of Allocating File Storage (9 marks)

#### (i) Contiguous Allocation

**Description:** Each file occupies a set of contiguous blocks on disk. The directory entry stores the starting block address and the length (number of blocks).

**Advantages:**
- **Excellent read performance:** Sequential and direct access are both fast because blocks are adjacent -- minimal disk head movement.
- **Simple implementation:** Only need to store start address and length.
- **Fast random access:** Block i of a file is at start + i.

**Disadvantages:**
- **External fragmentation:** As files are created and deleted, free space becomes fragmented into small non-contiguous gaps.
- **File size must be known at creation time** -- difficult to extend a file if adjacent blocks are occupied.
- Requires periodic compaction (defragmentation) which is expensive.

#### (ii) Linked Allocation

**Description:** Each file is a linked list of disk blocks. Each block contains a pointer to the next block. The directory entry stores the first and last block pointers. Blocks can be scattered anywhere on disk.

**Advantages:**
- **No external fragmentation:** Any free block can be used; files can grow dynamically.
- **No need to declare file size in advance** -- simply allocate blocks as needed.
- **Simple file growth:** Append a new block and update the link.

**Disadvantages:**
- **Poor random access:** To access block n, must traverse the chain from block 0 through n-1 (O(n)).
- **Space overhead:** Each block uses space for the pointer (e.g., 4 bytes per 512-byte block).
- **Reliability:** If one pointer is corrupted, the rest of the file is lost.

**Improvement:** File Allocation Table (FAT) -- keep all pointers in a table in memory for faster traversal.

#### (iii) Indexed Allocation

**Description:** Each file has an **index block** (inode) that contains an array of pointers to all the file's data blocks. The directory entry points to the index block.

**Advantages:**
- **Efficient random access:** To access block i, look up pointer i in the index block -- O(1).
- **No external fragmentation:** Blocks can be anywhere on disk.
- **Supports dynamic growth:** Add a new block pointer to the index.

**Disadvantages:**
- **Space overhead for small files:** Even a file using one data block still needs an entire index block.
- **Index block size limits file size:** If the index block is one disk block, the maximum file size is limited. Solution: multi-level indexing (indirect blocks) as in Unix inodes.
- **Extra disk access:** Must read the index block before accessing data blocks.

### 3d) Five Events During DMA Disk-to-Memory Transfer (5 marks)

1. **CPU initiates the transfer:** The CPU programs the DMA controller with the source device (disk), destination memory address, transfer size (byte count), and direction (read). The CPU writes these values to the DMA controller's registers.

2. **DMA controller requests disk data:** The DMA controller signals the disk controller to begin reading the specified data. The disk controller positions the read head and begins reading data from the disk surface into the disk controller's buffer.

3. **DMA controller transfers data to memory:** Once data is available in the disk controller buffer, the DMA controller takes control of the memory bus (cycle stealing or burst mode) and transfers the data directly from the disk controller to the specified memory address, bypassing the CPU.

4. **DMA controller updates its registers:** After each block/word is transferred, the DMA controller increments the memory address pointer and decrements the byte count. This continues until all data is transferred.

5. **DMA controller interrupts the CPU:** When the byte count reaches zero (transfer complete), the DMA controller asserts an interrupt line to the CPU. The CPU's interrupt handler then processes the completion (e.g., notifying the requesting process that data is available in memory).

### 3e) Elevator (SCAN) Disk Scheduling Algorithm (3 marks)

**Operation:**

The Elevator algorithm (also called SCAN) moves the disk arm in one direction (e.g., toward higher cylinder numbers), servicing all pending requests in that direction. When the arm reaches the last request (or the end of the disk) in that direction, it reverses direction and services requests going the other way. This mimics the behaviour of a building elevator.

**Example:** If the head is at cylinder 50 moving toward higher numbers, and requests are at cylinders 20, 30, 65, 80, 120, it services 65, 80, 120 first, then reverses to service 30, 20.

**Key advantage:** Provides more uniform wait times than FCFS or SSTF, and prevents starvation (every request will eventually be reached as the arm sweeps back and forth).

**One limitation:**
- **Uneven wait times for edge cylinders vs. middle cylinders:** Requests at the extremes of the disk may have longer wait times because the arm must travel to the end before reversing. Requests near the middle are passed twice per full sweep (once in each direction), while requests near the edges are only passed once. This leads to **unfairness** -- requests just behind the current head direction must wait for the arm to travel all the way to the end and back. The C-SCAN (Circular SCAN) variant addresses this by only servicing requests in one direction and jumping back to the beginning.

## Question 4: Process Synchronisation and Mutual Exclusion (25 Marks)

### 4a) Race Conditions (4 marks)

**What is a race condition?**

A race condition occurs when the outcome of a computation depends on the **relative timing or order of execution** of two or more concurrent processes/threads that access shared data. The result is non-deterministic and may be incorrect depending on which process "wins the race" to read or write the shared variable.

**How they occur:**

Race conditions occur when:
1. Two or more processes/threads access a **shared variable or resource**.
2. At least one of them performs a **write** operation.
3. There is **no synchronisation mechanism** to control the order of access.

**Example:**
```
Shared variable: counter = 5

Thread A: counter = counter + 1    (read 5, compute 6, write 6)
Thread B: counter = counter - 1    (read 5, compute 4, write 4)
```

If both read the value 5 before either writes:
- Thread A writes 6
- Thread B writes 4 (overwriting A's result)
- Final value: 4 (should be 5)

The low-level interleaving that causes this:
1. Thread A: LOAD counter (gets 5)
2. Thread B: LOAD counter (gets 5)
3. Thread A: ADD 1 (computes 6)
4. Thread B: SUB 1 (computes 4)
5. Thread A: STORE counter (writes 6)
6. Thread B: STORE counter (writes 4) -- A's update is lost

### 4b) Four Conditions for an Acceptable Solution to the Critical Section Problem (4 marks)

1. **Mutual Exclusion:** At most one process may be executing in its critical section at any given time. If process Pi is in its critical section, no other process can be in its critical section.

2. **Progress (No deadlock):** If no process is currently in its critical section and some processes wish to enter, only those processes that are trying to enter the critical section participate in the decision, and the decision cannot be postponed indefinitely. Some process must be able to enter -- the system must not deadlock.

3. **Bounded Waiting (No starvation):** There must exist a bound on the number of times other processes are allowed to enter their critical sections after a process has made a request to enter and before that request is granted. No process should wait forever.

4. **No assumptions about speed:** The solution must not depend on the relative speeds of processes or the number of CPUs. It must work regardless of execution speed or scheduling decisions.

---

### 4c) Strict Alternation Algorithm Implementation (4 marks)

```c
// Shared variable
int turn = 0;  // indicates whose turn it is (0 or 1)

// Process 0
void process0() {
    while (true) {
        while (turn != 0)
            ; // busy wait -- spin until it's our turn
        
        // CRITICAL SECTION
        critical_section_0();
        
        turn = 1;  // give turn to process 1
        
        // REMAINDER SECTION
        remainder_section_0();
    }
}

// Process 1
void process1() {
    while (true) {
        while (turn != 1)
            ; // busy wait -- spin until it's our turn
        
        // CRITICAL SECTION
        critical_section_1();
        
        turn = 0;  // give turn to process 0
        
        // REMAINDER SECTION
        remainder_section_1();
    }
}
```

The `turn` variable alternates between 0 and 1, enforcing strict alternation of access to the critical section.

---

### 4d) Critical Evaluation of Strict Alternation for Mutual Exclusion (7 marks)

**Evaluation against the four conditions:**

**1. Mutual Exclusion -- SATISFIED:**
The `turn` variable can only be 0 or 1 at any instant. Process 0 enters only when `turn == 0`, and Process 1 enters only when `turn == 1`. Since a single integer cannot be both 0 and 1 simultaneously, at most one process can be in the critical section at any time. Mutual exclusion is guaranteed.

**2. Progress -- NOT SATISFIED:**
This is the critical flaw. Strict alternation requires processes to take turns, even if one process does not want to enter its critical section. Consider:
- Process 0 finishes its critical section and sets `turn = 1`.
- Process 0 quickly completes its remainder section and wants to enter the critical section again.
- But Process 1 is still in its remainder section (a long computation) and has not yet entered and exited its critical section.
- Process 0 is blocked, even though no process is in the critical section.

A process **outside its critical section** (Process 1 in the remainder) is preventing another process from entering. This violates the progress condition, which states that only processes trying to enter should influence the decision.

**3. Bounded Waiting -- SATISFIED (trivially):**
Due to the strict alternation, each process waits at most one turn. After Process 1 enters and exits, it is guaranteed to be Process 0's turn next. No starvation is possible in terms of being permanently skipped.

**4. No assumption about speed -- PARTIALLY VIOLATED:**
The algorithm implicitly assumes both processes execute at roughly similar rates. If one process is much slower or has a much longer remainder section, the faster process is unnecessarily throttled.

**Additional problems:**

- **Poor performance when processes have unequal frequency of critical section access:** If Process 0 needs the critical section 100 times for every 1 time Process 1 needs it, Process 0 is blocked 99% of the time waiting for Process 1 to take its turn.
- **Coupling between processes:** The algorithm creates an artificial dependency between processes that may have no logical connection.
- **Not scalable:** Extending to n processes requires n-way alternation, which becomes increasingly restrictive.

**Conclusion:** While strict alternation does achieve mutual exclusion and bounded waiting, its failure to satisfy the progress condition makes it an **unacceptable** solution to the critical section problem. A process that does not wish to enter its critical section should not be able to block another process. Better alternatives include Peterson's algorithm, which satisfies all conditions.

### 4e) Busy Waiting (3 marks)

**Busy waiting** (also called spinning or spinlocking) is a synchronisation technique where a process **repeatedly tests a condition in a tight loop** while waiting for that condition to become true. The process remains in the running/ready state and continuously consumes CPU cycles executing the loop instructions.

```c
while (condition_not_met)
    ; // do nothing -- just keep checking
```

**Characteristics:**
- The process does **not yield the CPU** or enter a blocked/sleeping state.
- It wastes CPU cycles that could be used by other processes.
- On a **single-processor** system, it is particularly wasteful because the process holding the lock cannot make progress while the spinning process occupies the CPU.
- On **multiprocessor** systems with short expected waits, it can be more efficient than blocking because it avoids the overhead of context switching (saving state, scheduling another process, restoring state when unblocked).

**The term "spinlock"** refers to a lock implemented using busy waiting -- the process "spins" in a loop until the lock becomes available.

---

### 4f) Three Problems with Disabling Interrupts During a Long Critical Section (3 marks)

1. **System unresponsiveness:** With interrupts disabled, the system cannot respond to any external events -- keyboard input, network packets, disk completion signals, and timer ticks are all ignored. If the critical section is long, the system appears frozen to users and external devices may time out or lose data.

2. **Clock/timer disruption:** The system timer interrupt is suppressed, so the OS loses track of elapsed time. Scheduling decisions cannot be made (no preemption), time-of-day clock drifts, and time-dependent operations (timeouts, alarms, scheduling deadlines) are disrupted. In real-time systems, this could cause missed deadlines.

3. **Does not work on multiprocessor systems:** Disabling interrupts on one CPU only prevents interrupts on that specific processor. Other CPUs can still access the shared data concurrently, so mutual exclusion is **not achieved** on multi-CPU systems. This makes interrupt disabling fundamentally unsuitable as a general mutual exclusion mechanism in modern systems.
