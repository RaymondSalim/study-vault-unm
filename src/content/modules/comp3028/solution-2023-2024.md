## Question 1: Computer Security in General (20 marks)

### 1a) Prevention, Detection and Reaction — Physical vs Digital (6 marks)

Methods used for the prevention, detection, and reaction to theft of physical property are often inappropriate for digital information because of fundamental differences in the nature of physical and digital assets.

**Prevention:** Physical security relies on locks, guards, and barriers to prevent access. However, digital information can be copied remotely over networks without physical proximity. A thief does not need to "take" the data — they can copy it, and the owner may still retain the original, making prevention through physical barriers irrelevant.

**Detection:** With physical property, theft is usually noticed because the item is missing. Digital theft may go completely undetected because the original data remains intact — the owner is unaware that a copy has been made. Traditional inventory checks are meaningless when nothing appears to be gone.

**Reaction:** When physical property is stolen, law enforcement can search for the item, and recovery is meaningful — there is one object to retrieve. With digital information, once data is copied and distributed, it cannot be "un-stolen." Reaction is complicated by jurisdictional issues (the attacker may be in another country), anonymity of the internet, and the impossibility of knowing how many copies exist.

### 1b) Password System — Authentication and Access Control (5 marks)

#### i) Definitions (2 marks)

- **Authentication:** The process of verifying the identity of a user or entity — confirming that someone is who they claim to be. For password systems, this means checking that the user knows the correct password associated with their account.

- **Access control:** The process of determining whether an authenticated user is permitted to perform a specific operation on a specific resource. It enforces policies about who can read, write, execute, or delete particular objects.

#### ii) Intelligent, dictionary, and exhaustive search (3 marks)

- **Intelligent search:** The attacker uses knowledge about the target user to guess passwords — e.g., their name, birthday, pet's name, favourite sports team, or common patterns. This exploits the tendency of users to choose personally meaningful passwords.

- **Dictionary search:** The attacker tries every word in a dictionary (often extended with common substitutions like "p@ssw0rd", numbers appended, etc.). This exploits the fact that many users choose real words or common phrases as passwords.

- **Exhaustive search (brute force):** The attacker systematically tries every possible combination of characters up to a certain length. This guarantees finding the password but is the slowest approach — feasibility depends on password length and the character set used.

### 1d) Three network threats a firewall does not protect against (6 marks)

**1. Internal attacks / insider threats**
A firewall controls traffic crossing the network boundary. It cannot protect against attacks originating from within the trusted internal network. A malicious insider (or compromised internal machine) can attack other internal systems freely since this traffic never passes through the perimeter firewall. (Host-based firewalls can partially mitigate this, but a perimeter-only firewall cannot.)

**2. Application-layer attacks via allowed protocols (e.g., SQL injection, XSS)**
Packet-filtering firewalls and even stateful firewalls inspect only headers (IP, TCP/UDP). They allow traffic on permitted ports (e.g., port 80/443 for web). An attacker can embed malicious payloads within legitimate HTTP requests (SQL injection, cross-site scripting). Only application-layer firewalls/WAFs can inspect content deeply enough to detect these — basic firewalls cannot.

**3. Social engineering / phishing attacks**
A firewall cannot prevent a user from voluntarily clicking a phishing link in an email and entering their credentials on a fraudulent website. The traffic appears to be a normal HTTPS connection to a legitimate-looking external site. No firewall rule can distinguish a user's legitimate web browsing from a user being deceived into visiting a phishing site.

## Question 2: Cryptography (20 marks)

### 2a) Why modern ciphers encrypt blocks of data (4 marks)

**Reason 1: Resistance to frequency analysis**
If encryption operates on single characters or bits, the statistical patterns of the underlying language (e.g., letter frequency in English) are preserved in the ciphertext. By encrypting larger blocks, the input space becomes exponentially larger, destroying statistical patterns and making frequency analysis infeasible.

**Reason 2: Diffusion**
Encrypting blocks allows each ciphertext bit to depend on many plaintext bits (diffusion). A change in one plaintext bit affects many bits in the ciphertext block, making it much harder for an attacker to deduce relationships between plaintext and ciphertext. With single-bit or single-letter encryption, each unit is encrypted independently, providing no diffusion.

### 2b) Two "hard" problems for public-key encryption (4 marks)

**1. Integer Factorisation Problem (used by RSA)**
Given a large composite number *n = p × q* (product of two large primes), it is computationally infeasible to determine the prime factors *p* and *q*. This is believed to be secure because no polynomial-time algorithm for factoring has been found despite decades of research. The best known algorithms (General Number Field Sieve) are sub-exponential but still impractical for sufficiently large *n* (e.g., 2048+ bits).

**2. Discrete Logarithm Problem (used by Diffie-Hellman, ElGamal, DSA)**
Given a large prime *p*, a generator *g*, and a value *y = g^x mod p*, it is computationally infeasible to determine *x*. This is believed secure because computing discrete logarithms in large prime-order groups has no known polynomial-time classical algorithm. The best attacks (Index Calculus) are sub-exponential and infeasible for sufficiently large parameters.

### 2c) How RSA works (12 marks)

#### Key Generation

1. **Select two large primes** *p* and *q* (each typically 1024+ bits).
2. **Compute** *n = p × q*. This is the modulus.
3. **Compute Euler's totient:** *φ(n) = (p − 1)(q − 1)*.
4. **Choose public exponent** *e* such that:
   - 1 < e < φ(n)
   - gcd(e, φ(n)) = 1 (e is coprime to φ(n))
   - Common choice: e = 65537
5. **Compute private exponent** *d* = e^(−1) mod φ(n), i.e., find *d* such that *e × d ≡ 1 (mod φ(n))*. This is computed using the Extended Euclidean Algorithm.

**Public key:** PU = {e, n} — published openly.
**Private key:** PR = {d, n} — kept secret. The primes *p* and *q* are discarded.

#### Encryption

To encrypt a plaintext message *M* (where 0 ≤ M < n):

**C = M^e mod n**

The sender uses the recipient's public key (e, n).

#### Decryption

To decrypt ciphertext *C*:

**M = C^d mod n**

The recipient uses their private key (d, n).

**Correctness:** This works because M^(ed) ≡ M (mod n) by Euler's theorem, since ed ≡ 1 (mod φ(n)).

#### Security Basis

The security of RSA depends on the **difficulty of factoring large numbers**. An attacker who knows the public key (e, n) would need to factor *n* into *p* and *q* to compute *φ(n)* and then derive *d*. For sufficiently large *n* (2048+ bits), factoring is computationally infeasible with current technology.

Additional security considerations:
- *p* and *q* must be of similar size but not too close together.
- *p* and *q* must be chosen randomly and independently.
- *d* must be sufficiently large (small *d* is vulnerable to Wiener's attack).
- Proper padding schemes (e.g., OAEP) must be used to prevent attacks on textbook RSA.

### 3b) Access Control List vs Capability (12 marks)

#### (i) Ease of making an authorisation decision during execution (3 marks)

**Access Control List (ACL):**
Easy. When a subject requests access to an object, the system looks up the ACL attached to that object and checks if the subject has the required permission. This is a single lookup on the object's list.

**Capability:**
Also easy. The subject presents its capability (token) for the object. The system simply verifies the capability is valid and grants access. No lookup of external tables is needed — the capability itself is the proof of authorisation.

**Verdict:** Both are comparably easy during execution, though capabilities can be slightly faster as they avoid a lookup — the subject carries the proof.

#### (ii) Ease of adding access for a new subject (3 marks)

**ACL:**
Easy. Simply add the new subject's entry to the ACL of each object they need access to. Each object's list is independently modifiable.

**Capability:**
More difficult. You must issue new capability tokens to the new subject for each object they need. This requires identifying all relevant objects and creating/distributing capabilities to the subject. If there are many objects, this is cumbersome.

**Verdict:** ACL is easier — you modify lists on specific objects. Capabilities require distributing tokens to the new subject.

#### (iii) Ease of deleting access by a subject (3 marks)

**ACL:**
Easy. Remove the subject's entry from each object's ACL. The system immediately stops granting access since it checks the ACL on every request.

**Capability:**
Difficult (revocation problem). The subject holds the capability tokens. You must either: (a) track all capabilities issued and invalidate them (requires a central revocation list), (b) use short-lived capabilities that expire, or (c) change the object's identifier so existing capabilities no longer refer to it. All options add complexity.

**Verdict:** ACL is much easier for revocation. Capability-based systems have the well-known "revocation problem."

#### (iv) Ease of creating a new object to which all subjects have access by default (3 marks)

**ACL:**
Difficult. You must create an ACL for the new object and add every existing subject to it. If there are many subjects, this requires enumerating all of them and adding entries.

**Capability:**
Difficult. You must issue a new capability for the new object to every subject. This requires distributing tokens to all subjects, which can be complex in a large system.

**Verdict:** Both are similarly difficult — both require an action for every subject in the system. However, in practice, ACLs can use a "default allow" rule or a wildcard entry (e.g., "all users: read"), making it slightly easier. Capability systems would need to broadcast a new token to every subject.
