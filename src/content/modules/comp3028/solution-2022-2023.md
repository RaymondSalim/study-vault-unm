## Question 1: Computer Security in General (20 marks)

### 1a) Two risks to data on a laptop PC (4 marks)

**Risk 1: Theft or loss of the laptop**
- The laptop can be physically stolen or lost while travelling. This compromises **confidentiality** because an attacker who possesses the physical device can access the hard drive and read sensitive sales data (customer details, pricing strategies, etc.) unless the drive is encrypted.

**Risk 2: Interception of email over insecure networks**
- Sales staff often connect to public Wi-Fi (hotels, cafes). An attacker can perform a man-in-the-middle attack and intercept email communications. This compromises **confidentiality** (emails read by attacker) and potentially **integrity** (emails can be modified in transit).

### 1b) Biometrics (7 marks)

#### i) What is biometrics in relation to authentication? (3 marks)

Biometrics refers to the use of measurable physical or behavioural characteristics of a person to verify their identity. In authentication, biometrics serves as a "something you are" factor — the system captures a biometric sample from the user, extracts features, and compares them against a stored template to confirm identity. Unlike passwords or tokens, biometric traits are inherently tied to the individual and cannot be easily shared, lost, or forgotten.

#### ii) Two types of biometric with one example each (4 marks)

| Type | Description | Example |
|------|-------------|---------|
| **Physiological biometric** | Based on physical body characteristics | Fingerprint recognition |
| **Behavioural biometric** | Based on patterns of behaviour | Keystroke dynamics (typing rhythm) |

### 1c) Brute Force Attack (5 marks)

#### i) How a Brute Force Attack works (2 marks)

A brute force attack works by systematically trying every possible combination of keys (or passwords) until the correct one is found. The attacker encrypts the known plaintext (or decrypts the ciphertext) with each possible key and checks whether the result matches the expected output. Given enough time and computational resources, a brute force attack is guaranteed to find the correct key.

#### ii) Effect of different key sizes (3 marks)

The key size determines the total number of possible keys, which directly affects the time required for a brute force attack.

**Example:**
- A **56-bit key** (e.g., DES) has 2^56 ≈ 7.2 × 10^16 possible keys. At 10^13 decryptions per second, this can be exhausted in about 1 hour.
- A **128-bit key** (e.g., AES-128) has 2^128 ≈ 3.4 × 10^38 possible keys. Even at 10^13 decryptions per second, this would take approximately 10^18 years — far longer than the age of the universe.

Each additional bit doubles the number of possible keys, so increasing the key size exponentially increases the resistance to brute force attacks.

### 1d) SYN Spoofing Attack (4 marks)

A **SYN spoofing (SYN flood) attack** exploits the TCP three-way handshake:

1. The attacker sends a large number of TCP SYN (synchronise) packets to the target server, each with a **forged (spoofed) source IP address**.
2. The server responds to each SYN with a SYN-ACK and allocates resources (a Transmission Control Block) in its backlog queue, waiting for the final ACK.
3. Since the source IP addresses are spoofed, the SYN-ACK packets go to non-existent or uninvolved hosts, so the final ACK never arrives.
4. The server's connection table fills up with half-open connections.

**Effect:** The server exhausts its memory/connection resources and can no longer accept legitimate TCP connections, resulting in a **Denial of Service (DoS)**. Legitimate users are unable to establish connections to the server.

## Question 2: Public-Key Cryptosystem, Hash Function, and Modes of Operation (20 marks)

### 2a) RSA Key Generation (4 marks)

**Step 1:** Select two large prime numbers, *p* and *q*.

**Step 2:** Compute *n = p × q*. This is the modulus for both the public and private keys.

**Step 3:** Compute Euler's totient function: *φ(n) = (p − 1)(q − 1)*.

**Step 4:** Choose an integer *e* such that *1 < e < φ(n)* and *gcd(e, φ(n)) = 1* (i.e., *e* is coprime to *φ(n)*). Common choice: *e = 65537*.

**Step 5:** Compute *d* such that *d × e ≡ 1 (mod φ(n))*, i.e., *d = e^(−1) mod φ(n)*.

- **Public key:** (e, n)
- **Private key:** (d, n)

---

### 2b) RSA Encryption and Decryption Equations (2 marks)

Given message *M* where *M < n*:

- **Encryption:** C = M^e mod n (using receiver's public key)
- **Decryption:** M = C^d mod n (using receiver's private key)

---

### 2c) ECB and CBC Modes (6 marks)

When *M* is larger than *n*, the message must be broken into blocks *M₁, M₂, M₃, ...* each smaller than *n*.

#### ECB (Electronic Codebook) Mode

**Encryption:**
```
M₁ → [Encrypt with e,n] → C₁
M₂ → [Encrypt with e,n] → C₂
M₃ → [Encrypt with e,n] → C₃
...
```

**Decryption:**
```
C₁ → [Decrypt with d,n] → M₁
C₂ → [Decrypt with d,n] → M₂
C₃ → [Decrypt with d,n] → M₃
...
```

Each block is encrypted independently. Weakness: identical plaintext blocks produce identical ciphertext blocks.

#### CBC (Cipher Block Chaining) Mode

**Encryption:**
```
IV ──────────┐
             ⊕ ← M₁ → [Encrypt with e,n] → C₁
C₁ ──────────┐
             ⊕ ← M₂ → [Encrypt with e,n] → C₂
C₂ ──────────┐
             ⊕ ← M₃ → [Encrypt with e,n] → C₃
...
```

- C₁ = Encrypt(M₁ ⊕ IV)
- Cᵢ = Encrypt(Mᵢ ⊕ Cᵢ₋₁)

**Decryption:**
```
C₁ → [Decrypt with d,n] → ⊕ ← IV → M₁
C₂ → [Decrypt with d,n] → ⊕ ← C₁ → M₂
C₃ → [Decrypt with d,n] → ⊕ ← C₂ → M₃
...
```

- M₁ = Decrypt(C₁) ⊕ IV
- Mᵢ = Decrypt(Cᵢ) ⊕ Cᵢ₋₁

Advantage over ECB: identical plaintext blocks produce different ciphertext blocks due to chaining.

### 2d) Anti-replay and Non-repudiation using RSA (8 marks)

**Method: Digital Signature with Timestamp/Nonce**

**At the Sender (Alice):**
1. Alice prepares message *M* and appends a **timestamp *T*** (or sequence number) to create *M' = M || T*.
2. Alice computes the hash: *h = H(M')* using a cryptographic hash function (e.g., SHA-1).
3. Alice signs the hash with her **private key**: *Sig = h^(d_A) mod n_A*. This is the digital signature.
4. Alice sends *(M', Sig)* to Bob. (Optionally, encrypt M' with Bob's public key for confidentiality.)

**At the Receiver (Bob):**
1. Bob receives *(M', Sig)*.
2. Bob computes *h' = H(M')* independently.
3. Bob verifies the signature using Alice's **public key**: computes *h'' = Sig^(e_A) mod n_A*.
4. If *h' = h''*, the signature is valid.
5. Bob checks the **timestamp *T*** — if it is too old or has been seen before, the message is rejected (anti-replay).

**Non-repudiation:** Only Alice possesses her private key *d_A*, so only she could have produced the signature. She cannot later deny having sent the message.

**Anti-replay:** The timestamp/nonce ensures each message is unique. A replayed message will have an old or duplicate timestamp and will be rejected.

**Two factors on which security depends:**
1. **The secrecy of Alice's private key** — if compromised, an attacker can forge signatures.
2. **The collision resistance of the hash function H(x)** — if an attacker can find two messages with the same hash, they could substitute one signed message for another.

## Question 3: Passwords and Network Security (20 marks)

### 3a) Password Security (10 marks)

#### i) What can hackers do to get plaintext passwords (unsalted)? (2 marks)

Since the passwords are hashed but not salted, attackers can:
- Use **pre-computed rainbow tables** — large lookup tables mapping hash values back to common passwords. They simply look up each hash in the table to find the corresponding plaintext.
- Use **dictionary attacks** — hash common passwords/words and compare against the leaked hashes. Since there is no salt, one computation checks against ALL users simultaneously.

#### ii) What if passwords were salted with the same salt for each user? (2 marks)

If the same salt is used for all users:
- **Rainbow tables become ineffective** because the pre-computed tables were built without the salt. The attacker would need to regenerate tables incorporating the specific salt value.
- **However**, since the salt is the same for all users, if two users have the same password, they will still have the same hash. The attacker can still perform a dictionary attack once — hashing each candidate password with the known salt — and check the result against all users simultaneously.
- The benefit is limited compared to using a **unique salt per user**.

#### iii) Three recommendations to protect user passwords (6 marks)

1. **Use a unique random salt per user** — Generate a different random salt for each user and store it alongside the hash. This forces the attacker to attack each user's password individually, making pre-computation attacks infeasible and dramatically increasing the total work required.

2. **Use a slow/adaptive hashing algorithm (e.g., bcrypt, scrypt, or Argon2)** — These algorithms are intentionally computationally expensive, with configurable work factors. This makes brute force and dictionary attacks orders of magnitude slower compared to fast hashes like SHA-1 or MD5.

3. **Enforce strong password policies** — Require minimum length (e.g., 12+ characters), encourage passphrases, and check passwords against known breached password lists. This ensures that even if the hashes are leaked, the passwords are resistant to dictionary attacks.

### 3b) Firewall Deployment Scenarios (8 marks)

#### i) Firewall at the network perimeter (4 marks)

**Strengths:**
- Protects all internal machines (servers, desktops) from external threats with a single point of enforcement.
- Easier to manage and configure — one ruleset to maintain.
- Can inspect and filter all traffic entering/leaving the network.
- Good for blocking known malicious traffic, port scans, and external attacks.

**Weaknesses:**
- Does **not** protect against internal threats (e.g., a malicious employee on the LAN).
- **Laptops** that leave the network (e.g., at home, public Wi-Fi) are completely unprotected.
- Once an attacker is past the perimeter (e.g., via phishing), there is no further defence.
- Cannot protect against threats entering via encrypted tunnels (VPN) or allowed ports (HTTP/HTTPS).

#### ii) Firewalls on every end host machine (4 marks)

**Strengths:**
- Protects each machine individually, including laptops that roam outside the corporate network.
- Defends against internal threats — even if one machine is compromised, the firewall on other machines can block lateral movement.
- Rules can be tailored to the specific role of each machine (e.g., a server only accepts port 443).

**Weaknesses:**
- **Management overhead** — maintaining consistent firewall policies across hundreds/thousands of machines is complex.
- Users may disable or misconfigure their host firewall.
- Consumes resources (CPU, memory) on each end host.
- Lacks the network-wide visibility of a perimeter firewall — cannot correlate patterns across multiple hosts.

### 3c) Phishing Attack (2 marks)

A **phishing attack** is a social engineering attack where an attacker sends fraudulent communications (typically email) that appear to come from a trusted source (e.g., a bank, employer, or popular service). The message tricks the victim into clicking a link to a fake website that mimics the legitimate site's login page. When the user enters their username and password on the fake site, the credentials are captured by the attacker. This directly breaks password security because the user voluntarily (but unknowingly) discloses their plaintext password to the attacker, bypassing any cryptographic protections on the server side.
