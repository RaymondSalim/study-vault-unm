## Question 1: Segmentation and Motion Detection (20 marks)

**Given:** The GMM equation: p(x) = Sum(pi_i * p(x|theta_i)), for i = 1 to g

### (a) Definitions in Image Segmentation Context [4 marks]

- **g**: The number of Gaussian components (clusters) in the mixture model. In image segmentation, this represents the number of distinct segments or classes that the image is being divided into (e.g., foreground and background would give g = 2).

- **p(x)**: The overall probability density of observing a pixel value x. This is the likelihood of a particular pixel intensity (or colour vector) occurring anywhere in the image, computed as a weighted sum of all component densities.

- **pi_i** (mixing coefficients): The prior probability (weight) of the i-th Gaussian component. In segmentation, this represents the proportion of pixels in the image that belong to segment i. The mixing coefficients satisfy: Sum(pi_i) = 1, with 0 <= pi_i <= 1.

- **theta_i**: The parameters of the i-th Gaussian distribution, typically theta_i = {mu_i, Sigma_i}, where mu_i is the mean (average pixel intensity/colour for segment i) and Sigma_i is the covariance matrix (spread of pixel values within that segment).

### (b) Bayes' Rule and EM Algorithm for Model-Based Image Segmentation [8 marks]

**Bayes' Rule:**

P(omega_i | x) = p(x | omega_i) * P(omega_i) / p(x)

Where:
- P(omega_i | x) is the posterior probability that pixel x belongs to class omega_i
- p(x | omega_i) is the likelihood of observing x given class omega_i
- P(omega_i) is the prior probability of class omega_i
- p(x) is the evidence (total probability of observing x)

**Use in EM Algorithm for GMM-based segmentation:**

The Expectation-Maximisation (EM) algorithm iteratively refines the GMM parameters to maximise the likelihood of the observed data (pixel values). It alternates between two steps:

**E-Step (Expectation):** Using Bayes' Rule, compute the posterior probability (responsibility) that each pixel belongs to each component:

r_ik = P(omega_k | x_i) = (pi_k * N(x_i | mu_k, Sigma_k)) / Sum_j(pi_j * N(x_i | mu_j, Sigma_j))

This tells us how "responsible" each Gaussian component k is for generating pixel x_i. This is a direct application of Bayes' Rule where the likelihood is the Gaussian density and the prior is the mixing coefficient.

**M-Step (Maximisation):** Update the model parameters using the responsibilities:

- mu_k = Sum_i(r_ik * x_i) / Sum_i(r_ik) — weighted mean of pixels assigned to component k
- Sigma_k = Sum_i(r_ik * (x_i - mu_k)(x_i - mu_k)^T) / Sum_i(r_ik) — weighted covariance
- pi_k = (1/N) * Sum_i(r_ik) — proportion of data explained by component k

**Segmentation procedure:**
1. Initialise GMM parameters (e.g., using k-means).
2. Repeat E-step and M-step until convergence (log-likelihood stops increasing significantly).
3. Assign each pixel to the component with the highest posterior probability: segment(x_i) = argmax_k r_ik.

The key insight is that Bayes' Rule provides the principled mechanism in the E-step for soft-assigning pixels to segments, while the M-step finds the maximum likelihood parameters given those assignments.

### (c) GMMs for Motion Detection [4 marks]

**How GMMs detect motion:**

For each pixel location in the image, a GMM is built to model the distribution of pixel values observed over time at that location. Background pixels tend to cluster around certain values (captured by the Gaussian components), while foreground (moving) pixels deviate from this learned distribution.

- At each frame, a new pixel value is compared against the GMM for that location.
- If the pixel value has high probability under the background model (falls within a threshold of one of the dominant Gaussians), it is classified as background.
- If the pixel value has low probability (does not match any dominant component), it is classified as foreground (motion detected).

**Example where GMM is preferred over a single Gaussian:**

A scene with a tree swaying in the wind. The pixel values at a location on the tree alternate between leaf colour (green) and sky colour (blue) as branches move. A single Gaussian would model this as the average of green and blue (an intermediate value that never actually occurs), poorly representing either mode. A GMM with two or more components can capture both the green-leaf mode and the blue-sky mode as separate Gaussians, correctly identifying both as background. Only a genuinely new object (e.g., a person walking past) would be flagged as foreground.

Other examples include: rippling water surfaces, flickering monitors in the background, or flags waving — all exhibiting multi-modal background distributions.

### (d) Stauffer and Grimson's Adaptive Motion Detection Scheme [4 marks]

**Principle:**

Stauffer and Grimson (1999) proposed modelling each pixel's history with a mixture of K Gaussians (typically K = 3 to 5) that adapts online. The key idea is that background components are those that are both frequently observed (high weight) and low variance (consistent), while foreground components are infrequent or high-variance.

**Operation:**

1. **Initialisation:** Each pixel location is modelled with K Gaussian components, initialised with low weights and high variance.

2. **Matching:** For each new pixel value x_t, check if it matches any existing component (within 2.5 standard deviations of a component's mean).

3. **Update (matched component):**
   - The weight of the matched component is increased: pi_k = (1 - alpha) * pi_k + alpha
   - Unmatched components have weights decreased: pi_k = (1 - alpha) * pi_k
   - The matched component's mean and variance are updated using a learning rate rho:
     - mu_k = (1 - rho) * mu_k + rho * x_t
     - sigma_k^2 = (1 - rho) * sigma_k^2 + rho * (x_t - mu_k)^2

4. **Update (no match):** If no component matches, the least probable component (lowest pi_k / sigma_k) is replaced with a new Gaussian centred at x_t, with high variance and low weight.

5. **Background classification:** Components are ranked by pi_k / sigma_k. The first B components whose cumulative weight exceeds a threshold T are deemed background. If the current pixel matched one of these B components, it is classified as background; otherwise, it is foreground.

6. **Adaptation:** The learning rate alpha controls how quickly the model adapts to changes. This allows the scheme to handle gradual illumination changes, objects that become part of the background (e.g., a parked car), and multi-modal backgrounds.

## Question 2: Tracking (20 marks)

### (a) What is a "Particle" in CONDENSATION Tracking? [2 marks]

A particle (also called a sample) is a hypothesised state of the tracked object at a given time step. Each particle represents one possible configuration of the object's state vector (e.g., position, velocity, scale, orientation) and carries an associated weight (importance) representing how well that hypothesis explains the current observation.

The collection of weighted particles approximates the posterior probability distribution over the object's state. Rather than representing the distribution analytically (which may be intractable for multi-modal or non-Gaussian distributions), the particle set provides a discrete, sample-based approximation.

### (b) Effect of Increasing Number of Particles [4 marks]

**(i) Tracking performance:**

Increasing the number of particles improves tracking performance because:
- The posterior distribution is approximated more accurately with finer resolution.
- Multi-modal distributions (e.g., ambiguous object locations) are better represented, reducing the risk of losing track.
- The tracker is more robust to rapid motion or occlusion because a wider range of hypotheses is maintained.
- Less susceptible to sample impoverishment (where particles collapse onto a single mode).

However, there are diminishing returns — beyond a certain number, additional particles provide negligible improvement.

**(ii) Processing speed:**

Increasing the number of particles decreases processing speed (increases computation time) because:
- Each particle must be propagated through the dynamics model.
- Each particle's weight must be evaluated against the current observation (likelihood computation).
- Resampling from a larger set is more expensive.
- The computational cost scales linearly (O(N)) with the number of particles.

The trade-off is between accuracy and real-time requirements. Practical systems choose N to balance acceptable tracking quality with frame-rate constraints.

### (b) Tracking a Table Tennis Ball Bounced on a Bat [4 marks]

To build upon CONDENSATION for tracking a table tennis ball being bounced on a bat:

**State representation:** The state vector should include position (x, y), velocity (vx, vy), and possibly acceleration to capture the ballistic trajectory and impact dynamics.

**Dynamics model:** Rather than a simple constant-velocity model, use a physics-informed dynamics model:
- Between bounces: ballistic motion with gravity (parabolic trajectory), so the prediction step applies projectile motion equations.
- At bounce: model the sudden reversal in vertical velocity. This can be encoded as a mixture dynamics model — one component for free flight and one for the bounce event.

**Observation model:** Use colour and shape features. A table tennis ball is small, circular, and distinctly coloured (white or orange). The likelihood function evaluates how well each particle's hypothesised position matches a circular blob of the appropriate colour in the image.

**Handling the bounce:** The bounce introduces a sudden, discontinuous change in motion. To handle this:
- Use a mixed-state dynamics model where some particles are propagated as "in flight" and others as "bouncing."
- Alternatively, add noise in the dynamics model large enough to accommodate the velocity reversal, though this is less efficient.
- A switching state-space model (combining CONDENSATION with a discrete mode variable for flight vs. bounce) is the most principled approach.

**Additional considerations:** Increase the number of particles around the predicted bounce point to ensure the distribution covers both pre- and post-bounce trajectories. Motion blur can be handled by incorporating a temporal smearing in the observation model.

### (c) MCMC vs CONDENSATION for Tracking Multiple Similar Targets [10 marks]

**Difficulties faced by CONDENSATION with multiple similar targets:**

1. **Dimensionality curse:** If tracking M targets jointly, the state space has M times the dimensionality of a single target. CONDENSATION requires exponentially more particles as dimensionality grows — making it computationally intractable for even moderate numbers of targets.

2. **Mode coverage:** The posterior distribution over multiple targets is highly multi-modal (each mode corresponds to a different assignment of observations to targets). Standard particle filtering struggles to maintain coverage of all modes with a feasible number of particles.

3. **Identity switching:** When similar targets come close together or cross paths, particles representing different identity assignments get similar weights, leading to identity swaps.

4. **Interaction modelling:** CONDENSATION's resampling step can cause particle collapse, where all particles converge to a single mode, losing track of other targets entirely.

5. **Occlusion handling:** When one target occludes another, the observation model cannot distinguish between them, and the particle set may merge both targets into one.

**Difficulties faced by MCMC:**

1. **Mixing time:** MCMC may take many iterations to explore the full posterior, particularly when modes are well-separated (targets far apart). Poor mixing leads to slow convergence.

2. **Proposal design:** Designing good proposal distributions for joint multi-target states is challenging. Poor proposals lead to high rejection rates.

3. **Burn-in period:** MCMC requires an initial burn-in period before samples represent the true posterior.

4. **No natural temporal propagation:** Unlike particle filters, MCMC does not have an inherent mechanism for propagating samples through time, requiring careful integration with temporal dynamics.

**Why MCMC is better for multiple similar targets:**

1. **Efficient high-dimensional exploration:** MCMC explores the joint state space by making local moves (updating one target at a time while fixing others), avoiding the curse of dimensionality. The computational cost scales linearly with the number of targets rather than exponentially.

2. **Reversible jump MCMC:** Can handle varying numbers of targets by including birth/death moves that add or remove targets from the model, naturally handling targets entering or leaving the scene.

3. **Identity-aware moves:** MCMC can include specific proposal moves such as "swap" moves that explicitly exchange the identities of two nearby targets. This directly addresses the identity switching problem that plagues CONDENSATION.

4. **Joint reasoning:** MCMC naturally incorporates exclusion constraints (two targets cannot occupy the same location) through the acceptance ratio — proposals violating physical constraints are rejected. In CONDENSATION, such constraints are difficult to enforce.

5. **Better mode exploration:** Techniques like parallel tempering or simulated annealing within MCMC can help jump between distant modes, maintaining track of all targets even when they are well-separated.

6. **Computational efficiency:** For M targets, CONDENSATION needs O(N^M) particles for equivalent coverage, while MCMC methods like those proposed by Khan et al. achieve good results with O(M*N) computation per frame.

In summary, MCMC's ability to make targeted, local moves in high-dimensional spaces, combined with specialised proposals for identity management, makes it fundamentally better suited to the multi-target tracking problem than the brute-force sampling approach of CONDENSATION.

## Question 3: Multi-View Stereo (20 marks)

### (a) Advantages and Disadvantages of Correspondence-Based Methods [5 marks]

**Advantages:**

1. **Sparse but accurate:** Produces highly accurate 3D points at locations where correspondences are reliably established (e.g., strong texture, corners).

2. **Well-established algorithms:** Mature techniques exist (SIFT, SURF, ORB for feature detection; epipolar geometry for constraint; RANSAC for robust estimation).

3. **Handles wide baselines:** Feature-based correspondences can match across significant viewpoint changes where dense methods fail.

4. **Computationally efficient:** Only processes a subset of pixels (feature points) rather than every pixel, making it faster than dense methods.

5. **Works with unstructured image sets:** Does not require specific camera configurations; can work with arbitrary views.

**Disadvantages:**

1. **Sparse reconstruction:** Only reconstructs 3D points where features are detected — large textureless regions (walls, sky) produce no points, leaving gaps in the model.

2. **Correspondence errors:** Incorrect matches (outliers) can propagate errors into the 3D reconstruction despite robust estimation.

3. **Texture dependency:** Requires sufficient texture in the scene for feature detection. Fails on uniform or repetitive surfaces.

4. **No surface information:** Produces only a point cloud, not a surface. Additional processing (Poisson reconstruction, Delaunay triangulation) is needed to obtain a mesh.

5. **Repetitive structures:** Scenes with repeated patterns (e.g., windows on a building) cause ambiguous matches, potentially leading to incorrect geometry.

6. **Difficulty with thin structures:** Fine details and thin structures may not have enough distinctive features to be reconstructed.

### (b) Baumgart's Shape from Silhouettes in Voxel Framework [4 marks]

**Baumgart's shape from silhouettes:**

Baumgart (1974) proposed reconstructing a 3D object by intersecting the visual cones (back-projected silhouettes) from multiple viewpoints. The silhouette of an object from a given viewpoint defines a cone of possible 3D locations. The intersection of cones from multiple viewpoints yields the "visual hull" — the tightest convex approximation of the object's shape derivable from silhouettes alone.

**Implementation in a voxel colouring framework:**

1. **Voxel space initialisation:** Discretise the 3D space into a regular grid of voxels. Initially, all voxels are marked as "occupied."

2. **Silhouette extraction:** For each input image, segment the object from the background to produce a binary silhouette mask.

3. **Projection and carving:** For each voxel in the grid:
   - Project the voxel's 3D centre onto each camera's image plane using the known camera parameters (intrinsics and extrinsics).
   - If the projected point falls outside the silhouette in any view, the voxel cannot be part of the object — mark it as "empty" (carve it away).
   - If the projected point falls inside the silhouette in all views, the voxel is retained as potentially occupied.

4. **Result:** The remaining occupied voxels form the visual hull — a volumetric approximation of the object's shape.

### (c) The Visibility Problem and Solutions [6 marks]

**What is the visibility problem:**

The visibility problem in volumetric reconstruction arises because not all voxels are visible from all cameras. A voxel may be occluded by other parts of the object when viewed from certain cameras. If we naively compare the colours of a voxel's projection across all cameras (including cameras where the voxel is actually occluded), we get incorrect colour inconsistency measurements. This leads to erroneously carving away voxels that are actually part of the object, because the colour seen at the projected location in an occluding view belongs to a different surface point.

**Seitz and Dyer's Voxel Colouring (1999):**

Voxel Colouring solves the visibility problem by introducing an **ordinal visibility constraint**. The key insight is:

- Cameras are arranged on one side of the scene (or more precisely, the scene can be layered such that voxels can be processed in a depth order where nearer voxels are resolved before farther ones).
- Voxels are processed in a **front-to-back order** (from nearest to cameras to farthest). This is achieved by requiring cameras be placed such that the depth ordering of voxels is consistent across all views (the "ordinal visibility constraint").
- When evaluating a voxel, all voxels in front of it have already been classified. If a front voxel has been marked occupied, it occludes the current voxel in certain views — those views are excluded from the colour consistency check.
- A voxel is retained if its colour projections are consistent across all views in which it is visible; otherwise it is carved.

**Limitation:** Requires the restrictive camera placement constraint (all cameras on one side).

**Space Carving (Kutulakos and Seitz, 2000):**

Space Carving generalises Voxel Colouring to handle arbitrary camera configurations:

- It removes the ordinal visibility constraint by using an iterative, multi-sweep approach.
- Voxels are processed in multiple passes from different directions (e.g., six axis-aligned sweeps: left-right, right-left, top-bottom, etc.).
- In each sweep, voxels are processed in a depth order relative to the sweep direction. A voxel is carved if its colour projections are inconsistent across views where it is currently estimated to be visible.
- As voxels are carved, previously occluded voxels become visible, potentially revealing new inconsistencies in subsequent sweeps.
- The algorithm iterates until convergence (no more voxels are carved).
- This produces the **photo hull** — the maximal shape consistent with all input images.

### (d) Plant on Turntable: Problems and Solutions [5 marks]

**Problems with the volumetric approach for reconstructing a plant:**

1. **Thin structures:** Plant stems, leaves, and branches are very thin — often sub-voxel in width. The voxel grid resolution may be too coarse to capture these fine details. Increasing resolution leads to enormous memory and computation requirements (cubic scaling).

2. **Complex self-occlusion:** Plants have highly complex geometry with extensive self-occlusion. Many parts of the plant are never simultaneously visible from any pair of cameras, making colour consistency checks unreliable.

3. **Visual hull limitations:** Silhouette-based methods produce the visual hull, which fills in concavities. For a plant with many gaps between leaves, the visual hull would be a solid blob rather than the intricate open structure.

4. **Colour similarity:** Leaves are largely the same colour (green), making colour consistency a poor discriminator — many incorrect voxel configurations may appear colour-consistent.

5. **Transparency and translucency:** Thin leaves may be partially transparent, and backlighting can cause colour variations depending on viewing angle, violating the Lambertian assumption.

6. **Specular reflections:** Waxy leaf surfaces produce view-dependent specular highlights that break colour consistency assumptions.

7. **Motion between captures:** If using a turntable and the plant moves slightly (due to vibration or settling), the images are inconsistent with a single rigid model.

**How to address these problems:**

1. **High-resolution voxel grid with adaptive refinement:** Use an octree structure to adaptively refine resolution only where the object surface lies, reducing memory while capturing thin structures.

2. **Multi-scale approach:** Start with coarse voxels to establish the overall shape, then refine iteratively to capture finer details.

3. **Graph-cut optimisation:** Instead of simple thresholding for consistency, use graph-cut methods that impose spatial coherence and allow for thin structures through appropriate edge weights.

4. **Robust colour consistency measures:** Use normalised cross-correlation or mutual information rather than simple colour difference to handle illumination variation and specular effects.

5. **Many viewpoints:** Capture a dense set of images around the turntable (e.g., every 2-5 degrees) to reduce occlusion problems and provide more visibility constraints.

6. **Structured light or active sensing:** Supplement passive imaging with structured light patterns projected onto the plant to establish correspondences on textureless surfaces and resolve thin structures.

7. **Level-set or mesh-based methods:** Move away from voxels entirely, using implicit surface representations (level sets) that can represent thin structures more naturally and at sub-voxel precision.

## Question 4: Features and Object Detection (20 marks)

### (a) Patch Representations [10 marks]

#### (i) Texture Representation [5 marks]

Texture describes the spatial arrangement of intensity patterns within a patch. Common representations include:

**Filter bank responses (Leung-Malik, Schmid):**
- Convolve the patch with a bank of filters at multiple orientations and scales (e.g., Gabor filters, Gaussian derivatives, Laplacian of Gaussian).
- The filter responses at each pixel (or their statistics — mean, variance) form the texture descriptor.
- Captures information about edges, bars, and spots at different orientations and frequencies.

**Histogram of oriented gradients (within a texture context):**
- Compute gradient orientations within the patch.
- Build a histogram of these orientations, capturing the dominant texture directions.

**Local Binary Patterns (LBP):**
- For each pixel, compare it to its circular neighbours and encode the pattern as a binary number.
- Build a histogram of all LBP codes in the patch.
- Captures micro-texture patterns (edges, corners, flat regions, spots) in a compact, illumination-invariant way.

**Grey-Level Co-occurrence Matrix (GLCM):**
- Compute statistics of co-occurring pixel intensities at specified offsets.
- Extract features such as contrast, correlation, energy, and homogeneity from the matrix.
- Captures second-order texture statistics.

**Textons:**
- Cluster filter responses across a training set into a vocabulary of "textons" (prototypical filter response vectors).
- Represent a patch by the histogram of texton occurrences within it.
- Provides a compact, discriminative texture representation.

The key principle is that texture is characterised by repeated patterns, so statistical summaries of local measurements (rather than raw pixel values) provide invariance to exact spatial location while capturing the essential structure.

#### (ii) Shape Representation [5 marks]

Shape describes the geometric structure and boundaries of objects/regions within a patch. Common representations include:

**Histogram of Oriented Gradients (HOG):**
- Divide the patch into cells (e.g., 8x8 pixels).
- For each cell, compute a histogram of gradient orientations weighted by gradient magnitude.
- Normalise histograms across overlapping blocks for illumination invariance.
- Concatenate all cell histograms to form the descriptor.
- Captures the distribution of edge directions, which encodes shape information (e.g., vertical edges for legs, horizontal edges for shoulders).

**Shape Context:**
- Extract edge points from the patch.
- For each edge point, compute a log-polar histogram of the relative positions of all other edge points.
- This descriptor captures the spatial arrangement of the contour, providing a rich shape representation.
- Invariant to translation; can be made invariant to scale and rotation.

**SIFT descriptor (shape aspects):**
- Although often considered a general-purpose descriptor, SIFT captures shape through its gradient orientation histograms in a 4x4 grid of sub-regions.
- The spatial binning preserves the geometric arrangement of edges around the keypoint.

**Edge/contour-based representations:**
- Extract edges (Canny) within the patch.
- Encode the spatial layout of edges — their positions, orientations, and connectivity.
- Chain codes or curvature descriptors along contours.

**Geometric moments (Hu moments):**
- Compute moments of the binary or greyscale patch.
- Hu's seven invariant moments are invariant to translation, scale, and rotation.
- Capture global shape properties (elongation, symmetry, etc.).

The key distinction from texture is that shape representations preserve spatial structure and geometric relationships, encoding where edges and boundaries occur relative to each other, rather than just summarising their statistical properties.

### (b) Classification [10 marks]

#### (i) Parametric Form of a Linear Classifier [3 marks]

A linear classifier takes the form:

**f(x) = w^T * x + b**

Where:
- **x** is the feature vector (d-dimensional input representing the patch/image)
- **w** is the weight vector (d-dimensional parameter vector, also called the normal to the decision hyperplane)
- **b** is the bias term (scalar offset)
- **f(x)** is the decision function output

Classification rule:
- If f(x) >= 0, classify as positive class (object present)
- If f(x) < 0, classify as negative class (object absent)

The decision boundary is the hyperplane defined by w^T * x + b = 0, which divides the feature space into two half-spaces corresponding to the two classes.

#### (ii) SVM Criterion for Optimal Parameters [3 marks]

In simple words, the Support Vector Machine (SVM) finds the linear decision boundary (hyperplane) that **maximises the margin** — the distance between the decision boundary and the nearest training examples from each class.

Specifically:
- Among all hyperplanes that correctly separate the positive and negative training examples, the SVM selects the one that is as far as possible from the closest data points on both sides.
- These closest points are called **support vectors** — they "support" (define) the position of the optimal hyperplane.
- The intuition is that a large margin provides the best generalisation to unseen data, as the classifier is maximally confident about points near the boundary.

For non-separable data, a soft-margin SVM allows some training points to violate the margin (or even be misclassified), penalised by a parameter C that trades off margin size against training errors.

#### (iii) Evaluating a Trained Classifier for Object Recognition [4 marks]

To evaluate a trained classifier:

**Test set methodology:**
- Use a separate test set (never seen during training) containing positive examples (images with the object) and negative examples (images without the object).
- Run the classifier on all test examples and compare predictions to ground truth labels.

**Key metrics:**

- **Precision** = TP / (TP + FP) — of all detections, what fraction are correct?
- **Recall** = TP / (TP + FN) — of all actual objects, what fraction are detected?
- **Accuracy** = (TP + TN) / (TP + TN + FP + FN) — overall correctness (but can be misleading with imbalanced data).
- **F1-Score** = 2 * (Precision * Recall) / (Precision + Recall) — harmonic mean balancing precision and recall.

**Precision-Recall curve / ROC curve:**
- Vary the decision threshold and plot precision vs. recall (or true positive rate vs. false positive rate).
- Compute Area Under the Curve (AUC) or Average Precision (AP) as a single summary metric.

**Cross-validation:**
- Use k-fold cross-validation if the dataset is limited: partition data into k folds, train on k-1 folds, test on the remaining fold, and average results across all k iterations.
- This provides a more reliable estimate of generalisation performance.

**Confusion matrix:**
- For multi-class recognition, display a matrix showing how often each class is confused with each other class, revealing systematic errors.

**Practical considerations:**
- Ensure the test set is representative of real-world conditions (varied viewpoints, lighting, occlusion, scale).
- Report results on established benchmarks (e.g., PASCAL VOC, ImageNet) for comparability.
- Consider computational speed alongside accuracy for real-time applications.
