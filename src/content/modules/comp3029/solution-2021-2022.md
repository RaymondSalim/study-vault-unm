**How LBP features produce image features for classification:**

1. **Local texture encoding:** Each pixel in the image is assigned an LBP code (0-255 for 8 neighbours), encoding the local texture pattern around that pixel. The LBP captures micro-level structural information such as edges, corners, flat regions, and spots.

2. **Histogram construction:** The image (or a region/block of the image) is divided into cells. For each cell, a histogram of LBP values is computed, counting the frequency of each LBP pattern. This histogram serves as a feature descriptor for that region.

3. **Concatenation for classification:** The histograms from all cells are concatenated to form a single feature vector that captures both local texture information and spatial layout. This feature vector is then fed into a classifier (e.g., SVM, nearest neighbour) for tasks such as face recognition or texture classification.

4. **Properties:** LBP is invariant to monotonic changes in grey level (since it uses binary comparisons), computationally efficient, and captures fine-grained texture information.

### (b) SIFT Features on a Toy Car Image [8 marks]

The SIFT algorithm consists of four major stages. Each stage is reflected in the visual features (coloured circles of varying sizes and orientations) shown on the toy car:

**1. Scale-Space Extrema Detection:**

- The image is convolved with Gaussian filters at multiple scales, producing a scale-space representation. Differences of Gaussians (DoG) are computed between adjacent scales.
- Local maxima and minima in the DoG across both spatial dimensions and scale are identified as candidate keypoints.
- **Reflected in the image:** The circles have varying sizes (radii), indicating that keypoints have been detected at different scales. Larger circles correspond to features detected at coarser scales (larger sigma), while smaller circles correspond to finer-scale features. This shows the algorithm finds stable features across multiple resolutions.

**2. Keypoint Localisation:**

- Candidate keypoints are refined by fitting a 3D quadratic function to the local DoG samples. Sub-pixel and sub-scale accuracy is achieved via interpolation.
- Low-contrast keypoints and edge responses are rejected using thresholding on the DoG value and the ratio of principal curvatures (from the Hessian matrix).
- **Reflected in the image:** Only a subset of stable, well-localised keypoints remain. They appear at distinctive locations on the toy car (corners, high-contrast regions) rather than being uniformly distributed. Keypoints along plain surfaces or weak edges have been eliminated.

**3. Orientation Assignment:**

- For each keypoint, a histogram of gradient orientations is computed in a region around the keypoint (at the appropriate scale).
- The dominant gradient direction is assigned as the keypoint's canonical orientation, providing rotation invariance.
- **Reflected in the image:** Each circle has a line or arrow indicating its assigned orientation. Different keypoints have different orientations depending on the local gradient structure. This ensures that the descriptor is computed relative to this orientation, making it rotationally invariant.

**4. Generating the Keypoint Descriptor:**

- A 16x16 region around each keypoint is divided into 4x4 sub-regions. In each sub-region, an 8-bin gradient orientation histogram is computed (relative to the keypoint orientation).
- This produces a 4x4x8 = 128-dimensional descriptor vector, which is normalised to unit length for illumination invariance.
- **Reflected in the image:** The coloured circles represent distinct feature descriptors. Each keypoint's descriptor encodes the local gradient structure in a rotation- and scale-normalised frame. The variety of colours may represent different descriptor clusters or matching correspondences. The descriptors allow the toy car to be recognised from different viewpoints, scales, and under partial occlusion.

**Summary:** The SIFT visualisation on the toy car shows that features are detected at multiple scales (varying circle sizes), at distinctive locations (keypoint localisation), with assigned orientations (directional indicators), producing robust 128-D descriptors that enable reliable matching and object recognition.

### (a) Homogeneous Representation and Line Through Two Points [6 marks]

**Is P = (3, 4, 5)^T 2D or 3D?**

P = (3, 4, 5)^T is a **2D point** in homogeneous coordinates.

Reasoning: A point in 2D projective space P^2 is represented by a 3-vector (x1, x2, x3)^T. A point in 3D projective space P^3 would be represented by a 4-vector. Since P has 3 components, it represents a point in 2D projective space (i.e., a point in the 2D plane expressed in homogeneous coordinates).

**Inhomogeneous representation of P:**

To convert from homogeneous (x1, x2, x3)^T to inhomogeneous (Cartesian) coordinates, divide by the last coordinate:

P_inhomogeneous = (x1/x3, x2/x3) = (3/5, 4/5) = **(0.6, 0.8)**

**Line through P and Q:**

Given P = (3, 4, 5)^T and Q = (2, 1, 3)^T, both are 2D points in homogeneous coordinates.

In projective geometry, the line l through two points is given by the cross product:

l = P x Q

Computing the cross product:

```
l = P x Q = |  i    j    k  |
            |  3    4    5  |
            |  2    1    3  |
```

l_1 = (4)(3) - (5)(1) = 12 - 5 = 7

l_2 = (5)(2) - (3)(3) = 10 - 9 = 1

l_3 = (3)(1) - (4)(2) = 3 - 8 = -5

**l = (7, 1, -5)^T**

**Verification:** A point lies on a line if l^T * point = 0.

- l^T * P = 7(3) + 1(4) + (-5)(5) = 21 + 4 - 25 = 0 (correct)
- l^T * Q = 7(2) + 1(1) + (-5)(3) = 14 + 1 - 15 = 0 (correct)

The homogeneous representation of the line through P and Q is **l = (7, 1, -5)^T**.

### (b) Design a 3-Layer CNN [7 marks]

**Requirements:**
- Input: RGB image, 256 x 256 x 3
- Output: 64 x 64 x 256
- Allowed operations: 3x3 convolution, 1x1 convolution, 2x2 max pooling
- Must use exactly 3 layers (convolutional/pooling layers)

**Design:**

| Layer | Operation | Input Size | Output Size | Details |
|-------|-----------|-----------|-------------|---------|
| 1 | 3x3 Conv + 2x2 Max Pool | 256x256x3 | 128x128x64 | 64 filters of 3x3, padding=1, stride=1 gives 256x256x64; then 2x2 max pool halves spatial dims |
| 2 | 3x3 Conv + 2x2 Max Pool | 128x128x64 | 64x64x128 | 128 filters of 3x3, padding=1, stride=1 gives 128x128x128; then 2x2 max pool halves spatial dims |
| 3 | 1x1 Conv | 64x64x128 | 64x64x256 | 256 filters of 1x1 maps 128 channels to 256 channels |

**Detailed explanation:**

**Layer 1:** Apply 64 filters of size 3x3 with zero-padding of 1 (to maintain spatial dimensions) to the input 256x256x3 image, producing 256x256x64. Follow with 2x2 max pooling (stride 2) to produce **128x128x64**.

**Layer 2:** Apply 128 filters of size 3x3 with zero-padding of 1 to the 128x128x64 feature map, producing 128x128x128. Follow with 2x2 max pooling (stride 2) to produce **64x64x128**.

**Layer 3:** Apply 256 filters of size 1x1 to the 64x64x128 feature map, producing **64x64x256**.

This achieves the required output dimensions of 64x64 spatially with 256 channels, using only the permitted operations (3x3 conv, 1x1 conv, and 2x2 max pooling).

**Note:** If pooling counts as a separate layer, an alternative design is:

| Layer | Operation | Input Size | Output Size |
|-------|-----------|-----------|-------------|
| 1 | 3x3 Conv (stride 2, pad 1) | 256x256x3 | 128x128x64 |
| 2 | 3x3 Conv (stride 2, pad 1) | 128x128x64 | 64x64x128 |
| 3 | 1x1 Conv | 64x64x128 | 64x64x256 |

Here the stride-2 convolution effectively performs downsampling without a separate pooling layer.

### (b) K-Means vs GMM for Oil Palm Seed Segmentation [10 marks]

**Context:** Segmenting an oil palm seed image (with HSV channels and histograms visible) into three classes: seed body, plumule/radicle, and background.

**K-Means Clustering:**

*How it works:* K-means partitions pixels into K=3 clusters by minimising the within-cluster sum of squared distances. Each pixel is assigned to the nearest centroid, and centroids are iteratively updated.

*Advantages:*
1. **Simple and fast:** Computationally efficient, especially for large images. Linear complexity O(nKt) where t is iterations.
2. **Easy to implement:** Straightforward algorithm with few hyperparameters (only K).
3. **Hard assignments:** Each pixel belongs to exactly one segment, producing clean boundary delineations suitable for seed analysis.
4. **Works well when clusters are spherical:** If the three classes have compact, well-separated colour distributions in HSV space, k-means performs effectively.

*Disadvantages:*
1. **Assumes spherical, equal-sized clusters:** Cannot model elongated or irregularly shaped distributions in colour space. The seed body and background may have very different variances.
2. **Sensitive to initialisation:** Different random initialisations may converge to different solutions (local optima).
3. **Hard boundaries only:** Cannot express uncertainty about pixel assignments near class boundaries (e.g., transition regions between seed body and plumule).
4. **Cannot handle overlapping distributions:** If the HSV histograms of seed body and plumule overlap significantly, k-means will produce poor boundaries.

**Gaussian Mixture Model (GMM):**

*How it works:* GMM models the pixel distribution as a weighted sum of K=3 Gaussian distributions. The EM algorithm iteratively estimates the parameters (means, covariances, mixing weights) and computes soft assignments (responsibilities).

*Advantages:*
1. **Soft assignments:** Provides probability of membership for each pixel, useful for ambiguous regions between seed body and plumule/radicle.
2. **Flexible cluster shapes:** Full covariance matrices allow elliptical clusters, better modelling the potentially elongated distributions in HSV space.
3. **Models variance differences:** Can handle classes with very different spreads (e.g., uniform background vs variable seed body).
4. **Statistical framework:** Provides a principled probabilistic model; can use BIC/AIC for model selection.

*Disadvantages:*
1. **Computationally expensive:** EM is slower than k-means, especially with full covariance matrices in multi-dimensional (HSV) space.
2. **Sensitive to initialisation:** Can converge to local optima; may require multiple restarts.
3. **Overfitting risk:** With full covariance and limited data, GMM may overfit, especially if one class (e.g., plumule) has few pixels.
4. **Assumes Gaussian distribution:** If the actual colour distributions are multimodal or non-Gaussian within a class, a single Gaussian per class is insufficient.

**For this specific application:** GMM is likely more suitable because the three classes (seed body, plumule/radicle, background) likely have different variances and potentially overlapping distributions in HSV space. The soft assignments from GMM are valuable for accurately delineating the small plumule/radicle region from the larger seed body. However, k-means could serve as a quick initial segmentation or as initialisation for GMM.

### (b) Plane Sweep Stereo [10 marks]

**Concept of Plane Sweep Stereo:**

Plane Sweep Stereo is a multi-view stereo reconstruction method that estimates scene depth by sweeping a virtual plane through the scene volume at discrete depth levels and evaluating photo-consistency across multiple views at each depth.

**How it works:**

1. **Define the sweep volume:** A series of fronto-parallel planes are defined at discrete depth values d_1, d_2, ..., d_N between a near and far depth bound. These planes are typically perpendicular to the optical axis of a reference camera.

2. **Project and warp:** For each candidate depth plane d_k, every input image is warped (via a homography) to the reference view as if all scene content lay on that plane. The homography H_i for image i at depth d is computed from the relative camera pose and the plane equation:

   H_i(d) = K_i * (R_i - t_i * n^T / d) * K_ref^(-1)

   where K are intrinsic matrices, R_i and t_i are the relative rotation and translation, and n is the plane normal.

3. **Compute matching cost:** At each pixel in the reference view and each depth d_k, a photo-consistency measure is computed by comparing the warped images. Common measures include:
   - Sum of Absolute Differences (SAD)
   - Normalised Cross-Correlation (NCC)
   - Variance across warped images

4. **Select best depth:** For each pixel, the depth d_k that yields the best photo-consistency score (minimum cost or maximum correlation) is selected as the estimated depth. This produces a depth map for the reference view.

5. **Optional refinement:** The discrete depth map can be refined using sub-pixel interpolation, regularisation, or graph-cut optimisation.

**Does it require known camera parameters?**

**Yes.** Plane Sweep Stereo requires known intrinsic and extrinsic camera parameters for all views. The homographies used to warp images onto candidate planes are computed directly from the camera matrices and the plane equations. Without accurate camera calibration (both intrinsic K and extrinsic [R|t]), the warping would be incorrect and matching would fail.

**Does it require rectified images?**

**No.** Plane Sweep Stereo does **not** require rectified images. This is one of its key advantages over traditional binocular stereo methods. The homography-based warping inherently handles arbitrary camera configurations (convergent, divergent, non-parallel). The method generalises naturally to multiple views with arbitrary poses, unlike epipolar-line-based methods that require rectification. The plane sweep approach implicitly handles the correspondence search by warping all images to a common reference frame at each depth hypothesis.

**Advantages of Plane Sweep Stereo:**
- Naturally extends to multiple (>2) views
- Does not require rectification
- Efficient GPU implementation (parallel for each depth plane)
- Handles wide baselines and arbitrary camera configurations

```

```

**Step 2: Calculate the simple feature (vertical two-rectangle feature).**

The feature is a vertical two-rectangle feature where:
- **Dark-gray region (left column):** column 0 (pixels: 90, 85, 50)
- **Light-gray region (middle + right columns):** columns 1-2 (pixels: 95, 91, 120, 100, 70, 60)

**Step 3: How to generate multiple simple features.**

The Viola-Jones framework generates a large pool of simple Haar-like features by varying:

1. **Feature type:** Different rectangle configurations are used:
   - Two-rectangle features (horizontal or vertical): difference between two adjacent rectangles
   - Three-rectangle features: sum of outer rectangles minus centre rectangle
   - Four-rectangle features: diagonal difference pattern

2. **Position:** Each feature type is placed at every possible position within the detection window (e.g., 24x24 pixels). The top-left corner of the feature is varied across all valid positions.

3. **Scale/Size:** Each feature is varied in width and height (in multiples of the base rectangle size), from minimum size up to the full detection window size.

4. **Exhaustive enumeration:** For a 24x24 detection window, this produces over 160,000 candidate features. Each feature can be computed in constant time O(1) using the integral image (requiring only 6-8 array lookups regardless of feature size).

5. **AdaBoost selection:** From this massive overcomplete set, AdaBoost selects the most discriminative features sequentially, building a strong classifier as a weighted combination of weak classifiers (each based on a single feature with an optimal threshold).

### (b) Pictorial Structure vs Bag-of-Features for Face Detection [10 marks]

**Context:** The image shows a paper bag with face parts (eyes, nose, mouth) scattered in incorrect spatial arrangements, resembling a Picasso-like face.

**Pictorial Structure Model:**

A pictorial structure represents an object as a collection of parts with:
- **Appearance models** for each part (what each part looks like)
- **Spatial/geometric constraints** between parts (spring-like connections defining expected relative positions)

The energy function is: E(L) = Sum_i m_i(l_i) + Sum_(i,j) d_ij(l_i, l_j)

where m_i is the appearance match cost for part i at location l_i, and d_ij penalises deviations from expected spatial relationships between parts i and j.

**Will pictorial structure detect this as a face? NO (or very unlikely).**

Reasoning: While the individual parts (eyes, nose, mouth) would score well on their appearance models (m_i terms), the spatial relationship penalties (d_ij terms) would be very high because:
- The parts are not in their expected relative positions
- Eyes are not above the nose, nose is not above the mouth
- The spatial configuration violates the learned geometric constraints

The total energy would be too high (or equivalently, the detection score too low) to exceed the detection threshold. The pictorial structure model explicitly encodes that a face has specific spatial relationships between parts, and this "Picasso face" violates those constraints.

**Bag-of-Features Model:**

A bag-of-features (BoF) represents an image as an orderless histogram of visual words:
1. Extract local features (e.g., SIFT) from the image
2. Quantise features into visual words using a codebook
3. Build a histogram counting occurrences of each visual word
4. Classify based on the histogram (e.g., using SVM)

**Will bag-of-features detect this as a face? YES (likely).**

Reasoning: The bag-of-features model deliberately ignores spatial relationships. It only counts the presence and frequency of visual patterns (textures, edges, gradients). Since the image contains:
- Eye-like patterns (iris textures, eyelid edges)
- Nose-like textures
- Mouth-like features
- Skin-like colour patches

The histogram of visual words would be similar to that of a normal face image, because all the same visual elements are present -- just in different positions. Since BoF discards all spatial information, the scrambled arrangement does not affect the detection score.

**Summary comparison:**

## Question 7: Motion and Optical Flow [20 marks]

### (a) Aperture Problem [6 marks]

**Context:** An opaque gray square moving towards the top-right. Three apertures are positioned at: (a) bottom edge, (b) interior, (c) right edge (with a circle visible through a dashed line).

**(a) Aperture at bottom edge:**

Through this aperture, the observer sees the bottom horizontal edge of the square moving. The motion **cannot be fully recovered**. Only the **component of motion perpendicular to the edge** (i.e., the vertical/upward component) can be determined. The horizontal component along the edge is ambiguous because the edge is uniform -- any point on the edge looks identical to any other point.

**Recovered component:** Normal (vertical/upward) component only.

**(b) Aperture at interior:**

Through this aperture, the observer sees a uniform gray region with no features, edges, or texture. **No motion can be recovered at all.** There is no spatial gradient (Ix = Iy = 0), so the optical flow equation provides no constraint. The entire region is homogeneous, making it impossible to detect any motion component.

**Recovered component:** None (completely ambiguous).

**(c) Aperture at right edge with a circle visible:**

Through this aperture, the observer sees the right vertical edge of the square AND a distinct feature (circle). The circle provides a trackable feature point with gradients in both x and y directions. Therefore, **the full 2D motion can be recovered.** The circle breaks the aperture problem because it provides constraints in multiple orientations, allowing both horizontal (rightward) and vertical (upward) components to be determined.

**Recovered component:** Full 2D motion (both horizontal and vertical components).

**Summary:**
- (a) Bottom edge: Only normal component (upward motion) -- aperture problem
- (b) Interior: No motion recoverable -- blank aperture problem
- (c) Right edge with circle: Full motion recoverable -- feature resolves ambiguity

**Lucas-Kanade Method Explanation:**

The optical flow constraint equation for each pixel is:

Ix * u + Iy * v + It = 0

where u = dx/dt (horizontal flow) and v = dy/dt (vertical flow). This single equation has two unknowns (u, v), so it is underdetermined for a single pixel (the aperture problem).

**Lucas-Kanade assumptions:**
1. **Brightness constancy:** Pixel intensity does not change as it moves between frames.
2. **Small motion:** The displacement between frames is small enough for linearisation.
3. **Spatial coherence:** All pixels within a local neighbourhood (the 3x3 window) share the same flow vector (u, v).

**Formulation:** Given n pixels in the window, we have n equations with 2 unknowns:

```
Ix_1 * u + Iy_1 * v = -It_1
Ix_2 * u + Iy_2 * v = -It_2
...
Ix_n * u + Iy_n * v = -It_n
```

In matrix form: **A * [u; v] = b**

Where:
```
A = | Ix_1  Iy_1 |       b = | -It_1 |
    | Ix_2  Iy_2 |           | -It_2 |
    | ...   ...  |           |  ...  |
    | Ix_n  Iy_n |           | -It_n |
```

This is an overdetermined system (9 equations, 2 unknowns). The least-squares solution is obtained via the normal equations:

**(A^T A) [u; v] = A^T b**

```
A^T A = | Sum(Ix^2)    Sum(Ix*Iy) |
        | Sum(Ix*Iy)   Sum(Iy^2)  |

A^T b = | -Sum(Ix*It) |
        | -Sum(Iy*It) |
```

The solution is:

**[u; v] = (A^T A)^(-1) * A^T b**

This yields the optical flow vector (u, v) at the central pixel, provided that A^T A is invertible (i.e., has two large eigenvalues), which requires sufficient gradient variation in the window -- essentially the same condition as the Harris corner detector. The method fails in uniform regions (both eigenvalues near zero) or along edges (one eigenvalue near zero), which corresponds to the aperture problem.

For our computed values, A^T A would be:

- Sum(Ix^2) = 0 + 0 + 0 + 0 + 0 + 1 + 0 + 0 + 4 = 5
- Sum(Iy^2) = 1 + 1 + 1 + 0 + 0 + 0 + 1 + 1 + 1 = 6
- Sum(Ix*Iy) = 0 + 0 + 0 + 0 + 0 + 0 + 0 + 0 + 2 = 2

- Sum(Ix*It) = 0 + 0 + 0 + 0 + 0 + 20 + 0 + 0 + 44 = 64
- Sum(Iy*It) = -13 + (-18) + (-18) + 0 + 0 + 0 + 17 + 23 + 22 = 13

```
A^T A = | 5  2 |     A^T b = | -64 |
        | 2  6 |             | -13 |
```

The flow (u, v) would be obtained by solving this 2x2 system, yielding the estimated motion at the central pixel.
