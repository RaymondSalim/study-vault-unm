## Question 1: Stereo Correspondence (25 marks)

### (a) Sparse vs Dense Disparity Maps [4 marks]

**Sparse disparity maps** arise when:
- Only distinctive feature points (corners, edges, high-texture regions) are matched between images
- Feature-based matching methods (e.g., SIFT, Harris corners) are used
- The scene contains large textureless regions where no reliable correspondences can be found

**Dense disparity maps** arise when:
- Every pixel (or nearly every pixel) in the image is assigned a disparity value
- Block matching or area-based correlation methods are applied exhaustively across the image
- The scene is richly textured, providing sufficient information for pixel-wise matching

**Key distinction:** Sparse maps result from matching only salient features (fast but incomplete), while dense maps attempt to assign disparity to all pixels (computationally expensive but complete). Sparse maps are typical of feature-based stereo; dense maps are typical of area-based or energy-minimisation stereo.

### (b) Dense Disparity Map from Sparse Correspondences [7 marks]

**Drawing a plausible dense disparity map:**

Given sparse correspondences at features 1, 2, and 3, a plausible dense map would:
- Assign known disparity values at the matched feature locations
- Interpolate smoothly between known points
- Maintain disparity discontinuities at likely object boundaries

**Strategies for generating dense maps from sparse correspondences:**

1. **Surface interpolation:** Fit a smooth surface (e.g., thin-plate spline) through the known disparity points. This assumes the scene consists of smooth surfaces.

2. **Delaunay triangulation:** Triangulate the sparse feature points and linearly interpolate disparity within each triangle. This preserves discontinuities along triangle edges aligned with object boundaries.

3. **Propagation-based methods:** Use the sparse matches as seeds and propagate disparity values outward using local correlation matching, guided by intensity edges in the reference image.

4. **Regularisation / energy minimisation:** Formulate as an optimisation problem with a data term (fidelity to known matches) and a smoothness term (penalising disparity changes except at edges). Solve using graph cuts or belief propagation.

5. **Edge-guided interpolation:** Use intensity edges from the original image as constraints — disparity discontinuities should align with intensity edges, while disparity varies smoothly within homogeneous regions.

### (c) Four Main Steps for Computing Depth Maps [6 marks]

1. **Camera calibration:** Determine the intrinsic parameters (focal length, principal point, lens distortion) and extrinsic parameters (relative position and orientation) of the stereo camera pair. This establishes the geometric relationship between the two views.

2. **Rectification:** Transform both images so that epipolar lines become horizontal scanlines. After rectification, corresponding points lie on the same row in both images, reducing the search for correspondences from 2D to 1D.

3. **Correspondence/matching:** For each pixel (or feature) in the left image, find the corresponding point in the right image along the epipolar line. Methods include block matching (SSD, NCC), feature matching, or global optimisation (dynamic programming, graph cuts).

4. **Triangulation (depth computation):** Given a pair of corresponding points and the known camera geometry, compute the 3D position by triangulation. The disparity d = x_L - x_R is inversely proportional to depth: Z = (f * B) / d, where f is focal length and B is the baseline.

### (d) Artifacts in Disparity Maps [4 marks]

**Nature of artifacts:**
- Incorrect disparity values appearing as "speckle noise" or isolated erroneous patches
- Blurred or poorly localised object boundaries (foreground "bleeding" into background)
- Missing data in occluded regions (areas visible in one camera but not the other)
- Streaking along scanlines (from left-right matching inconsistencies)

**Causes:**
- Textureless regions providing insufficient matching evidence
- Repetitive patterns causing ambiguous matches
- Occlusions where no true correspondence exists
- Specular reflections appearing different in each view

**Solutions for cleaning up:**
- **Median filtering:** Removes isolated speckle errors while preserving edges
- **Left-right consistency check:** Compute disparity in both directions and reject pixels where forward and backward disparities disagree
- **Weighted median or bilateral filtering:** Smooths disparity while preserving edges guided by intensity
- **Graph-cut or MRF-based regularisation:** Enforces spatial coherence through global optimisation with smoothness priors

### (e) Two Important Assumptions [4 marks]

1. **Brightness constancy (photometric consistency):** The same scene point produces similar intensity/colour values in both images. This underpins all correlation-based matching methods. Without this assumption, measuring similarity between image patches would be meaningless. Violated by specular surfaces, differing exposure, or different viewpoints on non-Lambertian surfaces.

2. **Smoothness / spatial coherence:** Disparity varies smoothly across the image except at object boundaries. Neighbouring pixels tend to belong to the same surface and hence have similar disparities. This assumption enables interpolation of sparse matches, regularisation of dense methods, and helps resolve ambiguities by preferring solutions where nearby pixels have consistent depth values.

## Question 2: Segmentation (25 marks)

### (a) Histogram Peak Clustering — Is It Region-Based? [4 marks]

**Yes, this is a region-based segmentation method.** Justification:

A region-based segmentation method groups pixels into regions based on shared properties (homogeneity criteria). Clustering pixels according to distinctive peaks in the intensity histogram satisfies this definition because:

1. **Grouping by similarity:** Pixels are assigned to clusters (regions) based on their intensity values falling near a histogram peak — a homogeneity criterion.
2. **Implicit spatial coherence:** Connected components of pixels assigned to the same cluster form regions.
3. **Contrast with edge-based:** Edge-based methods find boundaries first, then infer regions. This method directly classifies pixels into groups without explicitly detecting edges.

However, one could argue it is a **weak** region-based method because it uses only intensity (no spatial information). Pure histogram thresholding may produce fragmented, non-connected regions. True region-based methods (e.g., region growing) explicitly enforce spatial connectivity.

### (b) Detecting Unruly Crowd Behaviour Using Segmentation [5 marks]

**System Design:**

1. **Background subtraction / temporal segmentation:** Segment the scene into static background and dynamic foreground (the crowd). Use a Gaussian mixture model for the background to handle lighting changes.

2. **Crowd density segmentation:** Segment the foreground crowd region by density. In densely packed areas, individual people cannot be resolved, so treat connected blobs as crowd segments. Compute crowd density as the ratio of foreground pixels to total area in each segment.

3. **Motion-based segmentation:** Segment the crowd into regions of coherent motion using optical flow. Normal behaviour shows smooth, uniform flow (e.g., all moving toward stage). Unruly behaviour produces:
   - Chaotic, multi-directional flow segments (mosh pit, fighting)
   - Sudden high-speed motion segments (stampede, people fleeing)
   - Rapidly growing/shrinking segments (crowd surge)

4. **Temporal change in segments:** Monitor how segments evolve over time. Alarm triggers:
   - A previously coherent segment fragments into many small segments (crowd breaking apart)
   - Sudden increase in number of motion-direction segments
   - Density segments exceeding a dangerous threshold (crush risk)

5. **Decision logic:** Flag behaviour as unruly when segmentation metrics exceed thresholds — e.g., motion entropy across segments is high, segment boundaries are unstable, or segment density exceeds safety limits.

### (c) Region Growing and Seed Initialisation Problems [4 marks]

**Region Growing Approach:**

1. Select seed pixels (starting points for regions)
2. Examine neighbouring pixels of each seed
3. If a neighbour satisfies the homogeneity criterion (e.g., intensity difference below threshold), add it to the region
4. Repeat from step 2 using newly added pixels as new seeds
5. Stop when no more pixels can be added to any region

**Seed Initialisation Problems:**

- **Sensitivity to seed placement:** If seeds are placed in non-representative locations (e.g., on a boundary, in a noisy pixel, or in a transition zone), the resulting region may be incorrect or may merge multiple distinct regions.
- **Under-segmentation:** Too few seeds means some regions are never initialised and get absorbed into neighbouring regions.
- **Over-segmentation:** Too many seeds (especially close together in the same true region) may produce redundant small regions that need merging.
- **Non-deterministic results:** Random seed placement produces different segmentations on each run, making results unreproducible.
- **Solution approaches:** Use local minima/maxima of intensity as seeds, place seeds on a regular grid, or use histogram peaks to identify likely region intensities and select seeds from those pixels.

## Question 3: Motion Analysis and Optic Flow (25 marks)

### (a) Camouflage and Occlusion Problems in Frame Differencing [8 marks]

**Simple Motion Detection by Frame Differences:**

The basic method computes |I(x,y,t) - I(x,y,t-1)| > threshold for each pixel. Pixels exceeding the threshold are classified as moving.

**Camouflage Problem:**

- **Explanation:** When a moving object has similar intensity/colour to the background it moves against, the frame difference at pixels where the object currently resides may be zero or near-zero. The object becomes "invisible" to the detector because I(x,y,t) approximately equals I(x,y,t-1) even though the pixel now shows the object instead of background.
- **Example:** A grey car moving against a grey road surface; a person wearing green walking past foliage.
- **Solution:** Use a **background model** (e.g., running average or Gaussian mixture model of the background) instead of simple frame-to-frame differencing. Compare each frame against the learned background model: |I(x,y,t) - B(x,y)| > threshold. Since the background model represents the true static scene, a camouflaged object will still differ from it even if consecutive frames look similar. Alternatively, use **three-frame differencing** — combining |F(t)-F(t-1)| AND |F(t)-F(t+1)| to capture both leading and trailing edges of motion.

**Occlusion Problem:**

- **Explanation:** Frame differencing detects changes at both the position an object has left (uncovered background) and the position it has moved to (newly covered background). This produces a "ghost" or "halo" at the previous position and a detected region at the new position. Additionally, when one object passes behind another, the occluded object disappears from detection, and the system cannot determine whether it has stopped or been hidden.
- **Example:** A person walks behind a pillar — the frame difference shows motion at the leading and trailing edges but nothing in the occluded region. The object's continuity is lost.
- **Solution:** Use **object tracking with a predictive model** (e.g., Kalman filter). When an object becomes occluded, the tracker maintains a predicted position based on previous velocity and trajectory. When the object reappears, it can be re-associated with the existing track. For the ghost/halo problem, use **background subtraction** with a maintained background model rather than frame differencing, which correctly identifies only the current object location (not where it was).

### (b) Image Pyramids in Optical Flow [8 marks]

**Why pyramids are needed:**

Optical flow computation (e.g., Lucas-Kanade) relies on the assumption that motion between frames is small (typically < 1 pixel for differential methods). When objects move by many pixels between frames, the linearisation of the brightness constancy equation breaks down, and the gradient-based computation gives incorrect results. Image pyramids address this **large displacement problem**.

**How pyramids work for optical flow:**

1. **Pyramid construction:** Build a multi-resolution pyramid for each frame by repeatedly smoothing (Gaussian blur) and downsampling (factor of 2) the image. Level 0 is the original resolution; level L is the coarsest.

2. **Coarse-to-fine estimation:**
   - At the coarsest level L, the large physical motion appears as a small motion (divided by 2^L). Compute optical flow at this level — the small-motion assumption now holds.
   - Propagate the computed flow to the next finer level (level L-1) by upsampling the flow field and multiplying by 2 (to account for the resolution change).
   - At level L-1, use the propagated flow as an initial estimate. Warp the second image according to this estimate, then compute the residual flow (remaining small motion).
   - Add the residual flow to the propagated flow.
   - Repeat until reaching the finest level (level 0).

3. **Result:** The final flow at level 0 accurately captures large motions that would have been impossible to estimate directly at full resolution.

**Benefits:**
- Handles large displacements (many pixels of motion)
- Maintains sub-pixel accuracy at fine levels
- Computationally efficient (coarse levels have far fewer pixels)
- Provides inherent regularisation through the smoothing at coarse levels

### (c) Optical Flow at Object Boundaries [6 marks]

**The problem:** At object boundaries, neighbouring pixels belong to different objects moving in different directions. Standard optical flow smoothness constraints force flow vectors to be similar to their neighbours, causing:
- Incorrect flow estimates at boundaries (averaged between foreground and background motion)
- Blurred motion boundaries rather than sharp discontinuities

**Forward and backward flows:**

- **Forward flow:** Computed from frame t to frame t+1. For each pixel in frame t, find where it moves to in frame t+1.
- **Backward flow:** Computed from frame t+1 to frame t. For each pixel in frame t+1, find where it came from in frame t.
- **Inconsistency detection:** At occlusion boundaries, forward and backward flows are inconsistent. If pixel p has forward flow f, then the pixel at position p+f should have backward flow -f. Where this fails, we have identified a motion boundary or occlusion.
- This consistency check helps identify where boundaries lie and where flow should NOT be smoothed across.

**Anisotropic diffusion:**

- Standard (isotropic) smoothing of the flow field applies equal smoothing in all directions, blurring motion boundaries.
- **Anisotropic diffusion** applies direction-dependent smoothing:
  - Strong smoothing along object surfaces (parallel to boundaries) where flow should be consistent
  - Weak or no smoothing across boundaries (perpendicular to edges) where flow discontinuities are expected
- The diffusion tensor is guided by image edges (intensity gradients) or flow gradients. Where a strong edge is detected, smoothing perpendicular to that edge is suppressed.
- This preserves sharp flow discontinuities at object boundaries while still regularising flow in smooth regions.

### (d) Three Examples: Optical Flow Not Equal to Motion Field [3 marks]

1. **Rotating uniform sphere:** A textured sphere rotating about its axis has a genuine motion field (every surface point moves), but a perfectly uniform sphere produces zero optical flow because no intensity changes are observed — the image looks identical despite the motion.

2. **Stationary scene with moving light source:** A fixed scene illuminated by a moving light source produces changing image intensities (moving shadows and highlights), generating optical flow even though no objects are actually moving. The motion field is zero everywhere.

3. **Flickering or pulsing object:** A stationary object that changes brightness (e.g., a flashing light, or a screen changing content) generates apparent optical flow in the radial direction from its centre (expansion/contraction pattern) due to intensity changes, despite having zero motion field.

## Question 4: Evaluation and Tracking (25 marks)

### (a) Evaluating Segmentation in Context of Larger Systems [6 marks]

**Approach: Task-based (extrinsic) evaluation**

Since segmentation is rarely an end in itself, we evaluate how well it serves the downstream task rather than comparing to a "ground truth" segmentation.

**Example:** A pedestrian detection system for autonomous driving:
- The system pipeline: video input → segmentation → candidate region extraction → classifier → pedestrian detection output
- The segmentation step separates potential pedestrian regions from background

**Evaluation approach:**

1. **Define the end-task metric:** Pedestrian detection accuracy (precision, recall, F1-score).
2. **Vary the segmentation algorithm** (or its parameters) while keeping all other pipeline components fixed.
3. **Measure end-task performance** for each segmentation variant.
4. **The best segmentation** is the one that maximises downstream detection performance, even if it does not perfectly delineate object boundaries.
5. **Rationale:** A segmentation that slightly over-segments but never misses a pedestrian region is better for this task than one that produces aesthetically pleasing boundaries but occasionally fails to separate a pedestrian from background.

This avoids the subjective problem of defining "correct" segmentation and directly measures utility.

### (b) Earth Mover's Distance (EMD) [5 marks]

**Setup:** Region A has darker pixels concentrated on the left side; Region B has darker pixels concentrated on the right side. We model each region as a distribution of "mass" (pixel intensity).

**Computing dist(A, B):**

The EMD measures the minimum "work" (mass x distance) needed to transform one distribution into another.

To transform A into B, we need to move the dark pixel mass from the left side to the right side. If the grid is, say, 4 cells wide and the mass needs to move approximately 3 cells to the right:

- EMD(A, B) = sum of (mass moved x distance moved)
- If total dark mass = 1 (normalised), and average distance to move = 3 cells
- dist(A, B) = 1 x 3 = 3 (in grid units)

**Computing dist(B, A):**

To transform B into A, we move dark mass from right to left — same amount of mass, same distance, just opposite direction.

- dist(B, A) = 1 x 3 = 3

**Result:** dist(A, B) = dist(B, A). The EMD is symmetric (it is a true metric), so the distance is the same regardless of direction. This follows from the fact that the optimal transport plan in one direction has the same total cost as in the reverse direction.

### (c) EMD for Evaluating Segmentation [6 marks]

The Earth Mover's Distance can evaluate segmentation algorithms by comparing a computed segmentation to ground truth:

1. **Representation:** Represent both the computed segmentation and the ground truth as distributions. Each segment boundary can be encoded as a distribution of edge pixels, or each region as a spatial distribution of pixels belonging to that label.

2. **Boundary-based evaluation:** Convert segmentation boundaries into distributions (histograms of boundary pixel locations). The EMD between the computed boundary distribution and ground truth boundary distribution measures how far boundaries need to "move" to match ground truth.

3. **Advantages over pixel-wise metrics:**
   - **Tolerance to small shifts:** A segmentation boundary that is 1 pixel off receives a small EMD, whereas pixel-wise accuracy harshly penalises it. EMD reflects perceptual quality better.
   - **Partial credit:** A nearly-correct segmentation receives a score proportional to how "close" it is, rather than a binary correct/incorrect.
   - **Handles different granularities:** Can compare over-segmented results to coarser ground truth meaningfully.

4. **Practical use:** Compute EMD between computed and ground truth for multiple test images. Lower average EMD indicates a better segmentation algorithm. Can also compare different parameter settings of the same algorithm.

### (d) Aperture Problem and Tracking [3 marks]

**Aperture problem:** When observing motion through a small aperture (local window), only the component of motion perpendicular to an edge can be determined. Motion parallel to the edge is invisible because the pattern looks identical when shifted along the edge direction.

**Relation to tracking:**
- When tracking objects using local template matching or optical flow within a window, the aperture problem means that features lying on straight edges (like the side of a car) cannot be tracked reliably — they can appear to "slide" along the edge.
- Only features with variation in multiple directions (corners, texture) can be tracked unambiguously.
- This is why feature-based trackers (e.g., KLT tracker) specifically select "good features to track" — points where the local autocorrelation matrix has two large eigenvalues (corners), ensuring motion is fully determined within the tracking window.

### (e) Template Update Problem and Solution [5 marks]

**The problem:**

When tracking an object over time using template matching, the object's appearance changes due to:
- Rotation, scale changes, or viewpoint changes
- Illumination variations
- Non-rigid deformations (facial expressions, articulated motion)

If the template is never updated, it gradually becomes less similar to the object's current appearance, leading to tracking failure (template "drift" or loss of lock).

If the template is updated every frame (replacing it with the current best match), small errors accumulate over time. Each update introduces slight inaccuracies, and the template gradually "drifts" away from the true object appearance, eventually locking onto background or a different object. This is the **template drift problem**.

**A simple solution — combined template approach:**

Maintain two templates:
1. **Fixed template:** The original template from the first frame (never updated). This prevents unbounded drift.
2. **Dynamic template:** Updated periodically to reflect the object's current appearance.

**Tracking procedure:**
- Match both templates against the search region
- Accept a match only if both templates agree (or use a weighted combination)
- Update the dynamic template only when the fixed template also produces a good match (confidence is high)
- If the dynamic template drifts too far from the fixed template, reset it

This balances adaptability (handling appearance change) with stability (preventing drift). The fixed template acts as an "anchor" that catches drift before it becomes catastrophic.

## Question 5: Hidden Markov Models (25 marks)

### (a) Three Main Functions of HMMs [5 marks]

The three fundamental problems solved by HMMs:

1. **Evaluation (Likelihood computation):** Given a model lambda = (A, B, pi) and an observation sequence O = (o1, o2, ..., oT), compute P(O|lambda) — the probability that the model generated the observations. Solved efficiently by the **Forward algorithm**.

2. **Decoding (Optimal state sequence):** Given a model and an observation sequence, find the most likely sequence of hidden states Q = (q1, q2, ..., qT) that produced the observations. Solved by the **Viterbi algorithm**.

3. **Learning (Parameter estimation):** Given observation sequences (training data), find the model parameters (A, B, pi) that maximise the likelihood of the observations. Solved by the **Baum-Welch algorithm** (a form of Expectation-Maximisation).

**Real-world example — Speech Recognition:**

- **States:** Phonemes (hidden speech sounds like /k/, /ae/, /t/ for "cat")
- **Observations:** Acoustic feature vectors (MFCCs) extracted from audio frames
- **Evaluation:** Given a trained HMM for the word "cat" and an audio observation sequence, compute the probability that this audio is the word "cat". Compare with HMMs for other words to recognise speech.
- **Decoding:** Given an utterance, determine the most likely sequence of phonemes (useful for phonetic transcription).
- **Learning:** Train HMMs from labelled audio examples of each word.

### (b) Simple Method for Computing P(O|lambda) [8 marks]

**The direct (brute-force) enumeration method:**

To compute P(O|lambda) for observation sequence O = (o1, o2, ..., oT) with N hidden states:

**Principle:** Enumerate ALL possible state sequences of length T, compute the probability of each generating the observations, and sum them.

**Procedure:**

1. Consider every possible state sequence Q = (q1, q2, ..., qT). There are N^T such sequences.

2. For each state sequence Q, compute:

   P(O, Q | lambda) = pi(q1) * b_{q1}(o1) * a_{q1,q2} * b_{q2}(o2) * a_{q2,q3} * b_{q3}(o3) * ... * a_{q_{T-1},q_T} * b_{q_T}(o_T)

   Where:
   - pi(q1) = initial probability of starting in state q1
   - b_{qi}(oi) = emission probability of observing oi in state qi
   - a_{qi,qj} = transition probability from state qi to state qj

3. Sum over all possible state sequences:

   P(O | lambda) = SUM over all Q of P(O, Q | lambda)

   = SUM_{q1,q2,...,qT} [ pi(q1) * product_{t=1}^{T} b_{qt}(ot) * product_{t=1}^{T-1} a_{qt,qt+1} ]

**Computational complexity:** O(N^T * T)

- There are N^T possible state sequences
- Each requires O(T) multiplications
- For even moderate N and T, this is astronomically expensive

**Example:** With N=5 states and T=100 time steps: 5^100 approximately equals 10^70 sequences to evaluate — completely intractable.

**This is why the Forward algorithm exists:** It exploits the Markov property to compute the same result in O(N^2 * T) time using dynamic programming, by reusing intermediate computations.

### (c) Detecting Abnormal Behaviour at Train Station [5 marks]

**Setup:** Model normal activities at a train station where people: queue, buy tickets, walk to train, walk from train.

**HMM Design:**

1. **Define hidden states** corresponding to normal activities:
   - S1: Walking to ticket machine / counter
   - S2: Queuing
   - S3: Buying ticket (interacting with machine/counter)
   - S4: Walking to platform
   - S5: Waiting on platform
   - S6: Walking from train toward exit

2. **Define observations:** Extract visual features from surveillance video:
   - Person's position (zone in the station)
   - Speed of movement (stationary, walking, running)
   - Direction of movement
   - Duration in current zone

3. **Training:** Record normal behaviour sequences over many days. Train the HMM using Baum-Welch to learn:
   - Transition probabilities A (e.g., high probability of Queue → Buy Ticket → Walk to Platform)
   - Emission probabilities B (e.g., in "Queuing" state, person is mostly stationary in the ticket zone)
   - Normal durations in each state

4. **Abnormality detection:** For a new observed sequence O of a person's behaviour:
   - Compute P(O | lambda_normal) using the Forward algorithm
   - If this probability falls below a threshold, the behaviour is abnormal
   - Examples of detected anomalies: running in the wrong direction, spending too long in an unusual area, skipping the ticket-buying step, entering restricted areas

5. **Refinement:** Train separate HMMs for specific normal patterns (arriving passenger, departing passenger, staff). Low likelihood under ALL normal models indicates truly abnormal behaviour.

### (d) Hand Gesture Recognition Using HMMs [7 marks]

**Complete System Description:**

**1. Image Acquisition and Preprocessing:**
- Capture video of hand gestures using a camera
- Apply skin colour detection (using a colour model in HSV or YCbCr space) to segment the hand from the background
- Alternative: use depth camera (Kinect) for robust hand segmentation

**2. Hand Region Extraction:**
- Apply connected component analysis to find the largest skin-coloured blob
- Compute bounding box around the hand region
- Normalise for scale (resize to fixed dimensions)

**3. Feature Extraction (Observations for HMM):**

For each frame, extract features describing the hand's:
- **Shape:** Hu moments, Fourier descriptors of the hand contour, or finger count
- **Position:** Centroid coordinates relative to a reference point (e.g., face or body centre)
- **Motion:** Displacement vector between consecutive frames (dx, dy)
- **Orientation:** Angle of the principal axis of the hand blob

Quantise these continuous features into discrete symbols using vector quantisation (codebook of K prototype feature vectors), producing a discrete observation symbol per frame.

**4. HMM Architecture:**

- Train one HMM per gesture class (e.g., HMM for "wave", HMM for "point", HMM for "circle")
- Each HMM has:
  - States representing temporal phases of the gesture (e.g., for "wave": hand raised, hand moving left, hand moving right, hand lowered)
  - Left-to-right topology (gestures progress forward in time)
  - Emission probabilities linking states to observation symbols

**5. Training:**
- Collect multiple examples of each gesture from different people
- For each gesture class, train its HMM using Baum-Welch on the corresponding observation sequences
- Optimise number of states via cross-validation

**6. Recognition:**
- Given a new gesture observation sequence O:
- Compute P(O | lambda_g) for each gesture model g using the Forward algorithm
- Classify the gesture as: argmax_g P(O | lambda_g)
- Apply a rejection threshold: if all likelihoods are below threshold, classify as "unknown gesture"

**7. Temporal Segmentation (for continuous gestures):**
- Use a sliding window or Viterbi decoding over a concatenated model to segment continuous gesture streams into individual gestures
