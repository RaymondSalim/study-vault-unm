## Question 1: Segmentation (25 marks)

### (a) Ill-posed problems and segmentation [3 marks]

An **ill-posed problem** is one that violates at least one of Hadamard's conditions for well-posedness:

1. A solution must **exist**.
2. The solution must be **unique**.
3. The solution must depend **continuously** on the input data (stability).

**Is segmentation ill-posed?** Yes. Segmentation is ill-posed because:

- There is no unique correct segmentation for a given image. Different humans will segment the same image differently depending on context, purpose, and interpretation.
- Small changes in intensity (e.g., noise) can lead to dramatically different segmentation results, violating the stability condition.
- Without additional constraints or prior knowledge, the problem is under-determined: there are infinitely many valid ways to partition an image into regions.

### (b) K-means clustering on S = {1, 1, 5, 6, 7} [6 marks]

#### k = 1

With k=1, there is only one cluster containing all points.

**Centroid:** (1 + 1 + 5 + 6 + 7) / 5 = 20 / 5 = **4.0**

**Sum of Squared Errors (SSE):**

### (c) Split and merge approach [5 marks]

**Description:**

The split and merge algorithm segments an image using a quad-tree structure:

1. **Split phase:** Start with the entire image as one region. If a region does not satisfy a homogeneity predicate P (e.g., variance below a threshold), split it into four quadrants. Recursively split non-homogeneous regions.
2. **Merge phase:** After splitting, examine adjacent regions. If two adjacent regions together satisfy the homogeneity predicate P, merge them into one region.

**Strengths:**
- Produces a complete segmentation (every pixel is assigned to exactly one region).
- Can handle varying levels of detail in different parts of the image.
- The hierarchical quad-tree structure is computationally efficient.
- Naturally handles regions of different sizes.
- Combining split and merge overcomes limitations of using either alone.

**Weaknesses:**
- Produces blocky boundaries due to the quad-tree structure (boundaries follow rectangular grid lines from the splitting phase, not actual object edges).
- Results depend heavily on the choice of homogeneity predicate P and threshold.
- The merge phase can only merge adjacent quad-tree regions; non-adjacent regions of the same type may remain separate.
- Sensitive to the starting level of the quad-tree.
- Cannot represent diagonal or curved boundaries naturally.

### (d) Parallels between region growing and k-means [4 marks]

| Aspect | Region Growing | K-means |
|--------|---------------|---------|
| **Seeds/Centroids** | Starts with seed pixels | Starts with k initial centroids |
| **Growth/Assignment** | Pixels are added to regions based on similarity to seed/region mean | Points are assigned to clusters based on distance to centroid |
| **Iterative refinement** | Region statistics (mean intensity) update as pixels are added | Centroids are recalculated after each assignment step |
| **Convergence** | Terminates when no more pixels can be added to any region | Terminates when centroids no longer change |
| **Similarity criterion** | Uses a threshold on difference from region mean | Uses minimum distance to centroid |
| **Sensitivity to initialisation** | Results depend on seed selection | Results depend on initial centroid placement |

Both are greedy iterative algorithms that start from initial estimates, grow/refine clusters based on similarity measures, and converge to a local optimum. Both partition the data space completely. The key difference is that region growing is spatially constrained (only adjacent pixels are considered) while k-means operates purely in feature space.

### (e) Graph cuts segmentation [7 marks]

**Given matrix:**

```
10  12  50  55
15  12  53  55
20  20  55  60
25  25  60  70
```

**Converting to a graph:**

Each pixel becomes a node in the graph. Edges connect adjacent pixels (4-connectivity or 8-connectivity). Edge weights represent the dissimilarity between adjacent pixels, typically:

w(i,j) = exp(-|I(i) - I(j)|^2 / 2sigma^2) or simply w(i,j) = |I(i) - I(j)|

Using simple absolute difference as edge weights:

**Horizontal edges (selected examples):**
- w(10,12) = |10-12| = 2
- w(12,50) = |12-50| = 38
- w(50,55) = |50-55| = 5

**Vertical edges (selected examples):**
- w(10,15) = |10-15| = 5
- w(50,53) = |50-53| = 3

**Three distinct cuts:**

**Cut 1 (Vertical cut between columns 2 and 3):**
Separates {10,12,15,12,20,20,25,25} from {50,55,53,55,55,60,60,70}

Edges cut: (12,50), (12,53), (20,55), (25,60)
Total weight = 38 + 41 + 35 + 35 = **149**

This is a natural segmentation boundary since the high weights indicate strong intensity differences (dark region vs. bright region).

**Cut 2 (Horizontal cut between rows 2 and 3):**
Separates {10,12,50,55,15,12,53,55} from {20,20,55,60,25,25,60,70}

Edges cut: (15,20), (12,20), (53,55), (55,60)
Total weight = 5 + 8 + 2 + 5 = **20**

This is a poor segmentation: it separates pixels that are relatively similar (low cut weight does not indicate a meaningful boundary in the min-cut formulation -- here low weights mean the pixels are similar, which means this cut is actually cheap but nonsensical from a segmentation perspective).

**Cut 3 (Isolate single pixel [70]):**
Separate the bottom-right pixel from the rest.

Edges cut: (60,70), (60,70)
Total weight = 10 + 10 = **20**

This is a degenerate cut that isolates a single pixel -- low cost but meaningless.

**Relating weights to segmentation quality:**

In graph cuts, a minimum cut partitions the graph by minimizing the total edge weight across the cut. High edge weights between two pixels indicate they are dissimilar, so cutting these edges corresponds to placing a boundary between different regions. Cut 1 has the highest total weight but gives the best segmentation since it separates genuinely different regions.

**Motivation for Normalized Cuts:**

The standard minimum cut has a bias toward cutting small, isolated sets of nodes (as shown in Cut 3 which isolates a single pixel cheaply). Normalized cuts (Shi and Malik, 2000) address this by normalizing the cut cost by the total edge connections to all nodes:

Ncut(A,B) = cut(A,B)/assoc(A,V) + cut(A,B)/assoc(B,V)

where assoc(A,V) is the sum of all edge weights from nodes in A to all nodes in the graph. This penalizes cuts that produce very small segments, encouraging more balanced partitions. Cut 3 would receive a high normalized cut value because assoc({pixel 70}, V) is small, making the fraction large.

### (b) Frame differencing [5 marks]

**Setup:** 1D image of 6 pixels. Object of intensity 200 moves right. Background: 100 (left), 200 (right).

| Time | Pixel 1 | Pixel 2 | Pixel 3 | Pixel 4 | Pixel 5 | Pixel 6 |
|------|---------|---------|---------|---------|---------|---------|
| t=1  | **200** | 100     | 100     | 200     | 200     | 200     |
| t=2  | 100     | **200** | 100     | 200     | 200     | 200     |
| t=3  | 100     | 100     | **200** | 200     | 200     | 200     |
| t=4  | 100     | 100     | 100     | **200** | 200     | 200     |
| t=5  | 100     | 100     | 100     | 200     | **200** | 200     |

Bold indicates the object position.

**Frame differencing:** D(t) = |Frame(t) - Frame(t-1)|

**D(t=2) = |Frame(2) - Frame(1)|:**
|100-200|, |200-100|, |100-100|, |200-200|, |200-200|, |200-200|
= 100, 100, 0, 0, 0, 0

Detected motion at pixels 1 and 2 (object left pixel 1, arrived at pixel 2).

**D(t=3) = |Frame(3) - Frame(2)|:**
|100-100|, |100-200|, |200-100|, |200-200|, |200-200|, |200-200|
= 0, 100, 100, 0, 0, 0

Detected motion at pixels 2 and 3.

**D(t=4) = |Frame(4) - Frame(3)|:**
|100-100|, |100-100|, |100-200|, |200-200|, |200-200|, |200-200|
= 0, 0, 100, 0, 0, 0

Detected motion only at pixel 3 (object left pixel 3). **Note:** The object arrives at pixel 4, but since the background at pixel 4 is already 200 (same as object intensity), no change is detected there!

**D(t=5) = |Frame(5) - Frame(4)|:**
= 0, 0, 0, 0, 0, 0

**No motion detected at all!** The object moved from pixel 4 to pixel 5, but both the background and object have intensity 200 in this region.

**Weaknesses highlighted:**

1. **Camouflage / aperture problem:** When the object has the same intensity as the background it moves into, frame differencing fails completely (as shown at t=4 and t=5). The object becomes invisible.
2. **Double detection (ghosting):** Frame differencing detects both where the object was and where it moved to, producing a "double" detection (two regions instead of one).
3. **No information when stationary:** If the object stops moving, it is no longer detected.
4. **Sensitivity to noise:** Small intensity fluctuations can create false positives.
5. **Only detects change, not the object itself:** Cannot distinguish between the object region and the previously-occupied region.

### (c) Optical flow and Horn-Schunck [5 marks]

**Assumptions of optical flow:**

1. **Brightness constancy:** The intensity of a point remains constant as it moves between frames: I(x,y,t) = I(x+dx, y+dy, t+dt).
2. **Small motion:** Movement between frames is small enough that Taylor expansion is valid.
3. **Spatial coherence (Horn-Schunck specific):** Neighbouring pixels have similar velocities (smoothness constraint).

**The Horn-Schunck equations:**

u = u_bar - lambda * (Ix) * (u_bar * Ix + v_bar * Iy + It) / (1 + lambda * (Ix^2 + Iy^2))

v = v_bar - lambda * (Iy) * (u_bar * Ix + v_bar * Iy + It) / (1 + lambda * (Ix^2 + Iy^2))

Where u_bar and v_bar are local averages of the velocity field, Ix, Iy, It are the spatiotemporal image gradients.

**What is lambda for?**

Lambda (lambda) is the **regularisation parameter** (smoothness weight) that controls the trade-off between:
- Faithfulness to the **data term** (the optical flow constraint equation)
- **Smoothness** of the resulting flow field

A larger lambda gives **more weight to the data term** (the brightness constancy constraint) relative to the smoothness term. A smaller lambda gives more weight to the smoothness constraint.

**Effect of increasing lambda from 0.000001 to 0.001:**

- With very small lambda (0.000001): The smoothness constraint dominates. The flow field will be very smooth (nearly uniform), potentially over-smoothing motion boundaries. Fine details and discontinuities in the flow are lost.
- With larger lambda (0.001): The data term has more influence. The flow field will be more faithful to the actual image gradients, allowing sharper velocity discontinuities. However, it may also be more susceptible to noise.

Increasing lambda produces a flow field that is less smooth but more accurate at motion boundaries and better at capturing local variations in velocity.

### (d) Detecting squares using active contours [6 marks]

**Design for detecting squares of different orientations and sizes:**

**1. Energy formulation:**

The active contour (snake) minimizes:

E_total = E_internal + E_external

**2. Internal energy (shape prior for squares):**

- **Elasticity term (E_elastic):** Controls the stretching of the contour. Use moderate elasticity to maintain roughly equal side lengths.
- **Bending/rigidity term (E_bending):** Penalizes curvature. For squares, we want exactly four corners (high curvature points) and straight edges between them.
- **Corner constraint:** Modify the internal energy to favour exactly 4 points with 90-degree angles. At corner points, reduce the bending penalty; elsewhere, keep bending penalty high to enforce straight edges.

**3. External energy (image forces):**

- **Edge-based:** Use gradient magnitude |nabla I| to attract the contour toward strong edges.
- **Balloon force:** Add an inflation force to expand the contour outward (to avoid collapse to a point) or deflation to shrink toward the square.
- **GVF (Gradient Vector Flow):** Extend capture range of edges using diffusion of gradient vectors.

**4. Initialisation:**

- Start with a circular contour (or rough quadrilateral) placed in the vicinity of each suspected square.
- Multi-scale approach: detect at coarse scale first, then refine.

**5. Handling different sizes:**

- Initialise contours at multiple scales or use a scale-space approach.
- The elasticity term naturally allows the contour to adapt to different sizes.

**6. Handling different orientations:**

- The energy formulation is inherently rotation-invariant since it only depends on local curvature and edge strength, not absolute orientation.
- The four corner points can appear at any rotation.

**7. Evolution:**

- Iteratively move control points to minimize total energy using gradient descent.
- At convergence, verify the detected shape has 4 corners with approximately equal side lengths and 90-degree angles.

### (e) Condensation algorithm [6 marks]

**CONDENSATION (CONditional DENSity propagATION)** is a particle filter-based algorithm for visual tracking, developed by Isard and Blake (1998).

**Core idea:** Represent the posterior probability distribution of the object state (position, velocity, scale, etc.) using a weighted set of samples (particles), rather than a single estimate.

**Algorithm steps at each time step t:**

1. **Initialisation (t=0):** Generate N random samples (particles) {s_0^(i)}, each representing a possible state of the object. Assign uniform weights w_0^(i) = 1/N.

2. **Selection (Resampling):** Draw N samples from the previous set {s_{t-1}^(i)} with probability proportional to their weights w_{t-1}^(i). Particles with higher weights are selected more often (survival of the fittest).

3. **Prediction (Drift + Diffusion):** Apply a dynamical model to each selected sample:
   - **Drift:** Apply deterministic motion model (e.g., constant velocity prediction)
   - **Diffusion:** Add random noise to account for uncertainty in the motion model
   
   s_t^(i) = A * s_{t-1}^(i) + B * noise

4. **Measurement (Weighting):** For each predicted particle, compute the observation likelihood -- how well the particle's hypothesised state matches the current image observation. Set the new weight:
   
   w_t^(i) = p(z_t | s_t^(i))
   
   where z_t is the current observation (e.g., edge strength along predicted contour).

5. **Estimation:** The tracked state is estimated as the weighted mean of all particles, or the particle with maximum weight.

**Advantages over Kalman filtering:**
- Can handle **multi-modal distributions** (multiple hypotheses)
- Does not require Gaussian assumptions
- Can recover from temporary occlusions (some particles survive)
- Naturally handles non-linear dynamics

**Application to motion tracking:** The state vector typically includes position (x, y), velocity (vx, vy), and possibly scale/orientation. The observation model measures how well a shape template matches the image at each particle's proposed location.

## Question 3: Evaluation of CV Algorithms (25 marks)

### (a) Three ways to evaluate optical flow algorithms [3 marks]

1. **Angular error (AE):** Compute the angle between the estimated flow vector (u_e, v_e, 1) and ground-truth flow vector (u_gt, v_gt, 1) in 3D space. Report mean and standard deviation of angular errors across all pixels.

2. **Endpoint error (EPE):** Compute the Euclidean distance between the endpoint of the estimated flow vector and the ground-truth flow vector: sqrt((u_e - u_gt)^2 + (v_e - v_gt)^2). Report mean EPE.

3. **Interpolation error / Warping error:** Use the estimated flow to warp frame t to frame t+1, then measure the difference between the warped frame and the actual frame t+1 (e.g., using SSD or NCC). This evaluates practical usefulness without requiring ground truth flow.

### (b) Earth Mover's Distance [5 marks]

**Setup:** Segment A has dark pixels on the left half and light pixels on the right half. B1 is a slight spatial shift of A. B2 is a different spatial pattern.

**Earth Mover's Distance (EMD):**

EMD measures the minimum amount of "work" required to transform one distribution into another, where work = amount of mass moved x distance moved.

**EMD(B1, A):** Since B1 is a slight shift of A, the pixel distributions are spatially offset by a small amount. The EMD would be **small** because only a small amount of "earth" needs to be moved a short distance to transform B1 into A.

**EMD(B2, A):** If B2 has a fundamentally different spatial pattern (e.g., dark pixels scattered throughout rather than concentrated on the left), the EMD would be **larger** because mass needs to be moved over greater distances.

**Another distance metric -- Hausdorff distance:**

The Hausdorff distance measures the maximum of the minimum distances between two point sets:

H(A,B) = max(max_a min_b d(a,b), max_b min_a d(a,b))

For segments: compare boundaries or pixel sets.

**Comparison:**

| Aspect | EMD | Hausdorff Distance |
|--------|-----|-------------------|
| **Sensitivity to outliers** | Robust (considers entire distribution) | Very sensitive (uses maximum) |
| **Partial matches** | Handles well (can use partial matching) | Poor (single outlier dominates) |
| **Computational cost** | Higher (solving transportation problem) | Lower (nearest-neighbour search) |
| **What it measures** | Average spatial rearrangement needed | Worst-case mismatch |
| **Pros** | Captures perceptual similarity well; handles different-sized segments | Simple to compute; provides guarantee on worst case |
| **Cons** | Computationally expensive (linear programming); requires ground distance | One outlier pixel can dominate; does not reflect overall quality |

### (c) Shared test benchmarks [6 marks]

**Why shared test benchmarks are important:**

1. **Fair comparison:** All algorithms are evaluated on the same data under identical conditions, eliminating bias from dataset selection.
2. **Reproducibility:** Other researchers can reproduce results and verify claims.
3. **Progress tracking:** The community can track improvements over time against a fixed standard.
4. **Ground truth availability:** Benchmarks provide carefully annotated ground truth (often expensive to produce), shared across the community.
5. **Standardised metrics:** Benchmarks define evaluation protocols, ensuring results are directly comparable.
6. **Reduced cherry-picking:** Researchers cannot selectively report results only on favourable images.

**When benchmarks can be a problem:**

1. **Over-fitting to the benchmark:** Algorithms may be tuned specifically for the benchmark dataset rather than developing genuinely better approaches. This leads to high benchmark scores but poor generalisation.
2. **Limited diversity:** A benchmark may not represent the full range of real-world conditions (e.g., Middlebury optical flow has limited motion ranges, no outdoor scenes in early versions).
3. **Stifling innovation:** Focus on incremental improvements on benchmarks may discourage fundamentally different approaches that score poorly on current metrics but address different aspects.
4. **Outdated benchmarks:** As the field advances, benchmarks may become saturated (all algorithms score similarly) or may not test the challenges that matter most.
5. **Metric limitations:** The chosen evaluation metric may not correlate with actual application performance (e.g., angular error may not reflect visual quality of flow-based video editing).
6. **Bias toward particular problem types:** If a benchmark focuses on one scenario (e.g., indoor scenes), methods optimised for it may fail in other contexts.

### (d) Four ways to compare velocity vectors [4 marks]

Given estimated flow vector **v_e** = (u_e, v_e) and ground-truth **v_gt** = (u_gt, v_gt):

**1. Angular Error (AE):**

Extend vectors to 3D: (u_e, v_e, 1.0) and (u_gt, v_gt, 1.0). Compute the angle between them:

AE = arccos((u_e * u_gt + v_e * v_gt + 1) / (sqrt(u_e^2 + v_e^2 + 1) * sqrt(u_gt^2 + v_gt^2 + 1)))

This measures directional accuracy, normalized by magnitude.

**2. Endpoint Error (EPE):**

EPE = sqrt((u_e - u_gt)^2 + (v_e - v_gt)^2)

The Euclidean distance between vector endpoints. Penalizes both magnitude and direction errors.

**3. Magnitude Error:**

ME = |magnitude(v_e) - magnitude(v_gt)| = |sqrt(u_e^2 + v_e^2) - sqrt(u_gt^2 + v_gt^2)|

Measures only the difference in speed, ignoring direction.

**4. Direction Error:**

DE = arccos((u_e * u_gt + v_e * v_gt) / (|v_e| * |v_gt|))

The angle between the 2D vectors in the image plane. Measures only directional accuracy, ignoring speed differences.

**Diagram description:** Consider a coordinate plane with the origin at a pixel. The ground-truth vector v_gt points from origin to point G. The estimated vector v_e points from origin to point E. The endpoint error is the distance from E to G. The angular error is the angle between the 3D-extended vectors. The magnitude error is the difference in the lengths of v_e and v_gt. The direction error is the 2D angle between the two arrows.

### (e) HMMs for scene analysis [7 marks]

**Nature of learned parameters:**

An HMM is defined by:
- **State transition probabilities A = {a_ij}:** The probability of transitioning from hidden state i to state j. Learned parameter reflects how likely it is to move between scene states.
- **Observation/emission probabilities B = {b_j(o_k)}:** The probability that observation o_k is emitted when in state j. These encode what visual features are expected in each scene state.
- **Initial state distribution pi = {pi_i}:** The probability of starting in each state.

**Example scenario: Traffic monitoring at an intersection**

- **Hidden states:** {Empty, Light_Traffic, Heavy_Traffic, Congestion, Accident}
- **Observations:** Extracted visual features per frame, quantized into symbols (e.g., {few_vehicles, moderate_vehicles, many_vehicles, stationary_vehicles, scattered_debris})
- **Application:** Given a sequence of observations, determine the most likely sequence of traffic states (Viterbi), or detect if an abnormal state (Accident) is reached.

**Learning stage (Baum-Welch / EM algorithm):**

Training uses the Baum-Welch algorithm to estimate A, B, and pi from training observation sequences.

**Issues with learning:**

1. **Local optima:** Baum-Welch uses EM which converges to local optima, not global. Different initialisations can give different models. Solution: multiple random restarts.
2. **Insufficient training data:** Rare states (e.g., Accident) may have very few examples, leading to poor estimation of their emission and transition probabilities.
3. **Number of states:** The number of hidden states must be chosen a priori. Too few states under-represent the scene complexity; too many lead to overfitting.
4. **Observation quantization:** Continuous visual features must be discretized (or Gaussian mixture HMMs used), and the choice of quantization affects performance.
5. **Non-stationarity:** Real-world scenes change over time (e.g., lighting conditions), but HMMs assume stationary parameters.
6. **Independence assumptions:** HMMs assume observations depend only on the current state (Markov property), which may not hold for complex visual scenes.
7. **Training time:** Convergence can be slow for large state spaces and long observation sequences.

## Question 4: Scene Analysis (25 marks)

### (a) Three problems HMMs generally tackle [3 marks]

1. **Evaluation (Likelihood computation):** Given a model lambda and an observation sequence O, compute P(O|lambda) -- the probability that the model generated the sequence. Solved using the **Forward algorithm**.

2. **Decoding (State sequence estimation):** Given a model lambda and observation sequence O, find the most likely hidden state sequence. Solved using the **Viterbi algorithm**.

3. **Learning (Parameter estimation):** Given observation sequences, find the model parameters (A, B, pi) that maximize P(O|lambda). Solved using the **Baum-Welch algorithm** (a specialisation of EM).

---

### (b) What is "hidden" in an HMM and why? [5 marks]

**What is hidden:**

The **states** are hidden. In an HMM, we cannot directly observe which state the system is in at any given time. We can only observe the **emissions/outputs** generated by the states. The actual state sequence is a latent (hidden) variable that must be inferred from the observations.

**Why are states hidden:**

In most real-world applications, the underlying process generating observations cannot be directly measured. We only have access to noisy, indirect measurements. The Markov chain of states models the true underlying process, while observations are probabilistic functions of those states.

**Example where the hidden aspect is useful: Speech recognition**

- **Hidden states:** Phonemes or sub-phoneme units (the actual sounds being produced)
- **Observations:** Acoustic features (spectral coefficients, MFCCs) extracted from the audio signal

We cannot directly observe which phoneme a speaker is producing -- we only measure acoustic features that are influenced by the phoneme, speaker characteristics, noise, etc. The hidden states capture the linguistic structure (e.g., which phoneme sequences are likely in English), while the emission probabilities model the acoustic variability. By keeping states hidden, we can:
- Model the temporal structure of speech (transition probabilities encode phoneme sequencing rules)
- Handle acoustic variability (the same phoneme sounds different in different contexts)
- Perform recognition by finding the most likely state sequence (phoneme sequence = word) given acoustic observations

### (c) Detecting abnormal behaviour with HMMs [6 marks]

**Approach:**

1. **Training phase (model normal behaviour):**
   - Collect video sequences of normal behaviour in the scene
   - Extract features from each frame (e.g., motion vectors, positions, velocities, trajectories)
   - Quantize features into discrete observation symbols
   - Train one or more HMMs on normal behaviour sequences using Baum-Welch
   - The trained HMM(s) encode the statistical patterns of normal activity

2. **Detection phase:**
   - Process incoming video and extract the same features
   - Compute the log-likelihood of the observation sequence under the trained normal model: P(O|lambda_normal)
   - If the likelihood falls **below a threshold**, the observed behaviour is flagged as **abnormal** (the normal model assigns low probability to sequences it has not seen during training)

3. **Threshold setting:**
   - Use validation sequences of normal behaviour to establish a baseline distribution of likelihoods
   - Set threshold at a certain number of standard deviations below the mean likelihood

4. **Sliding window approach:**
   - Evaluate likelihood over a sliding window of T frames
   - Allows real-time detection and temporal localisation of the anomaly

5. **Multiple models:**
   - Train separate HMMs for different types of normal behaviour (e.g., walking, running, cycling at a crossing)
   - An observation is abnormal if ALL normal models assign low likelihood

**Advantages:**
- Does not require examples of abnormal behaviour (unsupervised anomaly detection)
- Captures temporal patterns (not just instantaneous features)
- Probabilistic framework provides a principled threshold

**Challenges:**
- Defining what is "normal" requires sufficient training data covering all variations
- Setting the threshold involves a trade-off between false alarms and missed detections
- New normal behaviours not in training data will be flagged as abnormal

## Question 5: Stereo Vision (25 marks)

### (a) Relationship between disparity and depth [4 marks]

**Disparity** is the difference in horizontal image position of the same 3D point as seen by two cameras:

d = x_L - x_R

where x_L and x_R are the horizontal pixel coordinates of the point in the left and right images respectively.

**Relationship to depth:**

For a parallel stereo setup with baseline B (distance between camera centres) and focal length f:

**Z = (f * B) / d**

where Z is the depth (distance from the camera to the 3D point).

**Key properties:**
- Disparity is **inversely proportional** to depth: closer objects have larger disparity, farther objects have smaller disparity.
- At infinite depth, disparity approaches zero.
- The relationship is non-linear: a fixed change in disparity corresponds to a larger depth change at greater distances.
- Depth resolution decreases with distance (quantization of disparity to pixels means depth intervals become larger at greater distances).

### (b) Correspondence problem and low-level features [5 marks]

The **correspondence problem** is the task of finding which point in the left image corresponds to which point in the right image (i.e., which pixels in both images are projections of the same 3D point).

**Issues illustrated with low-level features:**

**1. Uniform/textureless regions:**
In a region of constant intensity (e.g., a white wall), every pixel looks identical. There is no distinguishing feature to establish correspondence. The matching cost is flat, giving ambiguous matches.

**2. Repetitive patterns:**
If the scene contains repeating structures (e.g., a picket fence, tiled floor), multiple false matches have the same matching cost as the true match. This causes ambiguity.

**3. Occlusion:**
Some points visible in one camera are occluded (hidden) in the other. These points have NO valid correspondence. The algorithm must detect and handle half-occluded regions.

**4. Specular reflections/highlights:**
Shiny surfaces produce view-dependent appearances. The same 3D point may have different intensities in left and right images, violating the brightness constancy assumption.

**5. Depth discontinuities:**
At object boundaries, neighbouring pixels in one image may correspond to very different depths. A window-based matching approach will average across the discontinuity, producing errors at boundaries.

Arrows show which rays from the two cameras intersect at scene points, illustrating where correspondence succeeds and fails.

### (c) Camera calibration for stereo [4 marks]

**Is calibration required?**

**Strictly speaking, yes** -- camera calibration is required for **metric (quantitative) depth** reconstruction. Without knowing the intrinsic parameters (focal length, principal point, distortion) and extrinsic parameters (relative position and orientation of cameras), we cannot compute actual depth values in world units.

**However**, for certain tasks, full calibration may not be necessary:
- **Relative depth ordering** can be obtained from disparity alone (larger disparity = closer) without calibration.
- **Self-calibration (auto-calibration)** methods can estimate parameters from image correspondences alone, given sufficient constraints.
- **Rectification** (aligning epipolar lines with image rows) can be done with the fundamental matrix, which does not require full calibration.

**Parameters that define camera setups:**

**Intrinsic parameters (per camera):**
- Focal length (f_x, f_y)
- Principal point (c_x, c_y)
- Skew coefficient
- Lens distortion coefficients (radial: k1, k2, k3; tangential: p1, p2)

**Extrinsic parameters (relative pose):**
- Rotation matrix R (3x3, or 3 angles) -- relative orientation between cameras
- Translation vector T (3x1) -- relative position (baseline)

**Together:** For a stereo pair, we need 2 sets of intrinsics + 1 relative pose (R, T) = approximately 11+ parameters total (after accounting for constraints).

### (d) Epipolar line [5 marks]

**What is an epipolar line?**

Given a point p in the left image, its corresponding point in the right image must lie on a specific line called the **epipolar line**. This line is the projection onto the right image of the ray from the left camera centre through point p in 3D space.

**Formally:** If F is the fundamental matrix relating two views, and p_L is a point in the left image, then the epipolar line in the right image is:

l_R = F * p_L

Similarly, the epipolar line in the left image for a point p_R in the right image is:

l_L = F^T * p_R

**Implications:**

1. **Reduces search from 2D to 1D:** Instead of searching the entire right image for the correspondence of a left image point, we only need to search along the epipolar line. This dramatically reduces computational cost and false matches.

2. **Epipolar constraint:** All correct correspondences must satisfy p_R^T * F * p_L = 0. This can be used to verify or reject potential matches.

3. **After rectification:** If images are rectified (warped so that epipolar lines are horizontal), corresponding points lie on the same row. This simplifies implementation to a 1D horizontal search.

4. **Epipoles:** All epipolar lines in one image pass through a single point called the epipole (the projection of the other camera's centre).

**What is needed to compute epipolar lines:**

- The **fundamental matrix F** (encodes the epipolar geometry between two views)
- To compute F, we need either:
  - **Calibration:** Full camera intrinsics + extrinsics (F = K_R^{-T} * [T]_x * R * K_L^{-1})
  - **Point correspondences:** At least 8 corresponding point pairs (using the 8-point algorithm) or 7 points (7-point algorithm), without requiring calibration
- Alternatively, the **essential matrix E** (for calibrated cameras): E = [T]_x * R, and F = K_R^{-T} * E * K_L^{-1}

### (e) Multi-scale optical flow adapted for binocular stereo [7 marks]

**Multi-scale optical flow recap:**

In multi-scale (coarse-to-fine) optical flow, a Gaussian pyramid is constructed. At the coarsest level, large displacements appear as small displacements that can be estimated. The estimate is then propagated to finer levels and refined.

**Adaptation to binocular stereo:**

In stereo, we estimate horizontal disparity (displacement) between left and right images instead of temporal motion. For rectified stereo pairs, the displacement is purely horizontal.

**Key modifications:**
- Instead of two temporal frames, use left and right images
- Displacement is only horizontal (1D search along epipolar lines after rectification)
- No temporal smoothness -- use spatial smoothness instead

**Pseudo-code:**

```
function MultiScaleStereo(I_left, I_right, num_levels):
    // Build Gaussian pyramids
    pyramid_L = BuildPyramid(I_left, num_levels)
    pyramid_R = BuildPyramid(I_right, num_levels)
    
    // Initialise disparity map at coarsest level to zero
    d = zeros(size(pyramid_L[num_levels]))
    
    // Coarse-to-fine estimation
    for level = num_levels down to 1:
        // Get images at current level
        L = pyramid_L[level]
        R = pyramid_R[level]
        
        // Warp right image using current disparity estimate
        R_warped = WarpHorizontal(R, d)
        
        // Compute residual disparity at this level
        // Using block matching or gradient-based method
        residual = ComputeDisparity(L, R_warped)
        
        // Accumulate disparity
        d = d + residual
        
        // If not at finest level, upsample for next level
        if level > 1:
            d = Upsample(d, size(pyramid_L[level-1]))
            d = d * 2  // Scale disparity (doubled resolution = doubled disparity)
    
    return d

function ComputeDisparity(L, R_warped):
    // Gradient-based approach (analogous to Lucas-Kanade)
    Ix = SpatialGradientX(L)
    Iy = SpatialGradientY(L)
    It = R_warped - L  // "Temporal" gradient is actually stereo difference
    
    // For each pixel/window, solve:
    // Ix * dd = -It  (since disparity is horizontal only)
    // Using a local window W:
    for each pixel (x,y):
        dd(x,y) = -sum_W(Ix * It) / sum_W(Ix * Ix)
    
    return dd
```

**Strengths:**

1. **Handles large disparities:** By working at coarse scales first, large disparities (which would violate linearisation assumptions) become small and estimable.
2. **Computational efficiency:** Coarse levels are small, so initial estimates are cheap. Fine levels only need to estimate small residuals.
3. **Robustness:** Coarse levels are less affected by noise and fine texture variations.
4. **Sub-pixel accuracy:** Gradient-based refinement at the finest level can achieve sub-pixel disparity estimates.
5. **Dense output:** Produces a disparity value for every pixel (unlike feature-based methods).

**Weaknesses:**

1. **Coarse-level errors propagate:** If the estimate at a coarse level is wrong (e.g., at depth discontinuities), the error propagates to finer levels and may not be correctable.
2. **Over-smoothing at boundaries:** The coarse levels blur depth discontinuities, causing "bleeding" of foreground disparity into background.
3. **Textureless regions:** Even at coarse scales, uniform regions lack gradient information needed for the estimation.
4. **Occluded regions:** Points visible in one image but not the other cannot have valid disparities; the method has no explicit occlusion handling.
5. **Linearisation assumption:** The gradient-based computation assumes small residual disparities at each level, which may not hold near discontinuities.
6. **Fixed number of levels:** The pyramid depth must be chosen to accommodate the maximum disparity expected; insufficient levels will fail for large disparities.
